# 4.1 TCP 커넥션

전 세계 모든 HTTP 통신은 **TCP/IP(Transmission Control Protocol / Internet Protocol)** 를 통해 이루어진다.  
세계 어디서든 클라이언트 애플리케이션은 서버 애플리케이션으로 TCP/IP 커넥션을 맺을 수 있다.  
일단 커넥션이 맺어지면 클라이언트와 서버 컴퓨터 간에 주고받는 메시지들은 손실 혹은 손상되거나 순서가 바뀌지 않고 안전하게 전달된다.

---

## 4.1.1 신뢰할 수 있는 데이터 전송 통로인 TCP

**HTTP 커넥션(HTTP Connection)** 은 몇몇 사용 규칙을 제외하고는 **TCP 커넥션(TCP Connection)** 에 불과하다.  
TCP 커넥션의 한쪽에 있는 바이트들은 반대쪽으로 순서에 맞게 정확히 전달된다.

> **Java 개발자 관점**: `java.net.Socket`이 바로 TCP 커넥션의 Java 추상화이다.  
> `Socket socket = new Socket("www.example.com", 80);`으로 TCP 커넥션을 생성하면, 그 아래에서 TCP가 데이터를 안전하게 전달해준다.

---

## 4.1.2 TCP 스트림은 세그먼트로 나뉘어 IP 패킷을 통해 전송된다

TCP는 **IP 패킷(IP Packet)**(혹은 IP 데이터그램)이라고 불리는 작은 조각을 통해 데이터를 전송한다.

### 프로토콜 스택

| 구분 | HTTP | HTTPS |
|------|------|-------|
| 애플리케이션 계층 | HTTP | HTTP |
| 보안 계층 | - | TLS or SSL |
| 전송 계층 | TCP | TCP |
| 네트워크 계층 | IP | IP |
| 데이터 링크 계층 | Network Interfaces | Network Interfaces |

- **HTTPS(HTTP Secure)** 는 HTTP와 TCP 사이에 **TLS(Transport Layer Security)** 혹은 **SSL(Secure Sockets Layer)** 이라 불리는 암호화(cryptographic encryption) 계층을 추가한 것이다.

### IP 패킷 구성

각 TCP 세그먼트는 하나의 IP 주소에서 다른 IP 주소로 IP 패킷에 담겨 전달된다.  
IP 패킷 각각은 다음을 포함한다:

- **IP 패킷 헤더** (보통 20바이트): 발신지/목적지 IP 주소, 크기, 기타 플래그
- **TCP 세그먼트 헤더** (보통 20바이트): TCP 포트 번호, 제어 플래그, 데이터 순서와 무결성 검사 숫자 값
- **TCP 데이터 조각** (0 혹은 그 이상의 바이트)

---

## 4.1.3 TCP 커넥션 유지하기

컴퓨터는 항상 TCP 커넥션을 여러 개 가지고 있다.  
TCP는 **포트 번호(Port Number)** 를 통해서 이런 여러 개의 커넥션을 유지한다.

TCP 커넥션은 네 가지 값으로 식별한다:

```
<발신지 IP 주소, 발신지 포트, 수신지 IP 주소, 수신지 포트>
```

이 네 가지 값으로 유일한 커넥션을 생성한다.  
서로 다른 두 개의 TCP 커넥션은 네 가지 주소 구성요소의 값이 모두 같을 수 없다 (단, 일부가 같을 수는 있다).

### 예시: 네 개의 TCP 커넥션

| 커넥션 | 발신지 IP 주소 | 발신지 포트 | 목적지 IP 주소 | 목적지 포트 |
|--------|----------------|------------|----------------|------------|
| A | 209.1.32.34 | 2034 | 204.62.128.58 | 4133 |
| B | 209.1.32.35 | 3227 | 204.62.128.58 | 4140 |
| C | 209.1.32.35 | 3105 | 207.25.71.25 | 80 |
| D | 209.1.33.89 | 5100 | 207.25.71.25 | 80 |

> C와 D는 동일한 목적지 포트(80)를 가리키지만, 발신지 IP 주소가 다르기 때문에 서로 다른 커넥션이다.

## 4.1.4 TCP 소켓 프로그래밍

운영체제는 TCP 커넥션의 생성과 관련된 여러 기능을 제공한다.  
소켓 API는 유닉스 운영체제용으로 먼저 개발되었지만, 지금은 대부분의 운영체제와 프로그램 언어에서 사용할 수 있다.

### 소켓 API 주요 함수

| 소켓 API 호출 | 설명 |
|---------------|------|
| `s = socket(parameters)` | 연결이 되지 않은 익명의 새로운 소켓 생성 |
| `bind(s, local IP:port)` | 소켓에 로컬 포트 번호와 인터페이스 할당 |
| `connect(s, remote IP:port)` | 로컬의 소켓과 원격의 호스트 및 포트 사이에 TCP 커넥션 생성 |
| `listen(s,...)` | 커넥션을 받아들이기 위해 로컬 소켓에 허용함을 표시 |
| `s2 = accept(s)` | 누군가 로컬 포트에 커넥션을 맺기를 기다림 |
| `n = read(s, buffer, n)` | 소켓으로부터 버퍼에 n바이트 읽기 시도 |
| `n = write(s, buffer, n)` | 소켓으로부터 버퍼에 n바이트 쓰기 시도 |
| `close(s)` | TCP 커넥션을 완전히 끊음 |
| `shutdown(s, side)` | TCP 커넥션의 입출력만 닫음 |
| `getsockopt(s, ...)` | 내부 소켓 설정 옵션값을 읽음 |
| `setsockopt(s, ...)` | 내부 소켓 설정 옵션값을 변경 |

### HTTP 트랜잭션 흐름 (클라이언트 & 서버)

```
[서버]
(S1) 새로운 소켓을 만든다 (socket)
(S2) 80 포트로 소켓을 묶는다 (bind)
(S3) 소켓 커넥션을 허가한다 (listen)
(S4) 커넥션을 기다린다 (accept)

[클라이언트]
(C1) IP 주소와 포트를 얻는다
(C2) 새로운 소켓을 생성한다 (socket)
(C3) 서버의 IP:포트로 연결한다 (connect)
          ↕ 커넥션 성공
(C4) 성공적으로 연결
          ← (S5) 애플리케이션 커넥션 통지
          ← (S6) 요청을 읽기 시작한다 (read)
(C5) HTTP 요청을 보낸다 (write)
          → (S7) HTTP 요청 메시지를 처리한다
(C6) HTTP 응답을 기다린다 (read)
          ← (S8) HTTP 응답을 보낸다 (write)
(C7) HTTP 응답을 처리한다
(C8) 커넥션을 닫는다 (close)
          ← (S9) 커넥션을 닫는다 (close)
```

> **Java 개발자 관점**: Java의 `HttpURLConnection`이나 `HttpClient`가 내부적으로 위의 소켓 API 흐름을 추상화하여 처리한다.  
> Spring의 `RestTemplate`이나 `WebClient`를 사용할 때 이 흐름이 자동으로 이루어지고 있다.
