# 4.2 TCP의 성능에 대한 고려

HTTP는 TCP 바로 위에 있는 계층이기 때문에, HTTP 트랜잭션의 성능은 그 아래 계층인 TCP 성능에 영향을 받는다.  
TCP 성능의 특성을 이해함으로써, HTTP의 커넥션 최적화 요소들을 더 잘 알게 되고, 더 좋은 성능의 HTTP 애플리케이션을 설계할 수 있다.

---

## 4.2.1 HTTP 트랜잭션 지연

HTTP 요청 과정에서 발생하는 주요 지연 흐름:

```
DNS 찾기 → 연결 → 요청 → 처리 → 응답 → 종료
```

HTTP 트랜잭션을 지연시키는 원인:

1. **DNS 이름 분석(DNS resolution)**: 클라이언트는 URI에서 웹 서버의 IP 주소와 포트 번호를 알아내야 한다. 최근에 방문한 적 없으면 DNS 이름 분석 인프라를 사용해 수십 초가 걸릴 수 있다.
2. **TCP 커넥션 설정**: 클라이언트는 TCP 커넥션 요청을 서버에게 보내고 서버가 허가 응답을 회신하기를 기다린다. 새로운 TCP 커넥션에서 항상 발생하며 보통 1~2초 소요된다.
3. **HTTP 요청 전송**: 커넥션이 맺어지면 클라이언트는 HTTP 요청을 새로 생성된 TCP 파이프를 통해 전송한다.
4. **HTTP 응답 수신**: 웹 서버가 HTTP 응답을 보내는 것 역시 시간이 소요된다.

> 대부분의 HTTP 지연은 TCP 네트워크 지연 때문에 발생한다.

---

## 4.2.2 성능 관련 중요 요소

HTTP 프로그래머에게 영향을 주는 가장 일반적인 TCP 관련 지연들:

- TCP 커넥션의 핸드세이크 설정
- 인터넷의 혼잡을 제어하기 위한 TCP의 **느린 시작(slow-start)**
- 데이터를 한데 모아 한 번에 전송하기 위한 **네이글(Nagle) 알고리즘**
- TCP의 **편승(piggyback) 확인응답(acknowledgment)** 을 위한 확인응답 지연 알고리즘
- **TIME_WAIT 지연과 포트 고갈**

---

## 4.2.3 TCP 커넥션 핸드세이크 지연

새로운 TCP 커넥션을 열 때, TCP 소프트웨어는 커넥션을 맺기 위한 조건을 맞추기 위해 연속으로 IP 패킷을 교환한다.

### 3-way 핸드세이크 순서

```
클라이언트                          서버
   |                                  |
   |  (a) SYN                         |
   | -------------------------------->|  커넥션 생성 요청
   |                                  |
   |  (b) SYN+ACK                     |
   |<-------------------------------- |  커넥션 수락 응답
   |                                  |
   |  (c) ACK + GET /index.html...    |
   | -------------------------------->|  확인응답 + HTTP 요청
   |                                  |
   |  (d) HTTP/1.1 304 Not modified   |
   |<-------------------------------- |  HTTP 응답
   |                                  |
         [커넥션 핸드세이크 지연]  [데이터 전송]
```

1. **SYN**: 클라이언트가 새 TCP 커넥션 생성을 위해 작은 TCP 패킷(보통 40~60바이트)을 서버에 보낸다. 'SYN'이라는 특별한 플래그를 가지며, 이는 커넥션 생성 요청을 의미한다.
2. **SYN+ACK**: 서버가 커넥션 요청이 받아들여졌음을 의미하는 'SYN'과 'ACK' 플래그를 포함한 TCP 패킷을 클라이언트에게 보낸다.
3. **ACK**: 마지막으로 클라이언트는 커넥션이 잘 맺어졌음을 알리기 위해 서버에게 확인응답 신호를 보낸다. 오늘날의 TCP는 클라이언트가 이 확인응답 패킷과 함께 데이터를 보낼 수 있다.

> **핵심**: 크기가 작은 HTTP 트랜잭션은 50% 이상의 시간을 TCP를 구성하는 데 쓴다.  
> 이 때문에 HTTP는 이미 존재하는 커넥션을 재활용하는 방식을 도입했다.

---

## 4.2.4 확인응답 지연

인터넷 자체가 패킷 전송을 완벽히 보장하지 않기 때문에, TCP는 성공적인 데이터 전송을 보장하기 위해 자체적인 확인 체계를 가진다.

- 각 TCP 세그먼트는 순번과 데이터 무결성 체크섬을 가진다.
- 수신자는 세그먼트를 온전히 받으면 작은 **확인응답(ACK)** 패킷을 송신자에게 반환한다.
- 송신자가 특정 시간 안에 확인응답 메시지를 받지 못하면 패킷이 파기되었거나 오류가 있는 것으로 판단하고 데이터를 다시 전송한다.

### 편승(Piggyback) 확인응답

확인응답은 크기가 작기 때문에, TCP는 같은 방향으로 송출되는 데이터 패킷에 확인응답을 **편승(piggyback)** 시킨다.  
네트워크를 더 효율적으로 사용하기 위해 확인응답 패킷과 데이터 패킷을 하나로 묶는다.

### 확인응답 지연 알고리즘

확인응답을 편승시키기 위해 많은 TCP 스택은 **확인응답 지연** 알고리즘을 구현한다:

- 송출할 확인응답을 특정 시간 동안(보통 **0.1~0.2초**) 버퍼에 저장
- 확인응답을 편승시키기 위한 송출 데이터 패킷을 찾는다
- 일정 시간 안에 송출 데이터 패킷을 찾지 못하면 확인응답은 별도 패킷으로 전송

> **주의**: HTTP의 요청-응답 두 가지 형식으로만 이루어지는 동작 방식은, 확인응답이 송출 데이터 패킷에 편승할 기회를 감소시킨다.  
> 이로 인해 확인응답 지연 알고리즘으로 인한 지연이 자주 발생한다.

## 4.2.5 TCP 느린 시작(slow start)

TCP의 데이터 전송 속도는 TCP 커넥션이 만들어진 지 얼마나 지났는지에 따라 달라질 수 있다.

- TCP 커넥션은 시간이 지나면서 자체적으로 '튜닝'되어, 처음에는 커넥션의 최대 속도를 제한하고 데이터가 성공적으로 전송됨에 따라 속도 제한을 높여나간다.
- 이를 **TCP 느린 시작(slow start)** 이라고 부른다 → 인터넷의 급작스러운 부하와 혼잡을 방지하는 데 쓰인다.

### 느린 시작 동작 원리

패킷이 성공적으로 전달되는 각 시점에 송신자는 추가로 2개의 패킷을 더 전송할 수 있는 권한을 얻는다. 이를 **혼잡 윈도우를 연다(opening the congestion window)** 라고 한다.

- 처음: 1개 패킷 전송
- 확인응답 수신: 2개 패킷 전송 가능
- 2개 모두 확인응답 수신: 4개 패킷 전송 가능
- ...

> **핵심**: 새로운 커넥션은 이미 이는 정도 데이터를 주고받은 '튜닝'된 커넥션보다 느리다.  
> 이 때문에 HTTP에는 이미 존재하는 커넥션을 재사용하는 **지속 커넥션** 기능이 있다.

## 4.2.6 네이글(Nagle) 알고리즘과 TCP_NODELAY

TCP는 데이터 스트림 인터페이스를 제공하여, 애플리케이션이 어떤 크기의 데이터든(심지어 1바이트라도) TCP 스택으로 전송할 수 있도록 한다.  
하지만 각 TCP 세그먼트는 40바이트 상당의 플래그와 헤더를 포함하여 전송하기 때문에, 작은 크기의 패킷을 대량으로 전송하면 네트워크 성능이 크게 떨어진다.

> 1바이트 크기의 패킷을 대량으로 전송하는 것을 **'실리 윈도 증후군(silly window syndrome)'** 전송자라 한다.

### 네이글 알고리즘 동작

**네이글(Nagle) 알고리즘** (RFC 896)은 네트워크 효율을 위해, 패킷을 전송하기 전에 많은 양의 TCP 데이터를 한 개의 덩어리로 합친다:

- 세그먼트가 최대 크기(LAN: ~1,500바이트, 인터넷: ~수백 바이트)가 되지 않으면 전송하지 않는다
- 다만 다른 모든 패킷이 확인응답을 받은 경우에는 최대 크기보다 작은 패킷의 전송을 허락한다
- 다른 패킷들이 아직 전송 중이면 데이터는 버퍼에 저장된다

### 네이글 알고리즘이 HTTP 성능에 미치는 문제

1. **크기가 작은 HTTP 메시지**: 패킷을 채우지 못하기 때문에, 앞으로 생길지 생기지 않을지 모르는 추가적인 데이터를 기다리며 지연된다.
2. **확인응답 지연과의 충돌**: 네이글 알고리즘은 확인응답이 도착할 때까지 데이터 전송을 멈추는 반면, 확인응답 지연 알고리즘은 확인응답을 100~200밀리초 지연시킨다 → 악순환 발생.

### TCP_NODELAY

HTTP 애플리케이션은 성능 향상을 위해 HTTP 스택에 `TCP_NODELAY` 파라미터 값을 설정하여 네이글 알고리즘을 비활성화하기도 한다.  
이 설정을 했다면, 작은 크기의 패킷이 너무 많이 생기지 않도록 큰 크기의 데이터 덩어리를 만들어야 한다.

## 4.2.7 TIME_WAIT의 누적과 포트 고갈

TCP 커넥션의 종단에서 TCP 커넥션을 끊으면, 종단에서는 커넥션의 IP 주소와 포트 번호를 메모리의 작은 **제어영역(control block)** 에 기록해 놓는다.

### TIME_WAIT 목적

이 정보는 같은 주소와 포트 번호를 사용하는 새로운 TCP 커넥션이 일정 시간 동안에는 생성되지 않게 하기 위한 것이다.

- 보통 세그먼트의 최대 생명주기의 두 배 정도인 **'2MSL'** (보통 **2분** 정도)의 시간 동안만 유지된다.
- 이전 커넥션과 관련된 패킷이 그 커넥션과 같은 주소와 포트 번호를 가지는 새로운 커넥션에 삽입되는 문제를 방지한다.

### 포트 고갈 문제

TCP 커넥션을 맺기 위한 네 가지 값 중 세 개는 고정이고 발신지 포트만 변경할 수 있다:

```
<클라이언트 IP, 발신지 포트, 서버 IP, 80>
```

- 사용할 수 있는 발신지 포트의 수는 제한되어 있다 (약 60,000개)
- 2MSL초 동안(약 120초) 커넥션이 재사용될 수 없으므로, 초당 최대 **500개 (60,000 / 120 = 500)** 로 커넥션이 제한된다
- 서버가 초당 500개 이상의 트랜잭션을 처리할 만큼 빠르다면 TIME_WAIT 포트 고갈이 일어난다

> **해결책**: 부하를 생성하는 장비를 더 많이 사용하거나, 클라이언트와 서버가 더 많은 커넥션을 맺을 수 있도록 여러 개의 가상 IP 주소를 쓸 수도 있다.
