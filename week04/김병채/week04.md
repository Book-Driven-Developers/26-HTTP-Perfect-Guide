# 4장. 커넥션 관리

## 소켓 인터페이스 함수

HTTP는 TCP/IP 위에서 동작하며, 실제 네트워크 통신은 소켓 API를 통해 이루어진다.

| 함수 | 설명 |
|------|------|
| `socket()` | 새로운 소켓을 생성한다. 아직 어디에도 연결되지 않은 상태 |
| `bind()` | 소켓에 로컬 포트 번호와 IP 주소를 할당한다 |
| `connect()` | 서버의 IP 주소와 포트로 TCP 커넥션을 수립한다 (클라이언트) |
| `listen()` | 소켓을 수신 대기 상태로 만들어 클라이언트 연결 요청을 받을 준비를 한다 (서버) |
| `accept()` | 새로운 클라이언트 연결이 들어올 때까지 대기하고, 연결이 오면 새 소켓을 반환한다 (서버) |
| `read()` / `write()` | 소켓을 통해 데이터를 읽거나 쓴다 |
| `close()` | TCP 커넥션을 완전히 종료한다. 양방향 모두 닫힘 |
| `shutdown()` | 입력 또는 출력 중 한 방향만 선택적으로 닫을 수 있다 |

클라이언트와 서버의 통신 흐름:
- 클라이언트: `socket()` → `connect()` → `write()` → `read()` → `close()`
- 서버: `socket()` → `bind()` → `listen()` → `accept()` → `read()` → `write()` → `close()`

## HTTP 트랜잭션을 지연시키는 원인

HTTP 요청-응답 한 사이클(트랜잭션)에서 실제 데이터 전송보다 네트워크 통신 자체에서 더 많은 시간이 소요되는 경우가 많다. 주요 지연 원인은 다음과 같다.

1. **DNS 조회 지연**: URI의 호스트명을 IP 주소로 변환하는 과정. 캐시가 없으면 DNS 서버에 쿼리를 보내야 하므로 수십~수백 ms 지연이 발생한다.

2. **TCP 커넥션 수립 지연**: 클라이언트가 서버와 새로운 TCP 커넥션을 맺을 때 3-way handshake(SYN → SYN+ACK → ACK)를 거쳐야 하므로 왕복 시간(RTT)만큼 지연된다.

3. **TCP 느린 시작(Slow Start) 지연**: 새로운 커넥션은 초기에 전송할 수 있는 데이터 양을 제한하고 점진적으로 늘린다. 따라서 새 커넥션은 오래된 커넥션보다 전송 속도가 느리다.

4. **네이글(Nagle) 알고리즘에 의한 지연**: 작은 패킷들을 모아서 한 번에 보내는 과정에서 전송 전 대기 시간이 발생할 수 있다.

5. **확인응답 지연 알고리즘**: ACK를 즉시 보내지 않고 잠시 버퍼링했다가 피기배킹(piggyback)하는 과정에서 지연이 생긴다.

6. **TIME_WAIT 상태**: 커넥션 종료 후 일정 시간 동안 같은 포트를 재사용할 수 없어 포트 고갈이 발생할 수 있다.

## 성능 관련 중요 요소

### TCP 커넥션의 핸드셰이크 설정

TCP 커넥션을 새로 수립하려면 3-way handshake를 거쳐야 한다.

```
클라이언트          서버
    |---SYN-------->|   (1) 연결 요청
    |<--SYN+ACK----|   (2) 연결 수락
    |---ACK-------->|   (3) 확인
    |               |
    |---HTTP 요청-->|   (4) 실제 데이터 전송 시작
```

HTTP 요청 전에 반드시 이 1 RTT의 지연이 발생한다. 작은 HTTP 트랜잭션의 경우 실제 데이터 전송보다 handshake에 더 많은 시간이 소요될 수 있다.

### 인터넷 혼잡을 제어하기 위한 TCP의 느린 시작(Slow Start)

네트워크 혼잡을 방지하기 위해 TCP는 새 커넥션에서 한 번에 전송할 수 있는 패킷 수를 제한하고 점진적으로 늘려나간다.

- 처음에는 최대 세그먼트 크기(MSS)의 1배만 전송
- ACK를 받을 때마다 전송 가능한 양을 2배씩 증가 (혼잡 윈도우 크기)
- 이 때문에 새로 수립된 커넥션은 기존에 사용하던 커넥션보다 성능이 낮다
- 지속 커넥션을 재사용하면 이 느린 시작 페널티를 피할 수 있다

### 데이터를 모아 한 번에 전송하는 네이글(Nagle) 알고리즘

작은 패킷을 무수히 많이 보내는 것(tinygram problem)을 방지하기 위한 알고리즘이다.

- 아직 ACK를 받지 못한 데이터가 있으면, 새 데이터를 버퍼에 쌓아두고 기다린다
- 이전 패킷의 ACK가 도착하거나, 버퍼가 MSS 크기에 도달하면 한 번에 전송한다

**문제점**: 확인응답 지연 알고리즘과 함께 사용할 때 서로 맞물려 추가 지연이 발생할 수 있다. HTTP 성능 향상을 위해 `TCP_NODELAY` 소켓 옵션으로 네이글 알고리즘을 비활성화하는 경우도 있다.

### TCP 편승 확인응답을 위한 확인응답 지연 알고리즘

ACK 패킷 하나를 보내기 위해 패킷 전체를 소비하는 것은 비효율적이다. 이를 위해 ACK를 즉시 보내지 않고, 같은 방향으로 보낼 데이터가 있을 때 ACK를 함께 실어 보내는 **피기배킹(piggyback)** 방식을 사용한다.

- ACK를 일정 시간(보통 100~200ms) 버퍼에 보관
- 그 사이에 같은 방향으로 보낼 패킷이 생기면 ACK를 그 패킷에 편승시켜 전송
- 그렇지 않으면 시간 초과 후 단독 전송

**문제점**: HTTP처럼 요청-응답이 반이중 방식으로 교대로 일어나는 프로토콜에서는 피기배킹 기회가 거의 없어 대기 시간만 늘어난다.

### TIME_WAIT 지연과 포트 고갈

TCP 커넥션을 끊은 쪽은 종료 후 **2MSL(Maximum Segment Lifetime, 보통 2분)** 동안 `TIME_WAIT` 상태를 유지한다.

**목적**: 네트워크 상에 남아 있는 패킷이 새로운 커넥션에 섞여 들어오는 것을 방지한다.

**포트 고갈 문제**:
- 클라이언트 IP와 포트, 서버 IP와 포트의 조합이 같은 커넥션은 TIME_WAIT 중에는 새로 수립할 수 없다
- 성능 테스트처럼 단시간에 대량의 커넥션을 생성하는 환경에서는 사용 가능한 포트가 모두 TIME_WAIT 상태가 되어 포트 고갈이 발생할 수 있다

## 커넥션 헤더

HTTP는 두 개의 인접한 노드 간 커넥션에만 적용되고, 그 다음 노드에는 전달되지 않아야 하는 헤더 목록을 지정하기 위해 `Connection` 헤더를 사용한다.

```
Connection: keep-alive
Connection: close
Connection: 헤더이름1, 헤더이름2
```

**역할**:
1. `Connection: keep-alive` — 지속 커넥션을 요청한다 (HTTP/1.0 방식)
2. `Connection: close` — 요청/응답 완료 후 커넥션을 끊겠다고 알린다
3. `Connection: 헤더이름` — 해당 이름의 헤더가 이 커넥션에서만 사용되며, 프록시가 다음 노드로 전달할 때 제거해야 하는 **홉별(hop-by-hop) 헤더**임을 명시한다

**중요**: 프록시는 `Connection` 헤더에 명시된 헤더들을 다음 노드로 포워딩하기 전에 반드시 제거해야 한다. 이를 지키지 않으면 **멍청한 프록시(blind relay)** 문제가 발생해 의도치 않은 동작을 야기할 수 있다.

## 병렬 커넥션

하나의 페이지를 로드할 때 필요한 여러 리소스를 동시에 여러 개의 TCP 커넥션으로 병렬로 받아오는 방식이다.

**장점**:
- 각 리소스를 순차적으로 기다리지 않고 동시에 내려받아 전체 로딩 시간을 줄일 수 있다
- 사용자 입장에서 더 빠르게 느껴진다 (화면에 점진적으로 렌더링됨)

**단점**:
- 클라이언트의 대역폭이 제한된 환경에서는 병렬 커넥션이 오히려 서로 경쟁해 각각의 속도가 느려진다
- 커넥션 수립 자체에 리소스(메모리, 소켓)가 소모된다
- 서버 부하가 증가한다

**브라우저의 병렬 커넥션 제한**:
- 브라우저는 실제로 병렬 커넥션을 사용하지만, **대부분 한 서버에 대해 4개** 정도의 병렬 커넥션만 허용한다
- 이는 서버와 네트워크에 과도한 부하를 주는 것을 방지하기 위한 현실적인 타협이다

**서버의 과도한 커넥션 차단**:
- 서버는 특정 클라이언트가 과도하게 많은 커넥션을 맺으면 이를 임의로 끊을 수 있다
- 서버 측에서 `RST(reset)` 패킷을 보내 강제로 TCP 커넥션을 종료한다
- `RST`를 받은 클라이언트의 소켓은 즉시 닫히며, 진행 중이던 데이터 전송은 중단된다
- 이는 DoS(Denial of Service) 방어와 서버 자원 보호를 위한 조치이다

## 지속 커넥션

### HTTP/1.0+의 Keep-Alive 커넥션

HTTP/1.0은 기본적으로 요청마다 새 커넥션을 맺고 끊는 방식이다. 하지만 비공식 확장으로 **Keep-Alive**가 도입되었다.

**동작 방식**:
- 클라이언트가 `Connection: keep-alive` 헤더를 요청에 포함한다
- 서버가 지원하면 응답에도 `Connection: keep-alive`를 포함해 커넥션을 유지한다
- 서버가 지원하지 않으면 응답에 해당 헤더가 없고, 클라이언트는 커넥션을 닫는다

**Keep-Alive 옵션 헤더 예시**:
```
Keep-Alive: max=5, timeout=120
```
- `timeout`: 커넥션을 몇 초 동안 유지할지
- `max`: 이 커넥션으로 최대 몇 번의 트랜잭션을 처리할지

**제약**:
- HTTP/1.0 Keep-Alive는 프록시와 함께 사용할 때 문제가 발생한다. 프록시가 `Connection` 헤더를 이해하지 못하고 다음 서버로 그대로 전달하면(멍청한 프록시), 서버는 커넥션이 유지된다고 생각하지만 클라이언트는 프록시가 끊었다고 생각해 혼선이 생긴다.
- 이를 방지하기 위해 `Proxy-Connection` 헤더가 임시방편으로 사용되기도 했다.

### HTTP/1.1의 지속 커넥션

HTTP/1.1에서는 **지속 커넥션이 기본값**이다. 별도로 명시하지 않아도 커넥션은 유지된다.

- 커넥션을 끊고 싶을 때 `Connection: close` 헤더를 명시한다
- Keep-Alive처럼 별도 헤더로 활성화할 필요가 없다
- 모든 커넥션은 기본적으로 persistent이므로, 프록시와의 호환성 문제도 줄어든다

## 지속 커넥션의 제한과 규칙

지속 커넥션을 사용할 때 지켜야 하는 규칙과 제약이 있다.

1. **응답 길이를 알 수 있어야 한다**: 클라이언트가 언제 응답이 끝났는지 알아야 커넥션을 재사용할 수 있다. 이를 위해 `Content-Length` 헤더나 청크 전송 인코딩(Chunked Transfer Encoding)을 사용해야 한다.

2. **커넥션 헤더**: `Connection: close`가 포함된 요청/응답 이후에는 커넥션을 재사용해서는 안 된다.

3. **요청 수 제한**: 서버는 언제든지 커넥션을 끊을 수 있다. 클라이언트는 필요하다면 다시 커넥션을 맺어야 한다.

4. **클라이언트 파이프라이닝**: 응답을 받기 전에 여러 요청을 보낼 수 있지만, 서버가 지원하지 않거나 커넥션이 끊기면 클라이언트는 재요청을 할 수 있어야 한다.

5. **멱등성 있는 요청만 재시도**: 커넥션이 끊겼을 때 자동으로 재시도할 수 있는 요청은 `GET`, `HEAD` 등 멱등성이 있는 메서드에 한한다. `POST`처럼 멱등성이 없는 요청은 자동 재시도하면 안 된다.

6. **하나의 클라이언트는 서버에 두 개 이하의 지속 커넥션**만 유지해야 한다 (HTTP/1.1 스펙 권고).

## 파이프라인 커넥션

HTTP/1.1의 지속 커넥션을 활용하면 응답을 기다리지 않고 여러 요청을 연속으로 보낼 수 있다. 이를 **파이프라이닝(pipelining)** 이라 한다.

```
일반 지속 커넥션:   요청1 → 응답1 → 요청2 → 응답2 → ...
파이프라인 커넥션:  요청1 → 요청2 → 요청3 → 응답1 → 응답2 → 응답3
```

**장점**:
- 요청-응답의 왕복 대기 시간을 줄여 전체 레이턴시를 낮출 수 있다

**제약사항**:
- HTTP 응답은 요청 순서와 동일한 순서로 반환되어야 한다 (서버는 요청 순서를 바꿔 응답할 수 없음)
- 지속 커넥션이 수립된 상태에서만 파이프라이닝을 사용할 수 있다
- 멱등성이 없는 요청(POST 등)은 파이프라이닝해서는 안 된다. 재시도가 필요할 경우 부작용이 발생할 수 있기 때문이다
- 커넥션이 중간에 끊기면 완료되지 않은 요청을 다시 보낼 준비가 되어 있어야 한다
- **HOL(Head-of-Line) Blocking**: 앞 요청의 응답이 늦어지면 뒤 요청의 응답도 함께 지연되는 문제가 있다. 이 문제는 HTTP/2의 멀티플렉싱으로 해결되었다

## 우아하게 커넥션 끊기

TCP 커넥션에는 **입력 채널**과 **출력 채널** 두 개가 있다. 커넥션을 끊는 방식에는 두 가지가 있다.

### 전체 끊기 vs 절반 끊기

- **전체 끊기(full close)**: `close()` 호출. 입력과 출력 채널 모두 닫힌다.
- **절반 끊기(half close)**: `shutdown()` 호출. 입력 또는 출력 채널 중 하나만 닫을 수 있다.

### 우아한 커넥션 종료 절차

갑작스럽게 `close()`를 호출하면 상대방이 아직 읽지 않은 데이터가 있는 경우 `Connection reset by peer` 에러가 발생할 수 있다. 이를 방지하기 위한 우아한 종료 방법은 다음과 같다.

1. **출력 채널 먼저 닫기**: `shutdown()`으로 출력(쓰기) 채널만 먼저 닫는다. 상대방에게 더 이상 데이터를 보내지 않겠다는 신호를 보낸다.
2. **데이터 읽기 계속**: 상대방이 아직 보내고 있는 데이터가 있을 수 있으므로, 상대방이 출력 채널을 닫을 때까지 데이터를 계속 읽는다 (버리더라도).
3. **완전히 닫기**: 상대방도 출력 채널을 닫으면(EOF 수신), 그때 `close()`로 완전히 종료한다.

```
클라이언트                       서버
    |                             |
    |--shutdown(출력 채널 닫기)-->|
    |<--남은 데이터 수신----------|
    |<--EOF(서버 출력 채널 닫힘)--|
    |--close()-------------------->|
```

이 방식을 통해 양쪽 모두 데이터 유실 없이 커넥션을 안전하게 종료할 수 있다.
