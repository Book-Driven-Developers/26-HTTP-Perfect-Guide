## TCP 커넥션

**전 세계 모든 HTTP 통신은 패킷 교환 네트워크 프로토콜들의 계층화된 집합인 tcp/ip를 통해 이뤄진다.**

일단 TCP 커넥션을 맺으면 클라이언트 서버 간 주고 받는 메시지는 손실/손상되지 않으며 순서가 유지되며 안전하게 전달된다.

### 신뢰할 수 있는 데이터 전송 통로

- 한 쪽에 있는 바이트들은 반대쪽으로 순서에 맞게 정확히 전달된다.

### TCP 스트림은 세그먼트로 나뉘어서 IP 패킷을 통해 전송된다.

- 데이터 스트림을 세그먼트라는 단위로 잘게 나눈다.
- 세그먼트는 IP 패킷이라고 불리는 봉투에 담긴다.
- 해당 봉투(데이터)를 인터넷을 통해 전달한다.

- TCP 세그먼트는 하나의 IP 주소에서 다른 IP 주소로 IP 패킷이 담겨서 전달된다.
  - IP 패킷 헤더
  - TCP 세그먼트 헤더
  - TCP 데이터 조각

IP 패킷 헤더

- 발신지, 목적지 IP 주소, 크기, 기타 플래그
  TCP 세그먼트 헤더
- TCP 포트 번호, TCP 제어 플래그, 데이터 순서와 무결성을 검사하기 위한 숫자

### TCP 커넥션 유지하기

컴퓨터는 항상 TCP 커넥션을 여러개 갖고 있다. TCP는 포트 번호를 통해서 여러개의 커넥션을 유지한다.

식별값 구성: `<발신지 IP 주소, 발신지 포트, 수신지 IP 주소, 수신지 포트>`

### TCP 소켓 프로그래밍

- 운영체제는 TCP 커넥션의 생성과 관련된 기능을 API로 제공함.
  - socket(): 생성
  - bind(): 포트 번호 할당
  - connect(): 로컬의 소켓과 원격 호스트/포트 간 커넥션 생성
  - listen(): 커넥션 연결 요청 허용
  - accept(): 커넥션 연결 요청 대기
  - 등등
- 소켓 API는 유닉스 운영체제용으로 개발되었지만 현재는 대부분의 운영체제와 프로그램 언어에서 활용할 수 있음.
- 소켓 API를 통해 TCP와 IP의 세부 사항을 숨긴다.

소켓 API를 사용함으로서 TCP 종단 간 데이터 구조를 생성하고 원격 서버의 TCP 종단에 그 데이터 구조를 연결하여 데이터 스트림을 읽고 쓸 수 있다.

즉, 소켓 API를 사용하면 깡통 + 실 전화기 같이 나와 상대간에 같은 구조의 깡통과 실을 연결하여 목소리(데이터)를 연속적으로(스트림) 통신할 수 있다는 뜻이다.

1. TCP 종단 데이터 구조를 생성 = 내 쪽에 깡통 하나를 준비한다
2. 원격 서버의 TCP 종단에 연결 = 그 깡통을 실로 상대방 깡통과 연결한다
3. 데이터 스트림을 읽고 쓸 수 있다 = 이제 그 깡통에 대고 말하면(write) 상대가 듣고, 상대가 말하면 내 깡통에서 들린다(read)

여기서 소켓이 깡통 역할이다. 운영체제가 관리하는 통신용 끝단(endpoint)이고, 그 깡통의 핸들(파일 디스크립터)을 쥐고 있을 뿐이다. 실(TCP 연결)을 직접 다루지 않는다. 해당 영역은 OS의 TCP 스택이 알아서 연결해준다.

```
// 1. 깡통 만들고 상대방 깡통이랑 실로 연결 (handshake)
Socket socket = new Socket("joes-hardware.com", 80);

// 2. 깡통에 대고 말하기 (write)
socket.getOutputStream().write("GET / HTTP/1.1\r\n...".getBytes());

// 3. 상대 깡통에서 들려오는 소리 듣기 (read)
socket.getInputStream().read(buffer);
```

위 TCP API (소켓 API)에 기본적인 네트워크 프로토콜의 3-way handshake, TCP 데이터 스트림과 IP 패킷 간의 분할 및 재조립에 관한 세부사항이 숨겨진다.

## TCP 성능에 대한 고려

### HTTP 트랜잭션 지연

- DNS 찾기, 클라 -> 서버 연결, 서버 -> 클라 연결, 클라 -> 서버 요청, [실제 서버 처리], 서버 -> 클라 응답, 종료
  - 트랜잭션을 처리하는 시간은 TCP 커넥션을 설정하고, 요청을 전송하고, 응답을 수신하는것에 비하면 상당히 짧다.
  - 대부분의 HTTP 지연은 TCP 네트워크 지연 때문에 발생한다.

- 로컬 캐시에 없다면 DNS에서 IP 주소 & 포트를 알아내는데 수 십초의 시간이 소요됨.

### 성능 관련 중요 요소

- TCP 커넥션의 핸드쉐이크 설정
- TCP의 편승(piggiyback) 확인 응답(ack)을 위한 확인 응답 지연 알고리즘
- 인터넷의 혼잡을 제어하기 위한 TCP의 느린 시작 (slow-start), congestion control, 혼잡 제어
- 데이터를 한데 모아 한번에 전송하기 위한 네이글 알고리즘
- TIME_WAIT 지연과 포트 고갈

  1.TCP 커넥션 핸드쉐이크 지연

- 어떤 데이터를 전송하든 새로운 TCP 커넥션을 열 때 TCP 소프트웨어는 커넥션을 맺기 위해 연속으로 IP 패킷을 교환한다.

즉, 데이터 한 바이트라도 보내려면 서로 "연결할 준비됐어?" 하고 확인하는 절차를 먼저 거쳐야하는데, 이게 3-way handshake 비용이고 아래와 같은 절차가 이뤄진다.

```
[클라이언트]                           [서버]
    │                                    │
    │  ① "연결할까?" (SYN)              │
    │ ─────────────────────────────────▶│
    │                                    │
    │  ② "응 좋아, 너도 준비됐어?"     │
    │     (SYN + ACK)                    │
    │ ◀─────────────────────────────────│
    │                                    │
    │  ③ "준비됐어, 시작하자" (ACK)    │
    │ ─────────────────────────────────▶│
    │                                    │
    │  ④ "GET /index.html ..." (DATA)   │
    │ ─────────────────────────────────▶│
```

데이터를 서버로 보내는데 1-RTT가 그대로 날라가는것이다.
왕복 통신에 150ms가 소요된다고 하면 아래와 같이 시간이 잡아 먹힌다.

```
SYN 보냄       → 75ms (한 방향 가는 시간)
SYN+ACK 받음   → 75ms (돌아옴)
ACK + 첫 데이터 → 75ms (가는 시간, 여기에 데이터 얹어 보냄)
```

2.확인응답 지연 (Delayed ACK)

ACK를 바로 주는게 아니라 응답을 줄 때까지 좀 기다렸다가 편승해서(piggyback)해서 같이 보내는것이다.

방식 A: 즉시 답

```
친구: "오늘 저녁 뭐 먹지?"
나:   "읽음"     ← ACK만 따로
나:   "치킨 어때?" ← 진짜 답
```

메시지 2개를 보냄. 응답 흐름이 2번 이뤄짐.

방식 B: 잠깐 기다렸다 같이 답

```
친구: "오늘 저녁 뭐 먹지?"
나:   "치킨 어때?"   ← 이 안에 "읽었어"가 이미 포함
```

근데 아래와 같은 응답 데이터 생성이 느린 상황에서는 오히려 성능 저하 이슈가 발생한다.

```
[클라이언트]  GET /api/user  ────────▶  [서버]
                                          (잘 받았는데... 곧 응답 보낼 거니까
                                           ACK 잠깐 묶어서 보내자)
                                          ⏳ 100~200ms 대기
                                          🤔 어? 응답 데이터가 아직 준비 안 됐네
                                          📨 어쩔 수 없이 ACK만 따로 전송
```

즉, 서버 쪽에서 응답을 빨리 못 만들면, 결국 기다린 시간만 손해가 발생한다.
만약 Nagle 알고리즘(작은 패킷을 모아서 보내는 또 다른 최적화)과 만나면 둘이 서로 기다리느라 최악의 경우 200ms씩 멈춰버리는 현상까지 생긴다.
이를 "Nagle + Delayed ACK 데드락"이라고 한다.

3.TCP 느린 시작 (slow start)

네트워크의 갑작스러운 부하와 혼잡을 방지하기 위해 TCP는 커넥션에 처음부터 최대의 전송 속도로 전달하지 않는다. 패킷이 성공적으로 전달될 때마다 패킷을 더 전송할 수 있는 권한을 얻는다.

이를 congestion control, 혼잡제어라고 하며 새로운 커넥션이 어느정도 데이터를 주고받은 튜닝된 커넥션보다 느리게 되는것이다. 따라서 튜닝된 커넥션이 더 빠르기 때문에 HTTP는 이미 존재하는 TCP 커넥션을 재사용하는 기능이 존재한다.

4.네이글(Nagle) 알고리즘과 TCP_NODELAY

애플리케이션이 단 1바이트의 데이터도 TCP 스택으로 전송할 수 있도록 TCP는 데이터 스트림 인터페이스를 제공한다. 다만, TCP가 작은 크기의 데이터를 포함해 많은 수의 패킷을 전송하면 네트워크 성능이 크게 떨어진다.

네이글 알고리즘은 네트워크 효율을 위해 패킷을 전송하기 전에 많은 양의 TCP 데이터를 한개의 덩어리로 합친것이다.

알고리즘 동작은 세그먼트가 최대 크기(1500바이트 정도)가 되지 않으면 전송을 안하는것이다. 단, 다른 모든 패킷이 ACK 응답을 받았다면 최대 크기보다 작더라도 패킷의 전송을 허용한다. 아직 다른 패킷들이 전송중이라면 전송 대상은 버퍼에 저장된다.

문제점

1. 크기가 작은 HTTP 메시지는 패킷을 채우지 못해서 앞으로 생길지 / 생기지 않을지 모르는 추가적인 데이터를 기다리면서 지연된다.
2. 확인 응답 지연과 함께 사용되면 형편없다. 확인 응답 지연은 ACK 응답을 100~200ms 지연 시키면서 주는건데 (응답줄 때 같이 주려고), 네이글 알고리즘은 확인 응답이 올 때까지 데이터 전송을 대기하고 있는것이다.

5.TIME_WAIT의 누적과 포트 고갈

- 연결을 닫아도 OS는 그 포트를 일정시간 동안 못 놓아서 가용 포트가 동나는 현상이다.

TCP 연결을 끊으면 커넥션이 즉시 깔끔하게 사라지는 것이 아니라, OS는 그 연결을 일정 시간(보통 60~120초) 동안 쓰지도 못하고 버리지도 못하는 상태로 잡고 있는다.
이러한 상태가 TIME_WAIT 상태이다. 짧은 연결을 폭발적으로 많이 맺는 서버는 이 잔재가 쌓이다가 결국 새 연결을 못 만드는 사고가 난다.

연결을 닫는 4-way handshake

```
[클라이언트]                            [서버]
    │  ① "이제 그만 보낼게" (FIN)      │
    │ ─────────────────────────────────▶│
    │                                    │
    │  ② "알겠어" (ACK)                 │
    │ ◀─────────────────────────────────│
    │                                    │
    │  ③ "나도 다 끝났어" (FIN)         │
    │ ◀─────────────────────────────────│
    │                                    │
    │  ④ "그래, 잘 가" (ACK)            │
    │ ─────────────────────────────────▶│
    │                                    │
    │  🕐 TIME_WAIT (2 × MSL 동안 대기) │
    │  보통 60초 ~ 120초                  │
```

여기서 먼저 FIN을 보낸 쪽(능동 종료 측)이 TIME_WAIT 상태가 된다.

TIME_WAIT을 하는 이유.

1. 마지막 ACK가 분실됐을 때 대비
   - 내가 ④번 "잘 가" ACK를 보냈는데 네트워크에서 사라졌다고 가정해보자. 서버는 ACK를 못 받으니까 "아직 못 받았나?" 하고 FIN을 한 번 더 보낸다.
   - 근데 이미 커넥션을 정리했다면 서버는 영영 ACK를 못받게 되고 좀비 커넥션으로 남게 된다.
2. 옛 연결의 패킷이 새 연결로 흘러드는 사고 방지
   - 네트워크 어딘가에서 길을 헤매고 있던 옛날 패킷이 뒤늦게 도착할 수 있다.
   - 만약 같은 포트로 새 연결이 바로 맺어졌으면, 그 옛 패킷이 새 연결의 데이터로 잘못 해석될 수 있다.

- 일반적으로 Segment의 Max Lifetime의 2배로 60~120초로 설정한다.

## HTTP 커넥션 관리

커넥션을 생성하고 최적화하는 HTTP 기술

### 흔히 잘못 이해하는 Connection 헤더

- HTTP 메시지는 클라에서 서버(혹은 리버스 프록시 서버)까지 중개 서버들을 하나하나 거치면서 전달된다.
  - 중개 서버: 프락시 서버, 캐시 서버 등
- 이때 인접한 애플리케이션 간 커넥션에서만 적용될 옵션을 지정해야할 때가 있다. 이러한 값들은 HTTP Connection 필드가 쉼표로 구분하여 갖는다.
  - Connection: meter, close

HTTP 헤더에는 두 종류가 있다.

| 종류       | 의미                                    | 예시                                                              |
| ---------- | --------------------------------------- | ----------------------------------------------------------------- |
| End-to-End | 최종 수신자까지 전달됨                  | `Content-Type`, `Authorization`, `User-Agent`                     |
| Hop-by-Hop | **바로 다음 노드까지만**, 거기서 제거됨 | `Connection`, `Keep-Alive`, `TE`, `Upgrade`, `Proxy-Authenticate` |

- `Connection` 헤더는 hop-by-hop이다.
- 클라이언트 → 프록시 구간의 약속이지, 클라이언트 → 최종 서버의 약속이 아니다.

`Connection: keep-alive`도 마찬가지이다. "나(클라이언트)랑 너(프록시 사이의 연결을 끊지 말자" 라는 hoy by hop 약속이지, "나랑 최종 서버 사이의 연결을 끊지 말자"가 아니다.

### 순차적인 트랜잭션 처리에 의한 지연

이미지가 3개가 들어있는 웹페이지라고 생각해보자. 이를 순차적으로 로드하는 방식은 단점이 많다.

- 지연이 발생한다.
- 빈 화면을 보여줘야한다.
  이를 개선하기 위한 HTTP 커넥션의 성능 향상 기술 4가지를 알아보자.

### 1. 병렬(Parallel) 커넥션

- 병렬 커넥션은 페이지를 더 빠르게 내려받는다.
  - 각 커넥션의 지연 시간을 겹치게 하면 총 지연 시간을 줄일 수 있다.
  - 인터넷 대역폭을 한개의 커넥션이 다 쓰는게 아니라면 나머지 다운로드에 남은 대역폭을 활용할 수 있다.
- 벙렬 커넥션이 항상 더 빠르지는 않다.
  - 클라이언트 대역폭이 좁다면 커넥션 대부분의 시간을 데이터를 전송하는데 사용하게 된다.
  - 대역폭이 제한되어있다면 객체를 전송받는 것도 느려져서 병렬이 성능상 장점이 거의 없어진다.
- 병렬 커넥션이 더 빠르게 느껴질 수 도 있다.
  - 화면에 여러개의 객체가 동시에 보이면서 내려받는 상황을 볼 수 있기 때문에 더 빠르게 내려받고 있다고 느 낄수 있다.

### 2. 지속(Persistent) 커넥션

지속 커넥션: 처리가 완료된 후에도 계속 연결된 상태로 있는 TCP 커넥션

- 클라이언트나 서버가 커넥션을 끊기 전까지는 트랜잭션 간에도 커넥션을 유지한다.
- 이미 맺어진 커넥션을 재사용하므로 커넥션을 맺기 위한 준비작업 시간을 아낄 수 있다.
- TCP의 느린 시작(slow start)으로 인한 지연을 피할 수 있다.

병렬 커넥션만 사용해서는 한계가 있다. 따라서 병렬 + 지속 커넥션을 함께 사용하는게 효과적이다.

- 오늘날 많은 웹 애플리케이션이 적은 수의 병렬 커넥션만 맺고 지속 커넥션으로 사용한다.

**HTTP 1.0+**

- Keep-Alive 커넥션
- 해당 설정이 1.1 버전에서는 제거되었다. 대신 기본적으로 Keep Alive 한다. 그렇다고 그것이 영원히 커넥션을 유지한다는걸 의미하지는 않는다.
- 커넥션을 유지하기 위해서는 `Connection: Keep-Alive` 헤더가 있어야 한다.
  - 헤더가 없으면 클라이언트는 서버가 응답 메시지가 전송되고 나서 서버 커넥션을 끊을거라 추정한다.

**Keep-Alive 옵션**

- 이건 커넥션을 유지하기를 바라는 요청이지, 무조건 유지해야하는 명령은 아니다.
- 언제든지 커넥션을 끊을 수 있고 keep alive 커넥션에서 처리되는 트랜잭션의 수도 제한할 수 있다.

```
Connection: Keep-Alive
Keep-Alive: max=5, timeout=120
```

- 위 설정은 전부 요청이지 명령은 아니다. 동작을 보장하는건 아님.

**Keep-Alvie 헤더 + 멍청한 프록시 문제**

- Connection 헤더는 hope-by-hope이라 다음 홉으로 전달되면 안된다. 하지만 멍청한 프록시 서버는 Keep-Alive를 인지하지 못하고 그대로 전달할 수 있다.
- 그러면 아래와 같은 데드락이 발생할 수 있다. (서버/클라 타임아웃 될때까지 대기함)

```
[클라이언트]              [프록시]              [서버]
  keep-alive 지원         keep-alive 미지원      keep-alive 지원
       │                       │                     │
       │  GET / HTTP/1.1       │                     │
       │  Connection:          │                     │
       │   keep-alive          │                     │
       │──────────────────────▶│                     │
       │                       │                     │
       │                       │  ⚠️ 멍청한 프록시:  │
       │                       │  헤더 의미를 모르고 │
       │                       │  그대로 전달        │
       │                       │                     │
       │                       │  GET / HTTP/1.1     │
       │                       │  Connection:        │
       │                       │   keep-alive        │
       │                       │────────────────────▶│
       │                       │                     │
       │                       │                     │ 🤔
       │                       │                     │ "프록시가
       │                       │                     │  keep-alive
       │                       │                     │  원하는구나"
       │                       │  HTTP/1.1 200 OK    │
       │                       │  (응답 후 연결 유지)│
       │                       │◀────────────────────│
       │                       │                     │
       │                       │  💀 데드락          │
       │                       │  - 프록시: 서버가   │
       │                       │    연결 닫기를 기다림│
       │                       │  - 서버: 다음 요청을│
       │                       │    기다림           │
       │                       │  - 클라이언트: 응답 │
       │                       │    을 기다림        │
```

-> 이러한 문제를 해결하고자 Proxy-Connection 헤더가 나왔다.
근데 이것도 멍청한 프록시 하나만 있어서 keep alive가 적용이 안되던가, 영리한 프록시 하나만 있어서 keep alive가 적용되어야 하는데, 멍청한 프록시 좌/우로 영리한 프록시가 하나 더 있게 되면 동일한 문제가 발생한다.

**HTTP 1.1**

- Connection: keep-alive 을 지원하지 않는다.
- 대신 비존적으로 지속 커넥션이 활성화되어있다.
- 명시적으로 Connection: close를 해야 커넥션이 종료된다.

### 3. 파이프라인(pipelined) 커넥션

HTTP 1.1 에서 지속 커넥션을 통해서 요청을 파이프라이닝 할 수 있다.

- keep-alive 성능을 더 높여준다.
- 여러개의 요청은 응답이 도착하기 전까지 큐에 쌓인다.

제약사항

- 지속 커넥션인지 확인이 되고 파이프라인을 이어야 한다
- 응답은 요청 순서와 같게 와야한다. HTTP 메시지에 번호가 있진 않아서 순서 없이 오면 정렬할 방법이 없다.
- 커넥션이 언제 끊어지더라도 다시 요청을 보낼 준비가 되어야 한다.
- POST 요청같이 비멱등 요청은 파이프라인으로 보내면 안된다. 파이프라인 요청 중 에러가 발생하면 어떤 요청이 서버에서 처리되었는지 클라이언트에서 알 수가 없다.

## 커넥션 끊기에 대한 미스터리

커넥션 관리(특히 언제 어떻게 커넥션을 끊는가)에는 명확한 기준이 없다.

**Content-Length와 Truncate**

- 각 HTTP 응답은 본문의 정확한 크기 값을 가지는 Contnet-Length 헤더를 갖고 있어야 한다.
- 커넥션이 끊어졌다는 HTTP 응답을 받고 전달된 엔터티의 길이와 Content-Length가 다르면 수신측은 데이터의 정확한 길이를 서버에 물어봐야 한다.

**우아한 커넥션 끊기**

- close: 전체 끊기 (입력 채널/출력 채널 둘다 한번에 끊음)
- shutdown: 절반 끊기

보통 출력채널 부터 끊고 입력 채널을 끊는게 안전하다.
이미 끊긴 입력 채널에 데이터를 전송하면 TCP 'connection reset by peer' 오류가 발생한다

아래와 같은 방식으로 이뤄진다.

```
shutdown(sock, SHUT_WR);   // 쓰기 채널만 닫음 (FIN 보냄)
shutdown(sock, SHUT_RD);   // 읽기 채널만 닫음
close(sock);               // 양쪽 다 닫고 소켓 자체를 해제
```

통신 흐름

```
[클라이언트]                              [서버]
    │  ① 요청 전송 완료                    │
    │  ② shutdown(SHUT_WR)                 │
    │     "나 송신 끝. 너는 계속 보내"     │
    │ ─── FIN ──────────────────────────▶ │
    │                                      │
    │                                      │ ③ "아 클라이언트가 송신 끝났구나
    │                                      │     난 응답 마저 보내야지"
    │                                      │
    │  ④ 응답 데이터 계속 수신             │
    │     read() 반복 호출                 │
    │ ◀───────────────────────────────────│
    │                                      │
    │                                      │  ⑤ 응답 전송 완료
    │                                      │     서버도 close()
    │ ◀── FIN ────────────────────────────│
    │                                      │
    │  ⑥ read()가 0 반환 (EOF)             │
    │     "아 서버도 끝났구나"             │
    │  ⑦ close()로 완전히 정리             │
```

- shutdown(SHUT_WR)
- 이걸로 출력은 종료되었다고 알려주고 읽기는 아직 열려있도록 하는게 핵심이다.
