### HTTP

- 신뢰성 있는 데이터 전송 프로토콜
- 신뢰성 있는 전송으로 HTTP 통신의 파괴, 중복, 왜곡을 방지한다.

### 클라이언트/서버

클라이언트와 서버는 HTTP를 이용하야 통신한다.

- HTTP 서버라고도 불린다.

### 리소스

웹서브는 요청 대상인 웹 콘텐츠의 원천인 리소스를 관리한다.

- 이미지, 파일 등 정적파일부터 검색, 커머스 등 모든것들이 리소스다.

인터넷은 수천가지의 타입을 다루고 HTTP는 이를 MINE 타입을 통해 관리한다.
[MINE 타입]

```
content-type: [application/json] <-- 요거
content-length: 1024
```

- text/plain
- text/html
- image/jpg
- ...

리소스는 각자 이름을 갖고 있고 클라이언트는 지목할 수 있다.
서버 리소스 이름은 통합 자원 식별자로 URI 로 관리된다.

URI 는 URL, URN으로 구분된다.

URL

- 통합 자원 지시자 (Locator)
- 특정 서버의 한 리소스에 대한 구체적인 위치를 가르킨다.
- 구성: [프로토콜].[도메인 주소].[리소스 이름]

URN

- 통합 리소스 이름 (Name)
- 잘 안쓴다.
- 리소스가 이름을 바꾸지 않는한 어떤 인터넷 프로토콜로도 접근이 가능하다.

### 트랜잭션

- 클라이언트와 서버가 리소스를 주고받기 위해 HTTP 메시지를 통한 상호작용
- HTTP 메시지란?

메서드

- HTTP는 여러 종류의 요청 명령을 HTTP 메서드 형태로 제공한다.
- HTTP 요청 메시지는 한 개의 HTTP 메서드를 갖는다.
  - 대표 종류: GET, PUT, DELETE, POST, HEAD

상태코드

- 모든 HTTP 응답 메시지는 상태코드와 함께 반환된다.
- 클라이언트에게 성공 (성공해도 여러 방식으로)부터 추가 조치사항까지 알려준다.
- 3자리 숫자로 구성된다.
- HTTP는 숫자 상태 코드에 텍스트로 "사유 구절" (reason phrase)도 함께 보낸다.

### 메시지

- 요청 메시지, 응답 메시지 2개가 존재한다.
- 구성 - 시작줄 - 헤더 - 본문 (GET에는 없음)
  시작줄: 뭘 해야하는지 / 뭘 해줬는지 알려준다 (형식 구체적)
  헤더: 요청 리소스 타입(MINE) 등이 포함된다. (문자열)
  본문: 자유로운 타입이 올 수 있다. 이진 데이터 등

### TCP 커넥션

- HTTP 메시지가 어떻게 전송 제어 프로토콜을 통해 옮겨가는지

TCP/IP

- HTTP는 OSI 계층 중 애플리케이션 계층이다. 네트워크 통신의 핵심적인 세부사항에 대해서는 신경쓰지 않는다. - 이를 신뢰성있는 인터넷 전송 프로토콜인 TCP/IP에 위임한다.
  TCP가 제공하는 것들
- 오류 없는 데이터 전송
- 순서에 맞는 전달
- 조각나지 않는 데이터 스트림

HTTP는 TCP위에서 동작하고, TCP는 IP위에서 동작한다.
HTTP가 통신하기 위해서는 TCP 커넥션을 맺어야하고, TCP 커넥션을 맺기 위해서는 IP:PORT를 알아야한다.
IP는 어떻게 알지? -> URL을 아니깐 이걸 이용해서 IP로 변환한다. DNS를 이용한다.
PORT는 어떻게 알지? -> 비어있다면 HTTP인 경우 묵시적으로 80 포트를 이용한다.
