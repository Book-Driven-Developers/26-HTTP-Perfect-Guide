# 1.6 딥다이브 — HTTP는 왜 직접 신뢰성을 구현하지 않고 TCP에 위임했는가

> HTTP는 신뢰성 있는 데이터 전송을 "직접 구현"하지 않는다.
> 그 책임을 TCP에 위임하고, 자신은 "무엇을, 어떻게 요청하고 응답할 것인가"에만 집중한다.
> 이 위임이 웹 전체 아키텍처의 설계 결정이었다.

---

## 1. 질문을 뒤집어보기 — HTTP가 직접 신뢰성을 구현했다면?

이 질문을 이해하는 가장 빠른 방법은 반대 상황을 상상해보는 것이다.
만약 HTTP 메시지가 신뢰성을 직접 책임졌다면, 헤더에 이런 필드들이 있었을 것이다.

```http
GET /index.html HTTP/1.0
Host: www.example.com
Sequence-Number: 1042
Ack-Number: 981
Checksum: 0xA3F2
Window-Size: 65535
Congestion-Window: 10
```

이것이 왜 이상한지를 이해하면, 위임의 이유가 보인다.

---

## 2. 핵심 원칙 — 계층화 아키텍처

인터넷은 **TCP/IP 4계층 모델** 위에 설계되어 있다.

```
[ 애플리케이션 계층 ]  ← HTTP, FTP, SMTP, DNS ...
[ 전송 계층 ]         ← TCP, UDP
[ 인터넷 계층 ]       ← IP
[ 네트워크 접근 계층 ] ← Ethernet, Wi-Fi
```

이 모델의 핵심 원칙은 다음과 같다.

> **각 계층은 하위 계층을 신뢰하고, 자신의 역할에만 집중한다.**

HTTP는 애플리케이션 계층이다. TCP는 전송 계층이다.
HTTP가 TCP의 책임인 신뢰성 보장을 직접 구현하는 것은, 계층 분리를 위반하는 것이다.

---

## 3. Java 인터페이스로 이해하는 위임

계층화 원칙은 Java의 인터페이스(Interface) 설계와 정확히 같은 구조다.

```java
// TCP가 구현하는 인터페이스
interface ReliableTransport {
    void send(byte[] data);     // 순서 보장, 손실 재전송
    byte[] receive();           // 순서 복원, 중복 제거
    void controlFlow();         // 흐름 제어
    void controlCongestion();   // 혼잡 제어
}

// HTTP는 이 인터페이스에만 의존한다
class HTTP {
    private ReliableTransport transport; // TCP 주입

    void get(String url) {
        transport.send("GET " + url + " HTTP/1.1\r\n...");
        // 신뢰성은 transport의 책임이다
        // HTTP는 어떤 데이터를 보낼지만 결정한다
    }
}
```

HTTP는 "데이터가 신뢰성 있게 도달할 것"이라는 **계약(Contract)** 을 TCP와 맺는다.
구현은 TCP가 책임진다.

---

## 4. HTTP의 실제 역할

신뢰성을 TCP에 위임함으로써, HTTP는 자신의 역할에 집중할 수 있게 된다.

| HTTP의 책임 | 해당 구성 요소 |
|-------------|---------------|
| 무엇을 요청하는가 | 메서드 (GET, POST, PUT...) |
| 어떤 리소스를 요청하는가 | URL |
| 어떤 형식으로 주고받는가 | MIME 타입, Content-Type 헤더 |
| 요청/응답의 의미 | 상태코드 (200, 404, 500...) |

패킷이 손실됐는지, 순서가 맞는지 — 이것은 HTTP의 관심사가 아니다.

---

## 5. 재사용성 — TCP 위에 HTTP만 있는 게 아니다

```
HTTP  ──┐
FTP   ──┤
SMTP  ──┤  모두 TCP 위에서 신뢰성을 재사용
SSH   ──┤
HTTPS ──┘
```

HTTP가 신뢰성을 자체 구현했다면, FTP도, SMTP도, SSH도 각자 신뢰성 로직을 구현해야 했다.
**중복 구현** 이 발생한다. TCP는 이 문제를 한 계층에서 한 번만 해결한다.

---

## 6. 이 설계의 비용 — 3-way handshake

계층 위임은 이점만 있는 게 아니다. 대가가 있다.

TCP는 연결을 수립할 때 반드시 **3-way handshake** 를 요구한다.

```
클라이언트                    서버
    │──── SYN ────────────────▶│   1. 연결 요청
    │◀─── SYN-ACK ─────────────│   2. 수신 확인 + 연결 수락
    │──── ACK ────────────────▶│   3. 확인
    │                          │
    │  (이제야 HTTP 요청 가능)  │
    │──── GET /index.html ────▶│   4. 실제 요청
```

HTTP 요청 하나를 보내기 위해, 왕복 1.5번의 네트워크 지연이 먼저 발생한다.
이것은 HTTP/1.0의 구조적 비용이었다. 매 요청마다 TCP 연결을 새로 맺었기 때문이다.

이 비용이 쌓이자 다음 해결책들이 등장했다.

| 해결책 | 방식 |
|--------|------|
| **Keep-Alive(지속 연결)** | 하나의 TCP 연결을 여러 요청이 재사용 |
| **HTTP/2 멀티플렉싱** | 하나의 TCP 연결에서 여러 요청을 동시에 처리 |
| **HTTP/3 (QUIC)** | TCP 대신 UDP를 선택, 연결 수립 비용 자체를 줄임 |

---

## 7. 역사적 맥락

HTTP(1989, Tim Berners-Lee)가 설계될 당시, TCP/IP는 이미 인터넷 표준으로 자리 잡혀 있었다.
기존 인프라를 활용하는 것이 합리적인 선택이었고, 신뢰성을 직접 구현할 이유가 없었다.
