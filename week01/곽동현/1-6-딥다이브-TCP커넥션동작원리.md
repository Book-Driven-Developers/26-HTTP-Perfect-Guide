# 1.6 딥다이브 — TCP 커넥션 동작 원리

> HTTP는 신뢰성을 TCP에 위임한다.
> 그렇다면 TCP는 그 신뢰성을 구체적으로 어떻게 구현하는가?
> 연결 수립부터 종료까지 전체 생명주기를 단계별로 살펴본다.

---

## 1. TCP가 다루는 데이터 단위 — 세그먼트

HTTP는 "메시지"를 주고받지만, TCP는 그것을 **세그먼트(Segment)** 단위로 쪼개 전송한다.

```
HTTP 요청 (수천 바이트)
    ↓ TCP가 분할
세그먼트 1: [헤더 | 데이터 1~500바이트]
세그먼트 2: [헤더 | 데이터 501~1000바이트]
세그먼트 3: [헤더 | 데이터 1001~1500바이트]
```

TCP 헤더의 핵심 필드:

```
┌──────────────────────┬──────────────────────┐
│  출발지 포트 (16bit)  │  목적지 포트 (16bit)  │
├──────────────────────┴──────────────────────┤
│              순서 번호 (32bit)               │  ← 신뢰성의 핵심
├─────────────────────────────────────────────┤
│           확인 응답 번호 ACK (32bit)          │  ← 신뢰성의 핵심
├────────────┬────────────────────────────────┤
│ 헤더 길이  │  SYN │ ACK │ FIN │ RST │ ...   │  ← 연결 제어 플래그
├────────────┴────────────────────────────────┤
│              윈도우 크기 (16bit)              │  ← 흐름 제어
├─────────────────────────────────────────────┤
│                체크섬 (16bit)                │  ← 오류 감지
└─────────────────────────────────────────────┘
```

## 2. 연결 수립 — 3-way handshake

TCP 커넥션을 맺기 전에 양쪽이 **서로의 준비 상태와 초기 순서 번호를 합의**해야 한다.

```
클라이언트                              서버
    │                                   │
    │──── SYN, seq=3000 ───────────────▶│
    │     "연결 원해. 내 순서 번호는 3000"│
    │                                   │
    │◀─── SYN-ACK, seq=7000, ack=3001 ──│
    │     "OK. 내 순서 번호는 7000,      │
    │      네 3000번 받았어 (ack=3001)"  │
    │                                   │
    │──── ACK, ack=7001 ───────────────▶│
    │     "네 7000번 받았어. 연결 완료." │
    │                                   │
    │   ← 여기서부터 HTTP 전송 가능 →    │
```

### 왜 2-way가 아니라 3-way인가?

2-way라면 클라이언트만 서버의 준비를 확인할 수 있다.
서버는 클라이언트가 자신의 응답(SYN-ACK)을 제대로 받았는지 알 수 없다.
TCP는 **양방향 통신**이므로 양쪽 모두 상대의 수신 능력을 확인해야 한다.

### 초기 순서 번호는 왜 랜덤인가?

항상 0부터 시작하면 두 가지 문제가 생긴다.

- **지연 패킷 오염**: 이전 커넥션의 늦게 도착한 패킷이 새 커넥션의 데이터로 오해될 수 있다.
- **보안**: 순서 번호가 예측 가능하면 제3자가 가짜 패킷을 끼워넣을 수 있다 (TCP 세션 하이재킹).

> **비유:** 처음 만나는 두 사람이 통화를 시작할 때 "내 대화 번호는 3000번부터야, 너는?"처럼
> 암호화된 번호를 먼저 교환한다. 이전 통화에서 늦게 도착한 메시지와 혼동하지 않기 위해서다.

## 3. 데이터 전송 — 순서 번호 + ACK + 재전송

### 순서 번호와 ACK의 의미

- **순서 번호**: 이 세그먼트의 첫 바이트가 전체 스트림에서 몇 번째인가
- **ACK 번호**: 이 번호 이전까지 다 받았어, 다음에 이 번호부터 줘

```
세그먼트 1: seq=3001, 500바이트 → 서버
서버 응답:  ACK=3501  ("3001~3500 받았어, 3501번 줘")

세그먼트 2: seq=3501, 500바이트 → 서버
서버 응답:  ACK=4001  ("3501~4000 받았어, 4001번 줘")
```

### 패킷 손실 시 — 빠른 재전송 (Fast Retransmit)

```
클라이언트                              서버
    │──── seq=3001 ─────────────────────▶│  → ACK=3501
    │──── seq=3501 ──────────✗ ──────────│  (손실)
    │──── seq=4001 ─────────────────────▶│
    │──── seq=4501 ─────────────────────▶│
    │                                   │
    │◀─── ACK=3501 ──────────────────────│  (중복 ACK 1)
    │◀─── ACK=3501 ──────────────────────│  (중복 ACK 2)
    │◀─── ACK=3501 ──────────────────────│  (중복 ACK 3)
    │                                   │
    │──── seq=3501 재전송 ──────────────▶│  ← 3중 ACK → 즉시 재전송
    │◀─── ACK=5001 ──────────────────────│  (버퍼에 있던 4001, 4501도 처리)
```

같은 ACK가 3번 연속 오면 타임아웃을 기다리지 않고 즉시 재전송한다.
이것이 **빠른 재전송(Fast Retransmit)** 이다.

## 4. 흐름 제어 — 슬라이딩 윈도우

수신 측이 처리하는 속도보다 빠르게 데이터가 쏟아지면 버퍼가 넘친다.
수신 측은 매 ACK마다 **"나 지금 이만큼 더 받을 수 있어"** 를 윈도우 크기로 알린다.

```
윈도우 크기 = 4 세그먼트

전송 가능 범위:
[1: 전송·미확인][2: 전송·미확인][3: 전송·미확인][4: 전송·미확인][5: 대기][6: 대기]

ACK=3 수신 → 윈도우 슬라이드:
[1: 확인][2: 확인][3: 전송·미확인][4: 전송·미확인][5: 전송 가능][6: 전송 가능]
```

윈도우가 앞으로 "슬라이드"하면서 전송 가능 범위가 이동한다.

> **비유:** 컨베이어 벨트에 4칸짜리 트레이가 있고, 앞에서 하나 처리될 때마다
> 뒤에서 새 물건 하나가 올라올 수 있다. 처리 속도에 맞춰 투입 속도가 자동 조절된다.

---

## 5. 혼잡 제어 — 네트워크 과부하 방지

흐름 제어가 수신 측 보호라면, **혼잡 제어는 네트워크 전체 보호**다.
TCP는 패킷 손실을 "네트워크가 혼잡하다"는 신호로 해석하고 전송 속도를 줄인다.

```
연결 초기 — Slow Start (지수 증가):
  윈도우: 1 → 2 → 4 → 8 → 16 ...

임계값(ssthresh) 도달 — Congestion Avoidance (선형 증가):
  16 → 17 → 18 → 19 ...

패킷 손실 감지:
  ssthresh = 현재 윈도우 / 2
  윈도우 = 1 → Slow Start 재시작
```

> **비유:** 고속도로 진입 시 서행하다가 점점 속도를 올린다.
> 앞에서 사고(패킷 손실)가 나면 급브레이크를 밟고 다시 서행부터 시작한다.

## 6. 연결 종료 — 4-way handshake

TCP는 양방향(Full-Duplex) 통신이다. 각 방향을 **독립적으로 닫는다.**

```
클라이언트                              서버
    │──── FIN ─────────────────────────▶│
    │     "나는 보낼 데이터 다 보냈어"   │
    │                                   │
    │◀─── ACK ───────────────────────────│
    │     "알겠어. 근데 나는 아직 있어"  │
    │          (서버가 마저 전송 중...)  │
    │◀─── FIN ───────────────────────────│
    │     "나도 다 보냈어"               │
    │                                   │
    │──── ACK ─────────────────────────▶│
    │     "알겠어. 연결 종료."           │
```

### 왜 3-way가 아니라 4-way인가?

연결 수립 시 서버는 SYN을 받는 순간 즉시 SYN+ACK를 함께 보낼 수 있었다.
하지만 종료 시 클라이언트의 FIN을 받더라도 **서버가 아직 보낼 데이터가 남아있을 수 있다.**
그래서 ACK와 FIN을 분리해서 전송한다. 반-닫힘(Half-Close) 상태가 존재하는 이유다.

---

## 7. TIME_WAIT

마지막 ACK를 보낸 클라이언트는 즉시 연결을 끊지 않고 **TIME_WAIT 상태**에서 대기한다.

```
클라이언트가 마지막 ACK 전송 후:
  → TIME_WAIT 진입 (2 * MSL, 보통 60초~240초)
  → 이 기간 동안 동일 포트 재사용 불가
  → TIME_WAIT 종료 후 완전히 CLOSED
```

### 왜 필요한가?

**이유 1 — 마지막 ACK 손실 대비**
클라이언트의 마지막 ACK가 유실되면 서버는 FIN을 재전송한다.
TIME_WAIT 없이 이미 연결을 끊었다면, 재전송된 FIN에 RST(비정상 종료)로 응답하게 된다.
TIME_WAIT 동안 대기하면 재전송된 FIN을 다시 ACK 할 수 있다.

**이유 2 — 지연 패킷 오염 방지**
네트워크에 떠돌던 이전 연결의 패킷이 같은 포트로 새 연결을 맺을 때 끼어드는 것을 막는다.

> HTTP/1.0처럼 요청마다 커넥션을 맺고 끊으면, TIME_WAIT 소켓이 수백 개씩 쌓인다.
> 이것이 HTTP/1.1 지속 커넥션 도입의 이유 중 하나다.

## 8. 전체 생명주기 한눈에

```
① DNS 조회: 도메인 → IP 변환

② 3-way handshake (1.5 RTT)
   SYN → SYN-ACK → ACK

③ 데이터 전송
   ├─ 세그먼트 분할 + 순서 번호 부여
   ├─ ACK 수신 전까지 재전송 대기
   ├─ 손실 시 빠른 재전송 (3 Dup ACK)
   ├─ 슬라이딩 윈도우로 흐름 제어
   └─ 혼잡 제어 (Slow Start → Congestion Avoidance)

④ 4-way handshake
   FIN → ACK → FIN → ACK

⑤ TIME_WAIT → CLOSED
```

## 9. Java 개발자 관점

```java
// Java의 Socket = TCP 커넥션 하나
Socket socket = new Socket("www.example.com", 80);
// → 내부적으로 3-way handshake 수행

OutputStream out = socket.getOutputStream();
out.write("GET / HTTP/1.1\r\nHost: www.example.com\r\n\r\n".getBytes());
// → TCP가 세그먼트로 분할, 순서 번호 부여, ACK 관리

InputStream in = socket.getInputStream();
// → TCP가 순서 재조립 후 순서대로 전달

socket.close();
// → 4-way handshake + TIME_WAIT
```

`RestTemplate`, `WebClient`는 이 과정을 전부 내부적으로 처리한다.
커넥션 풀(`PoolingHttpClientConnectionManager`)은 `socket.close()` 없이 소켓을 재사용해
3-way handshake와 TIME_WAIT 비용을 절감한다.

## 아직 열린 질문 — 다음 탐구의 시작점

> **HTTP 요청을 보냈는데 응답이 오지 않는 경우, 이것은 TCP 실패인가 HTTP 레벨의 문제인가?**
>
> TCP는 세그먼트가 도착하면 ACK를 보낸다. 즉, 서버가 HTTP 요청 세그먼트를 받았다는 TCP ACK와
> 서버가 HTTP 요청을 처리해 HTTP 응답을 보냈다는 것은 **전혀 다른 사건**이다.
>
> TCP ACK가 왔는데 HTTP 응답이 없는 상황은 어떻게 해석해야 하는가?

*힌트: TCP 타임아웃과 HTTP 타임아웃은 독립적으로 설정된다.*
