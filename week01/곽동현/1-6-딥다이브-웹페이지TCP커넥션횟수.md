# 1.6 딥다이브 — 웹페이지 하나를 불러올 때 TCP 커넥션은 몇 번 맺을까?

> TCP 커넥션 수립에는 비용이 든다.
> 그런데 웹페이지 하나를 불러오려면 HTML 뼈대 하나만이 아니라, 수십 개의 리소스를 추가로 요청해야 한다.
> HTTP는 이 문제를 어떻게 해결해왔는가? — HTTP/1.0 → 1.1 → 2의 진화를 통해 확인한다.

*연결 섹션: 1.4.3 웹페이지는 여러 객체로 이루어질 수 있다 + 1.6 TCP 커넥션*

---

## 1. 출발점 — 1.4.3의 단서

1.4.3은 이미 핵심 단서를 담고 있다.

> "웹페이지는 보통 하나의 리소스가 아닌 **리소스의 모음**이다."

브라우저가 HTML 뼈대를 받으면, 그 안에 숨겨진 리소스들이 드러난다.

```html
<html>
  <link rel="stylesheet" href="/style.css">    <!-- 요청 2 -->
  <script src="/bundle.js"></script>            <!-- 요청 3 -->
  <img src="/logo.png">                         <!-- 요청 4 -->
  <img src="/hero.jpg">                         <!-- 요청 5 -->
  <img src="/product1.jpg">                     <!-- 요청 6 -->
  ...
</html>
```

HTML 1개를 받는 순간, 브라우저는 이 안의 리소스들을 추가로 요청해야 한다는 사실을 알게 된다.
**현대 웹페이지의 평균 리소스 수는 약 70~80개**이다 (HTTP Archive 기준).

---

## 2. HTTP/1.0의 동작 — 요청마다 새 TCP 커넥션

HTTP/1.0은 기본적으로 **요청 하나당 TCP 커넥션 하나**를 맺고, 응답 후 즉시 끊는다.

```
[3-way handshake]  →  GET /index.html HTTP/1.0  →  응답  →  [TCP 종료]
[3-way handshake]  →  GET /style.css  HTTP/1.0  →  응답  →  [TCP 종료]
[3-way handshake]  →  GET /logo.png   HTTP/1.0  →  응답  →  [TCP 종료]
[3-way handshake]  →  GET /bundle.js  HTTP/1.0  →  응답  →  [TCP 종료]
...  (리소스 개수만큼 반복)
```

### 실제 비용 계산

| 항목 | 값 |
|------|-----|
| 3-way handshake 비용 | 1.5 RTT |
| RTT 예시 (서울 → 미국 서버) | ~150ms |
| 리소스 80개 기준 연결 수립 비용 | 80 × 150ms = **12초** |

12초가 실제 데이터 전송이 아닌, 오직 **연결 수립에만** 소비된다.
리소스 80개라면 TCP 커넥션도 **80번** 맺는다.

---

## 3. HTTP/1.1의 해결 — 지속 커넥션 (Persistent Connection)

HTTP/1.1은 **지속 커넥션을 기본값**으로 도입했다.
하나의 TCP 커넥션을 여러 요청이 재사용한다.

```
[3-way handshake]  (딱 1번)
  →  GET /index.html HTTP/1.1  →  응답
  →  GET /style.css  HTTP/1.1  →  응답  ← 같은 커넥션 재사용
  →  GET /logo.png   HTTP/1.1  →  응답  ← 같은 커넥션 재사용
  →  GET /bundle.js  HTTP/1.1  →  응답  ← 같은 커넥션 재사용
  ...
[TCP 종료]  (딱 1번)
```

TCP 커넥션 수립 비용이 여러 요청에 분산된다.

### 한계 — 순차 처리

하지만 이 구조에는 근본적인 한계가 있다.
**하나의 커넥션에서 요청은 순서대로만 처리된다.**

```
GET /index.html  →  응답 대기 중...
                     (이 응답이 오기 전까지 다음 요청을 보낼 수 없다)
GET /style.css   →  응답 대기 중...
                     (이 응답이 오기 전까지 다음 요청을 보낼 수 없다)
GET /logo.png    →  응답 대기 중...
```

이것이 **HOL Blocking (Head-of-Line Blocking)** 이다.
앞 요청이 막히면 뒤 요청 전체가 기다린다.

---

## 4. 브라우저의 현실적 해결 — 병렬 커넥션 6개

HOL Blocking을 우회하기 위해 브라우저는 **도메인당 최대 6개의 병렬 커넥션**을 열었다.

```
커넥션 1:  GET /style.css   →  응답
커넥션 2:  GET /bundle.js   →  응답
커넥션 3:  GET /logo.png    →  응답
커넥션 4:  GET /hero.jpg    →  응답
커넥션 5:  GET /product.jpg →  응답
커넥션 6:  GET /api/user    →  응답
── 위 6개가 완료되면 다음 리소스들 처리 ──
```

6개 초과 요청은 큐에서 대기한다.

이 제한을 우회하기 위해 **도메인 샤딩(Domain Sharding)** 이라는 기법이 등장했다.
같은 서버를 `img1.cdn.com`, `img2.cdn.com` 처럼 서브도메인으로 분산시켜, 도메인당 커넥션 제한을 피하는 방식이다.
HTTP/2 이후 이 기법은 오히려 성능을 떨어뜨리는 **안티패턴**이 되었다.

---

## 5. HTTP/2의 해결 — 멀티플렉싱 (Multiplexing)

HTTP/2는 구조 자체를 바꿨다.
**1개의 TCP 커넥션** 위에서 여러 **스트림(Stream)** 을 동시에 처리한다.

```
TCP 커넥션 1개
  ├─ 스트림 1:  GET /index.html  ──────────────▶  응답
  ├─ 스트림 2:  GET /style.css   ──────────────▶  응답  (동시)
  ├─ 스트림 3:  GET /logo.png    ──────────────▶  응답  (동시)
  └─ 스트림 4:  GET /bundle.js   ──────────────▶  응답  (동시)
```

도메인 샤딩도, 병렬 커넥션도 필요 없어졌다.

---

## 6. HTTP 버전별 커넥션 횟수 비교

| 버전 | 리소스 80개 기준 커넥션 수 | 방식 |
|------|--------------------------|------|
| HTTP/1.0 | **~80개** | 요청마다 새 커넥션 |
| HTTP/1.1 | **~6개** | 병렬 지속 커넥션 |
| HTTP/2 | **1개** | 멀티플렉싱 |

## 6.5 Chrome Network 탭으로 직접 확인하기

위의 이론이 실제로 어떻게 보이는지, Chrome 개발자 도구의 Network 탭이 그대로 보여준다.

---

### Waterfall 차트 — 각 색깔이 말하는 것

Network 탭의 오른쪽 Waterfall 열에서 각 요청 막대는 여러 색으로 나뉜다.

```
요청 하나의 Waterfall 분해:

│ Queueing    │ Stalled │ DNS │ Initial Connection │ SSL │ TTFB │ Download │
│ (연회색)     │ (연회색) │(청색│     (주황색)       │(보라)│(초록) │  (파랑)  │
```

| 색상 | 의미 | 이 글과의 연결 |
|------|------|--------------|
| Queueing (연회색) | 브라우저가 요청을 큐에 쌓고 기다리는 시간 | **도메인당 6개 커넥션 한계**에 걸려 대기 중일 때 길어진다 |
| Initial Connection (주황색) | TCP 3-way handshake | HTTP/1.0에서 리소스마다 이 주황색 구간이 반복된다 |
| SSL (보라색) | TLS handshake | TCP handshake 위에 추가 비용 |
| TTFB (초록색) | 서버가 첫 바이트를 보내기까지 대기 | 커넥션이 아닌 서버 처리 시간 |
| Download (파란색) | 실제 응답 데이터 수신 | 여기가 진짜 데이터 전송 구간 |

> **"Initial Connection이 주황색으로 길게 보인다"** = 지금 이 요청은 새 TCP 커넥션을 맺고 있다는 뜻이다.
> HTTP/1.0처럼 요청마다 새 커넥션을 맺으면, 모든 요청의 Waterfall 막대에 주황색 구간이 반복된다.

---

### Waterfall 패턴 — 버전별로 모양이 다르다

Waterfall을 멀리서 보면 HTTP 버전별 특징이 패턴으로 드러난다.

**HTTP/1.1 (6개 병렬 커넥션) — 계단 모양(Staircase)**

```
요청 1  ████████████████░░░░░░░░░░░
요청 2  ████████████░░░░░░░░░░░░░░░
요청 3  ██████████████████░░░░░░░░░
요청 4  ████████████████████░░░░░░░
요청 5  ██████████░░░░░░░░░░░░░░░░░
요청 6  █████████████░░░░░░░░░░░░░░
요청 7             ████████████████   ← 앞의 6개가 끝날 때까지 대기 후 시작
요청 8             ██████████████░░
...
```

7번 요청부터 Queueing 구간(연회색)이 길게 앞에 붙는다.
이것이 **"6개 커넥션 한계에 걸려 대기 중"** 인 상태가 Waterfall에서 보이는 방식이다.

**HTTP/2 (멀티플렉싱) — 수평선 패턴**

```
요청 1  ████████████████████
요청 2  ███████████████░░░░░
요청 3  █████████████████░░░
요청 4  ██████████████░░░░░░
요청 5  █████████████████░░░
요청 6  ████████████████████
요청 7  ██████████████████░░   ← 앞 요청 완료 여부와 무관하게 거의 동시 시작
요청 8  ███████████████░░░░░
...
```

모든 요청이 거의 같은 시작점에서 출발하고, Queueing 구간이 사라진다.

---

### Connection ID — 같은 커넥션인지 눈으로 확인하기

Chrome Network 탭에서 컬럼 헤더를 **우클릭** → **Connection ID** 체크 시, 각 요청이 어떤 TCP 커넥션을 사용했는지 숫자로 표시된다.

| 상황 | Connection ID 패턴 |
|------|-------------------|
| HTTP/1.0 (요청마다 새 커넥션) | 요청마다 모두 다른 숫자 |
| HTTP/1.1 (6개 병렬 지속 커넥션) | 6~7가지 숫자가 돌아가며 재사용됨 |
| HTTP/2 (멀티플렉싱) | 거의 모든 요청이 **동일한 숫자** |

> HTTP/2 사이트에서 Connection ID 컬럼을 켜면, 수십 개의 요청이 전부 같은 숫자를 공유하고 있음을 확인할 수 있다.
> 그것이 **"1개의 TCP 커넥션 위에서 멀티플렉싱"** 이 동작하고 있다는 증거다.

---

### Protocol 컬럼 — 버전 직접 확인

컬럼 헤더 우클릭 → **Protocol** 체크 시, 각 요청의 HTTP 버전이 표시된다.

| Protocol 값 | 의미 |
|-------------|------|
| `http/1.1` | HTTP/1.1 지속 커넥션 |
| `h2` | HTTP/2 멀티플렉싱 |
| `h3` | HTTP/3 (QUIC 기반) |

사이트에 따라 같은 페이지 안에서도 `h2`와 `http/1.1` 이 섞여 있을 수 있다.
이는 **도메인마다 서버 지원 버전이 다르기 때문**이다.

---

### 도메인 샤딩의 흔적 찾기

Network 탭에서 요청 목록을 보면 `img1.cdn.com`, `img2.cdn.com` 처럼 서로 다른 도메인에서 오는 리소스가 보일 때가 있다.
Protocol이 `http/1.1` 인데도 병렬 요청이 12개 이상 동시에 진행되고 있다면, **도메인 샤딩** 이 적용된 것이다.
HTTP/2 이후의 사이트에서 이 패턴이 보인다면, 최적화가 아닌 **오히려 커넥션 낭비**가 일어나고 있다는 신호다.

## 7. Java 개발자 관점

Spring의 `RestTemplate`은 기본적으로 요청마다 새 TCP 커넥션을 맺는다 — HTTP/1.0 방식과 같다.
`HttpComponentsClientHttpRequestFactory`에 `PoolingHttpClientConnectionManager`를 연결하면 커넥션 풀을 통해 지속 커넥션을 재사용할 수 있다.

`WebClient` (Spring WebFlux)는 기본적으로 HTTP/2를 지원하며, Reactor Netty 기반으로 멀티플렉싱 커넥션을 사용한다.
