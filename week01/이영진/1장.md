## 1. HTTP: 인터넷의 멀티미디어 운반책

- 매일 수십억 개의 JPEG, HTML, MPEG, WAV, Java applet 등이 인터넷 위를 떠돈다. 이 트래픽 대부분을 HTTP가 옮긴다.
- HTTP는 **신뢰성 있는 전송 프로토콜(TCP)** 위에서 동작하기 때문에, 전송 중 데이터가 손상되거나 뒤섞이지 않음을 보장한다.
- 덕분에 애플리케이션 개발자는 **네트워크 신뢰성 문제를 직접 처리할 필요 없이** 비즈니스 로직에 집중할 수 있다.

---

## 2. 웹 클라이언트와 서버
- **웹 서버 (HTTP server)**: 콘텐츠를 보유하고, HTTP 요청에 응답한다.
- **웹 클라이언트 (HTTP client)**: 서버에 요청을 보내고 응답을 받는다. 가장 흔한 클라이언트는 브라우저.
- 통신 모델: **요청(Request) → 응답(Response)** 의 단순한 왕복 구조.
---

## 3. 리소스 (Resources)
- 웹 리소스 = 웹 콘텐츠의 출처. 정적 파일일 수도, 동적으로 생성되는 콘텐츠일 수도 있다.
- 정적 리소스: HTML, JPEG, PDF, 텍스트 파일 등
- 동적 리소스: CGI 프로그램, API 엔드포인트, 검색 엔진, 전자상거래 게이트웨이 등 — 요청 시점에 콘텐츠 생성
---

## 4. 미디어 타입 (MIME Types)
- 인터넷은 수천 가지 데이터 타입을 다루므로, HTTP는 모든 객체에 **데이터 포맷 라벨(MIME type)** 을 붙인다.
- MIME = **M**ultipurpose **I**nternet **M**ail **E**xtensions. 원래는 메일용이었으나 HTTP가 채택.
- 형식: `주타입/서브타입`

| MIME Type                       | 설명             |
|---------------------------------|----------------|
| `text/html`                     | HTML 문서        |
| `text/plain`                    | 일반 텍스트         |
| `image/jpeg`                    | JPEG 이미지       |
| `image/gif`                     | GIF 이미지        |
| `video/quicktime`               | QuickTime 영상   |
| `application/vnd.ms-powerpoint` | PowerPoint 프레젠테이션 |

## 5. URI, URL, URN

### 5.1 URI (Uniform Resource Identifier)
- 리소스의 **고유 식별자**. URI는 URL과 URN의 **상위 개념**.

### 5.2 URL (Uniform Resource Locator)
- 가장 흔한 형태. 리소스의 **구체적 위치**를 알려준다.
- 3부분 구조:

```
http :// www.joes-hardware.com / specials/saw-blade.gif
└┬┘   └─────────┬───────────┘ └────────┬───────────┘
scheme        server                 path
(프로토콜)    (서버 주소)            (서버 내부의 리소스 경로)
```

### 5.3 URN (Uniform Resource Name)
- 위치와 무관하게 **이름만으로 리소스를 식별**. 예: `urn:ietf:rfc:2141`
- 서버가 이사 가도 식별자가 유지됨이 장점이지만, **이를 해석할 인프라가 부족해 거의 쓰이지 않는다.**
---

## 6. 트랜잭션 (Transactions)
HTTP 트랜잭션 = **요청 명령 + 응답 결과** 1쌍.

### 6.1 메서드 (Methods)
| Method | 설명                                  |
|--------|-------------------------------------|
| GET    | 서버의 명명된 리소스를 클라이언트로 가져옴             |
| PUT    | 클라이언트의 데이터를 서버의 명명된 리소스에 저장         |
| DELETE | 서버의 명명된 리소스를 삭제                     |
| POST   | 클라이언트 데이터를 서버의 게이트웨이 애플리케이션으로 전송    |
| HEAD   | 응답 본문 없이 헤더만 받기 (메타 정보 확인용)          |

### 6.2 상태 코드 (Status Codes)
3자리 숫자 + 사람이 읽는 reason phrase로 구성된다. 처리 로직은 **숫자만** 본다.

| Code | 의미                            |
|------|-------------------------------|
| 200  | OK. 요청 성공                     |
| 302  | Redirect. 다른 곳으로 가서 리소스를 가져와라 |
| 404  | Not Found. 리소스를 찾을 수 없다       |

### 6.3 웹 페이지는 보통 여러 트랜잭션이다
- HTML "골격"을 한 번 받고, 안의 이미지·CSS·스크립트마다 또 다른 트랜잭션이 발생한다.
- 한 페이지가 곧 한 리소스가 아니라 **여러 리소스의 묶음**이다.

---

## 7. 메시지 (Messages)
HTTP 메시지는 **사람이 읽을 수 있는 텍스트** 시퀀스다.

### 7.1 메시지 구조
```
[Start line]              ← 무엇을 할지(요청) / 무슨 일이 일어났는지(응답)
[Header fields]           ← 0개 이상. name: value 형태
[Blank line]              ← 헤더 끝 표시
[Body]                    ← 선택적. 텍스트 또는 임의의 바이너리
```

### 7.2 요청 메시지 예시
```http
GET /tools.html HTTP/1.0
User-agent: Mozilla/4.75 [en] (Win98; U)
Host: www.joes-hardware.com
Accept: text/html, image/gif, image/jpeg
Accept-language: en
```

### 7.3 응답 메시지 예시
```http
HTTP/1.0 200 OK
Date: Sun, 01 Oct 2000 23:25:17 GMT
Server: Apache/1.3.11
Content-length: 403
Content-type: text/html

<HTML>...</HTML>
```

## 8. 연결 (Connections)

### 8.1 TCP/IP 위에 얹힌 HTTP
HTTP는 application layer 프로토콜이며, **전송 책임은 TCP/IP에 위임**한다.

```
HTTP                        ← Application
TCP                         ← Transport
IP                          ← Network
Network-specific link
Physical hardware
```

TCP가 보장하는 것:
- **에러 없는** 데이터 전송
- **순서 보장** 전달
- **세그먼트 없는 스트림** (임의 크기, 임의 시점에 흘려보낼 수 있음)

### 8.2 연결을 맺기 위해 필요한 것
- **IP 주소** (어떤 머신인지)
- **포트 번호** (그 머신의 어느 프로그램인지)
- HTTP의 기본 포트는 **80**.

### 8.3 브라우저가 페이지를 표시하는 절차
1. URL에서 호스트명 추출
2. DNS로 호스트명을 IP 주소로 변환
3. URL에서 포트 번호 추출 (없으면 80)
4. 서버와 TCP 연결 수립
5. HTTP 요청 메시지 전송
6. HTTP 응답 메시지 수신
7. 연결 종료, 문서 표시
---

## 9. 프로토콜 버전

| Version    | 특징                                           |
|------------|----------------------------------------------|
| HTTP/0.9   | 1991. GET만 지원, 헤더·MIME·버전 없음. 거의 사용 안 함     |
| HTTP/1.0   | 첫 광범위 배포. 버전 번호, 헤더, 다양한 메서드, 멀티미디어 처리 추가      |
| HTTP/1.0+  | keep-alive 연결, virtual hosting, 프록시 등 비공식 확장 |
| HTTP/1.1   | 아키텍처 결함 수정, 의미 명확화, 성능 최적화. **현재의 표준**       |
| HTTP-NG    | HTTP/2.0의 초기 제안. 결국 다른 형태로 후속 표준화             |

## 10. 웹의 아키텍처 컴포넌트

브라우저 ↔ 서버 외에도, 실세계 웹 트래픽은 여러 중간자를 거친다.

### 10.1 프록시 (Proxy)
- 클라이언트와 서버 사이에 위치, 요청을 받아 서버로 전달.
- 보안(필터링), 로깅, 인증, 콘텐츠 검사 용도.

### 10.2 캐시 (Cache)
- 자주 요청되는 문서의 사본을 보관하는 특수 프록시.
- 가까운 곳에서 받기 때문에 **응답이 빠르고 원 서버 부하가 줄어든다**.

### 10.3 게이트웨이 (Gateway)
- 다른 프로토콜로 변환해주는 중간 서버.
- 예: HTTP/FTP 게이트웨이는 클라이언트의 HTTP 요청을 받아 실제로는 FTP로 가져와 HTTP 응답으로 포장.
- 클라이언트는 자신이 게이트웨이와 대화하는 줄 모를 수도 있다.

### 10.4 터널 (Tunnel)
- 셋업된 후에는 원본 데이터를 **그대로 양방향 중계**.
- 대표 예: HTTP CONNECT로 만든 SSL 터널 — HTTP만 허용된 방화벽을 넘어 암호화된 SSL 트래픽을 흘려보낼 때 사용.

### 10.5 에이전트 (Agent)
- 사람을 대신해 HTTP 요청을 보내는 클라이언트 프로그램.
- 브라우저뿐 아니라 검색 엔진의 spider, 가격 비교 봇 등 모든 자동화 클라이언트가 포함된다.

---

## 정리
1. HTTP는 **클라이언트-서버 모델** 위의 요청-응답 프로토콜이며 **TCP** 위에서 동작한다.
2. 리소스는 **URI(주로 URL)** 로 식별되고, **MIME 타입**으로 콘텐츠 종류가 표시된다.
3. 트랜잭션은 **메서드 + 상태 코드**로 의미가 결정된다.
4. 메시지는 **start line + headers + (blank line) + body** 의 텍스트 구조다.
5. 실제 웹 트래픽은 **프록시·캐시·게이트웨이·터널·에이전트**가 얽힌 그물망 위를 흐른다.
