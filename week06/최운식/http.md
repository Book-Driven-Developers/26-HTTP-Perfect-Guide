### 웹 프락시 서버
- 중개자이다.
- 클라이언트와 서버 사이에서 HTTP 메시지를 정리하는 중개인처럼 동작함.

### 웹 중개자

프락시 서버는 클라이언트 입장에서 트랜잭션을 수행하는 중개인이다.
- 프락시 서버가 없다면 클라이언트는 HTTP 서버와 직접 통신한다.
- 프락시 서버와 통신하므로서 프락시 서버가 제공하는 좋은 서비스를 이용할 수 있다.
- 서버 입장에서는 클라이언트처럼 동작해야한다.

클라이언트 <-> 프락시 <-> 서버

**개인 프락시와 공유 프락시**

개인 프락시: 하나의 클라이언트가 독점적으로 사용
공유 프락시: 복수의 클라이언트가 공유하여 사용

대부분의 프락시는 공용이며 공유된 프락시이다.
사용자가 많을수록 유리한데, 여러 사용자들의 공통된 요청으로부터 이득을 취할 수 있기 때문이다. (캐시)

**프락시 vs. 게이트웨이**

프락시ㅌ
- 같은 프로토콜을 사용하는 둘 이상의 애플리케이션을 연결한다.

예시)
> Spring Cloud Gateway (단순 리버스 프락시 활용)
> 
> 마이크로서비스 아키텍처(MSA)에서 외부 클라이언트의 HTTP 요청을 내부의 다른 스프링 부트 서비스(HTTP)로 그대로 넘겨주는 역할을 할 때.
> 
> 클라이언트의 `HTTP 요청` -> Spring Cloud Gateway -> 내부 서비스의 `HTTP 요청` (같은 프로토콜)

**프락시**는 요청을 감싸서 통제(제어)하는 것이 목적이다. 
- 겉보기에는 원래 요청과 완전히 똑같은 프로토콜/인터페이스로 움직이기 때문에 클라이언트는 자신이 프락시와 말하고 있는지 실제 서버와 말하고 있는지 모른다.

---

게이트웨이
- 서로다른 프로토콜을 사용하는 둘 이상의 애플리케이션을 연결한다.
- 서로 다른 프로토콜로 말해도 서로간의 트랜잭션을 완료할 수 있도록 해주는 프로토콜 변환기로서 동작함.

예시)
> Spring Integration의 Gateway
> 
> 스프링 생태계에서 프로토콜 변환을 가장 명확하게 보여주는 모듈. 다른 시스템이나 프로토콜과 메시지를 주고받을 때 사용된다.
> 
> **예시 시나리오**: 우리 시스템은 현대적인 **HTTP REST API**를 사용하지만, 연동해야 하는 사내 레거시 시스템은 내부 메시지 큐인 RabbitMQ(AMQP 프로토콜)나 gRPC를 사용하고 있는 경우.
> 
> **동작 방식**:
    1. 외부 클라이언트가 스프링 부트 서버로 `HTTP POST /order` 요청을 보냅니다.
    2. **Spring Integration Gateway**가 이 HTTP 요청을 받아 내부에서 사용하는 메시지 포맷(`Message<T>`)으로 파싱합니다.
    3. 게이트웨이 내부의 어댑터가 이를 **AMQP 프로토콜**로 변환하여 RabbitMQ 브로커로 발행(Publish)합니다.
>
 >   **프로토콜**
 >   - `HTTP/JSON` -> [게이트웨이의 프로토콜 변환] -> `AMQP(RabbitMQ)` 또는 `SOAP(XML)`
 
게이트웨이는 서로 다른 두 세계를 "통역(변환)"하는 것이 목적. 
클라이언트는 게이트웨이가 제공하는 프로토콜(예: HTTP)로 편하게 말하고, 게이트웨이는 뒷단 시스템이 알아들을 수 있는 프로토콜(예: gRPC, DB 프로토콜, 레거시 소켓 통신)로 통역하여 트랜잭션을 성사시킨다.

### 왜 프락시를 사용할까?

실용적이고 유용한것이라면 무엇이든함.
- 보안을 개선
- 성능을 높임
- 비용을 절약

모든 HTTP 트래픽을 들여다보고 건드릴 수 있으므로 부가적인 가치를 주는 여러 웹서비스를 구현하기 위해 트래픽을 감시하고 수정할 수 있다.
1. 어린이 필터 (학교의 필터 프락시)
	1. 교육 사이트를 제공하면서 성인 콘텐츠를 차단
2. 문서 접근 제어자 (접근 제어 프락시)
	1. 웹 서버 및 웹 리소스에 대한 단일한 접근 제어 전략
	2. 감사 추적 (audit trail)
	3. 특정 자료에 대한 접근 요청 시 비밀번호 요청 등
3. 방화벽 프락시
	1. 트래픽을 세심하게 살펴볼 수 있는 Hook (훅)을 제공
	2. 바이러스 등을 제거
4. 웹 캐시 (프록시 캐시)
	1. 인기있는 문서의 로컬 사본을 관리
	2. 해당 문서에 대한 요청이 오면 빠르게 제공
5. 리버스 프록시 (대리 프록시)
	1. 웹 서버인것처럼 위장하는 프록시 서버
	2. 요청 받은 콘텐츠의 위치를 찾아내기 위해 다른 서버와 커뮤니케이션을 시작함.
	3. 공용 콘텐츠에 대한 느린 웹 서버의 성능을 개선하기 위해 사용될 수 있음
		1. 이런 대리 프록시를 서버 가속기라고 부름.
		2. 콘텐츠 라우팅 기능과 결합되어 주문형 복제 콘텐츠의 분산 네트워크를 만들기 위해 사용될 수 있음.
6. 익명화 프록시
	1. HTTP 메시지에서 신원을 식별할 수 있는 특성들 (클라이언트 IP 주소, From 헤더, Referer 헤더, 쿠키, URI 세션 아이디, User-Agent 내 컴퓨터/OS 명)을 제거한다.
	2. 개인정보 보호 및 익명성 보장에 기여
	3. 크롤링 시 주로 사용된다.
		1. 수천 개의 익명화 프록시 IP를 번갈아 가며 사용하여 일반 사용자인 것처럼 위장해 데이터를 수집.
	

> *누가 돈들여가면서 웹 캐시 서버같은 프록시 캐시 서버를 만드는거지?*
7. 콘텐츠 전송 네트워크 (CDA/CDN 기업)
	- 전 세계 프락시 캐시 서버를 가장 많이 만들고 운영하는 주체는 **CDN(Content Delivery Network)** 기업들입니다. 대표적으로 Cloudflare나 Akamai 같은 기업이 있습니다.
 - **누가 만드는가?**: 전문 CDN 사업자 및 글로벌 테크 기업.
- **돈을 왜 쓰는가?**: 콘텐츠 제공업체(넷플릭스, 유튜브, 대형 쇼핑몰 등)에게 돈을 받기 때문입니다.
- **비용 절감 원리**:
    - 넷플릭스 본사 서버가 미국에 있다면, 한국 사용자가 영상을 볼 때마다 미국까지 트래픽이 이동해야 하므로 통신 비용이 폭증하고 느려집니다.
    - CDN 기업은 한국에 프락시 캐시 서버를 둡니다. 한국 사용자가 영상을 요청하면 미국 본사가 아닌 **한국 프락시 서버에서 바로 캐시된 데이터를 응답**합니다.
    - 콘텐츠 기업은 서버 부하와 국제 회선 비용을 줄일 수 있으므로, CDN 기업에 기꺼이 막대한 비용을 지불합니다. 
1. 거대 글로벌 인터넷 기업 (자체 CDN 구축)
	- 구글, 넷플릭스, 메타, 애플 같은 기업들은 CDN 기업에 돈을 주는 것을 넘어, **자신들이 직접 전 세계에 프락시 캐시 서버를 구축**합니다. 이를 **자체 CDN(Private CDN)** 또는 **에지(Edge) 인프라**라고 부릅니다.
- **누가 만드는가?**: 구글(Google Global Cache), 넷플릭스(Open Connect) 등.
- **돈을 왜 쓰는가?**: **자신들의 인터넷 통신 비용(대역폭 비용)을 직접 아끼기 위해서**입니다.
- **비용 절감 원리**:
    - 인기 있는 유튜브 영상이나 신작 드라마는 수천만 명이 동시에 봅니다.
    - 이 데이터를 매번 본사 데이터센터에서 쏴주면 통신망 대역폭(Bandwidth) 비용으로 회사가 파산할 수 있습니다.
    - 전 세계 통신사(ISP)의 데이터센터 안에 자신들의 프락시 캐시 서버를 무상으로 설치해 두고, 전 세계 트래픽의 90% 이상을 이 캐시 서버 안에서 해결하여 **본사의 트래픽 비용을 수조 원씩 절감**합니다.
 3. 통신사, ISP, VPN 업체 등

> 리버스 프록시 기능 중 공용 콘텐츠에 대한 느린 서버 응답을 개선하기 위해 사용된다는게 뭔소리일까?
> 특히 콘텐츠 라우팅은 뭐고, 주문형 복제 콘텐츠의 분산 네트워크를 만든다는게 뭘까?

쉡게 말하면 유튜브, 넷플릭스, 쿠팡 같은 대형 서비스가 전 세계 사용자에게 데이터를 빠르게 나눠주는 원리(CDN)를 말하는 것이다.

#### 1. 느린 웹 서버의 성능 개선 (서버 가속기)
"유명 맛집(웹 서버)"과 "유능한 매니저(리버스 프록시)"로 비유

- **상황**: 맛집 주방장(웹 서버)은 요리를 정말 잘하지만, 주문을 받고 음식을 서빙하는 속도가 느립니다. 게다가 손님이 몰리면 주방이 터져버립니다.
- **해결책**: 가게 앞에 **매니저(리버스 프록시/서버 가속기)**를 둡니다.
- **동작 방식**:
    1. 매니저는 주방장에게 미리 **가장 인기 있는 메뉴(공용 콘텐츠: 이미지, JS 파일 등)**를 대량으로 받아 가게 앞 가판대에 쌓아둡니다.
    2. 손님이 와서 "저 콜라랑 감자튀김 주세요!"라고 하면, 매니저는 **주방장에게 물어보지 않고 가판대에서 바로 꺼내서 줍니다.**
    3. 주방장은 메인 요리(실시간 데이터 처리)에만 집중할 수 있게 되어, 전체적인 가게 성능이 엄청나게 빨라집니다.

> **정리**: 실제 웹 서버가 직접 처리하면 시간이 오래 걸리는 일(정적 이미지 파일 읽기 등)을 리버스 프록시가 미리 복사(캐시)해 두고 대신 응답해 줌으로써, **웹 서버의 부담을 덜고 속도를 지극히 높여주기 때문에 "서버 가속기"라고 부릅니다.**

#### 2. 콘텐츠 라우팅 + 주문형 복제 콘텐츠의 분산 네트워크
- **콘텐츠 라우팅**: 사용자가 요청한 콘텐츠가 무엇인지 확인하고, 가장 가까운 프록시 서버로 길을 안내(Routing)해 주는 기능
- **주문형 복제 콘텐츠**: 사용자가 원할 때(주문형, On-Demand)만 원본에서 복사해 오는 데이터
- **분산 네트워크**: 전 세계 여기저기에 흩어져 있는 프록시 서버들의 모임

예시 상황
- 넷플릭스 본사(미국 웹 서버)와 한국/일본/유럽의 프록시 서버들(분산 네트워크)

흐름으로 보는 예시 (넷플릭스 신작 드라마 출시)
```
[미국 본사 서버] ── 신작 드라마 업로드! (아직 각국 프록시에는 파일이 없음)
      │
      ├─► [한국 리버스 프록시] (현재 텅 빔) ◄─ 한국 사용자 A: "신작 드라마 보여줘!" (첫 요청)
      ├─► [일본 리버스 프록시] (현재 텅 빔)
      └─► [유럽 리버스 프록시] (현재 텅 빔)
```

1. 첫 번째 한국 사용자의 요청: 
	1. 한국 사용자 A가 오늘 개봉한 신작 드라마를 보려고 재생 버튼을 누릅니다.
2. 콘텐츠 라우팅: 
	1. 넷플릭스 시스템은 이 사용자가 '한국인'인 것을 감지하고, 미국 본사가 아닌 [한국 리버스 프록시 서버]로 연결해 줍니다. (**콘텐츠 라우팅**)
3. 주문형 복제 (On-Demand Copy): 
	1. 그런데 한국 프록시 서버에는 아직 이 신작 드라마 영상 파일이 없습니다. 
	   한국 프록시는 급히 미국 본사 서버에 요청해서 영상을 받아온 뒤, 사용자 A에게 틀어줍니다. 
	   그리고 동시에 자기 저장소에 이 영상을 복사(**주문형 복제**)해 둡니다.
4. 두 번째 사용자부터 가속: 
	1. 잠시 후 한국 사용자 B, C, D가 동일한 드라마를 요청합니다. 
	   시스템은 똑같이 [한국 리버스 프록시]로 안내합니다. 
	   이번에는 프록시 서버에 이미 영상이 복사되어 있으므로, 태평양을 건너지 않고 한국 안에서 엄청나게 빠른 속도로 영상을 바로 쏴줍니다. (**분산 네트워크 일부**)

```
[미국 본사 서버] 
      │
      ├─► [한국 리버스 프록시] (영상 복사 완료!) ◄─ 한국 사용자 B, C, D: "나도 보여줘!" 
      │           │                               (미국 안 가고 한국 프록시가 바로 응답!)
      │           └─► [초고속 스트리밍 서비스 완수]
      │
      ├─► [일본 리버스 프록시] 
      └─► [유럽 리버스 프록시]
```

이 시점에 일본이나 유럽 프록시 서버에는 아직 사용자가 요청하지 않았다면 영상이 복사되지 않은 상태입니다. 즉, 사용자가 주문(요청)하는 지역의 프록시 서버에만 콘텐츠가 실시간으로 복제되면서 전 세계에 **분산망을 형성**하게 됩니다.

### 프락시는 어디에 있는가?

**프락시 서버 배치**

- 출구(Egress) 프락시
	- 로컬 네트워크와 더 큰 인터넷 사이를 오가는 트래픽을 제어
	- 로컬 네트워크의 출구에 박아 넣음
	- 학교에서 부적절한 콘텐츠를 브라우징 하는것을 막을 때도 사용
- 접근(입구) 프락시
	- 고객으로부터의 모든요청을 종합 처리하기 위해 ISP 접근 지점에 위치
	- ISP는 사용자들의 다운로드 속도를 개선하고 인터넷 대역폭 비용을 줄이기 위해 사용
		- ex) 캐시 프락시를 사용해서 많이 찾는 문서들의 사본을 저장함
- 대리 프락시 (리버스 프락시)
	- 네트워크의 가장 끝에 있는 웹 서버들의 바로 앞에 위치.
	- 웝 서버로 향하는 모든 요청을 처리하고 필요할 때만  웹 서버에 자원을 요청함.
	- 웹 서버에 보안 기능을 추가하거나 웝 서버 캐시를 웹 서버 앞에 둬서 성능을 개선함.
	- 웹 서버의 이름과 IP로 스스로를 웹서버로 가장하기 때문에 모든 요청은 웹 서버가 아닌 프록시로 온다.
- 네트워크 교환 프락시
	- 캐시를 이용해서 인터넷 교차로의 혼잡을 완화하고 트래픽 흐름을 감시하는 역할
	- 충분한 처리 능력을 갖춘 프락시가 네트워크 사이의 인터넷 피어링 교환 지점에 놓인다.
	- 라우터와 라우터 사이에 위치함.

**프락시 계층**

프락시 계층은 프락시들이 연쇄적으로 구성된 것을 말한다.
메시지는 원 서버에 도착할 때까지 프락시 to 프락시를 거쳐서 이동한다. (돌아올때도 마찬가지)

프락시 계층에서 프락시들은 서로간 부모 자식 관계를 맺는다.
서버에 가까운 쪽이 인바운드 프락시, 먼쪽이 아웃바운드 프락시이며 인바운드 프록시가 아웃바운드 프록시의 부모가 된다.
- 클라이언트 -> 손자 프록시 -> 아들 프록시 -> 부모 프록시 -> 할아버지 프록시 -> 원 서버

**프록시 계층 콘텐츠 라우팅**

더 빠른 콘텐츠 제공을 위해 동적으로 적절히 라우팅을 할 수 있다.
- 요청에 대해 적절히 가깝고 빠른 캐시 프록시 서버에 보내거나, 원 서버에 보내거나 할 수 있음.

동적 라우팅
- 부하 균형
- 지리적 인접성에 근거한 라우팅
- 프로토콜/타입 라우팅
- 유료 서비스 가입자를 위한 라우팅

**어떻게 프락시가 트래픽을 처리?**
- 크롬 등 브라우저 같은 웹 클라이언트들은 수동/자동 프락시 설정을 지원한다.
	- if (프락시 사용 ON) -> 클라이언트는 HTTP 요청을 프락시로 보냄
- 네트워크를 수정한다
	- 클라이언트는 알지도 못하고 간섭할수도 없는 상태
	- 네트워크 인프라를 가로채서 웹 트래픽을 프락시로 가도록 조정
	- 스위칭 장치와 라우팅 장치가 필요함. 이를 인터셉트 프락시라고 부름.
- DNS 이름공간을 수정
	- 웹 서버 앞에 위치하는 프락시 서버를 둠
		- 웹 서버 이름과 IP 주소를 자신이 직접 사용
		- 따라서 모든 요청이 원 서버가 아닌 프록시 서버로 감
- 웹 서버를 수정
	- HTTP 리다이렉션 명령 (응답코드 305)를 클라이언트에게 돌려줘서 클라 요청을 프락시로 리다이렉트하게 설정.

### 클라이언트 프락시 설정

브라우저가 프락시를 사용할수 있도록 설정할 수 있다.

수동 설정
- 프락시를 사용하겠다고 명시적 설정

브라우저 기본 설정
- 브라우저 벤더나 배포자가 브라우저를 소비자에게 전달하기 전에 프록시를 미리 설정

프락시 자동 설정 (PAC)
- 자바스크립트 프락시 자동 설정 파일에 대한 URI를 제공
- 프락시를 써야하는지, 어떤 프락시 서버를 써야하는지 판단을 위해 파일을 가져와서 실행
	- `FindProxyForUrl(url, host)`라는 함수를 정의해야한다.

WPAD
- 자동 설정 파일을 다운받을 수 있는 설정 서버를 자동으로 찾아주는 프로토콜을 제공
- Web Proxy AutoDiscovery Protocol (WPAD)
	- PAC URI를 찾기 위해 WPAD를 사용
	- 주어진 URI에서 PAC 파일을 가져옴
	- 프라깃 서버를 알아내기 위해 PAC 파일을 실행
	- 알아낸 프락시 서버를 이요해서 요청을 처리

### 프락시 요청의 미묘한 특징

**프락시 URI는 서버 URI 와 다르다**
- 웹 서버와 웹 프락시 메시지의 문법은 서로 같지만 하나 예외가 있다.
	- 클라이언트가 프락시 대신 서버로 요청을 보내면 요청의 URI가 달라진다.
	- 서버로 보내면
		- 스키마, 호스트, 포트번호가 없는 생략된 부분 URI를 가진다.
		- GET /idnex.html HTTP/1.0
	- 프록시로 보내면
		- 완전한 풀 URI를 가진다.
		- GET http://www.naver.com/index.html HTTP/1.0
- 왜 이렇게 했을까?
	- 원래 HTTP 설계에서 클라이언트는 단일 서버와 직접 대화함.
	- 단일 서버는 자신의 호스트 명과 포트 번호를 알고 있으므로 클라이언트는 불필요한 정보 발송을 피하기 위해 생략했다.
	- 근데 프록시는 원 목적지 서버와 커넥션을 맺어야 하는데 해당 정보가 없으면 맺을수가 없다.
	- 따라서 풀 스키마를 보내줘야 한다.

**가상 호스팅에서 일어나는 문제**
- 가상 호스팅 웹서버는 호스트와 포트에 대한 정보가 담겨있는 Host 헤더를 요구

**리버스 프락시 서버는 부분 URI를 받음**
- 클라이언트가 통신하는 서버가 프록시인걸 항상 아는게 아니다.
- 따라서 프록시 서버의 경우 생략된 스키마를 받게 된다.
	- Host 헤더로 대부분 대응한다.
	- 만약 Host 헤더가 없다면 프락시에 실제 서버의 주소와 포트번호가 설정되어있을 수 있다.

**전송중 URI 변경 금지**
- 수정하지 말자

### 메시지 추적

현대 웹 요청이 클라이언트에서 서버로 향하는 도중에 둘 이상의 프록시를 지나게 되는건 매우 흔하다.
이러한 메시지의 흐름을 추적하고 문제를 찾아내는것도 필요하다.

#### via 헤더
- via 헤더 필드는 메시지가 지나는 각 중간 노드(프록시, 게이트웨이)의 정보를 나열한다.
- 목록의 끝에 추가한다.

`Via: 1.1 proxy-62.irenes-isp.net, 1.0 cache.joes-hardware.com`
- 2개의 프록시 서버를 지났다.

#### Via 문법
- 쉼표로 경유지를 구분한다.
- 프로토콜 이름 (기본 HTTP, 선택), 프로토콜 버전, 노드 이름, 코멘트(선택) 총 4개의 구성요소를 담을 수 있다.

요청과 응답 경로
- 응답의 Via 헤더는 거의 언제나 요청의 Via 헤더 반대이다.


#### TRACE 메서드

**Max-Forwards**
- TRACE, OPTIONS 요청의 프록시 홉 개수를 제한할 수 있다.

### 프록시 인증

프록시는 접근 제어 장치로서 역할을 할수 있다.

제한된 콘텐츠에 대한 요청이 프락시 서버에 오면 프락시 서버는 접근 자격을 요구하는 407 Proxy Authorization Required 상태 코드를 자격 제출 방법 설명을 담은 Proxy-Authenticate 헤더 필드와 함께 반환한다.

응답 시 Proxy-Authorization 헤더 필드에 담아서 요청을 다시 보내면 된다.

### 프락시 상호 운용성

프락시 서버는 헤더 필드들을 모두 이해하지 못할 수도 있다.
