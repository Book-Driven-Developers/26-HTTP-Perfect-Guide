## 1. 웹 중개자

웹 프록시 서버는 **클라이언트와 서버 사이에 위치하는 중개자**다.

- 클라이언트는 직접 서버와 대화하는 대신 프록시와 대화한다.
- 프록시는 **서버처럼** 동작해야 한다 (클라이언트의 요청/커넥션 수신, 응답 반환).
- 프록시는 **클라이언트처럼**도 동작해야 한다 (요청 발송, 응답 수신).

### 개인 프록시 vs 공유 프록시
| 구분 | 설명 |
|------|------|
| 개인 프록시 | 하나의 클라이언트만을 위한 프록시 |
| 공유 프록시 | 여러 클라이언트가 함께 사용하는 프록시 (일반적) |

### 프록시 vs 게이트웨이
| 구분 | 역할 |
|------|------|
| 프록시 | **같은 프로토콜**을 사용하는 둘 이상의 애플리케이션 연결 |
| 게이트웨이 | **서로 다른 프로토콜**을 사용하는 둘 이상을 연결 |

> 실제로는 경계가 모호하다. 프록시가 다른 HTTP 버전을 연결할 때 프로토콜 변환이 발생하기도 하고, 상용 프록시가 SSL, FTP 등을 지원해 게이트웨이 역할도 수행한다.

## 2. 왜 프록시를 사용하는가?

| 용도 | 설명 |
|------|------|
| **어린이 필터** | 성인 콘텐츠 필터링 |
| **문서 접근 제어자** | 여러 웹 서버에 대한 단일 접근 제어 정책 구현 및 감사 추적 |
| **보안 방화벽** | 조직 내/외부 트래픽의 응용 레벨 프로토콜 흐름을 한 지점에서 제어 |
| **웹 캐시** | 인기 있는 문서의 로컬 사본 관리 → 응답 속도 향상, 대역폭 절약 |
| **대리 프록시(Reverse Proxy)** | 웹 서버인 척 위장하여 요청 수신 → 느린 서버의 성능 개선 |
| **콘텐츠 라우터** | 트래픽 조건/콘텐츠 종류에 따라 특정 웹 서버로 요청 유도 |
| **트랜스코더** | 전달 전 콘텐츠 포맷 변환 (예: GIF → JPG, 이미지 압축) |
| **익명화 프록시** | 클라이언트 IP, From, Referer, 쿠키, URI 세션 ID 등 신원 정보 제거 |

## 3. 프록시는 어디에 있는가?

### 3-1. 프록시 서버 배치 유형

```
[클라이언트] → [출구 프록시] → 인터넷 → [입구 프록시] → [대리 프록시] → [웹 서버]
```

| 배치 위치 | 유형 | 주요 용도 |
|-----------|------|-----------|
| 로컬 네트워크 출구 | 출구(Egress) 프록시 | 방화벽, 필터링, 사용자 요청 제어 |
| ISP 접근 지점 | 접근(Ingress) 프록시 | 캐시로 대역폭 비용 절감, 속도 개선 |
| 원 서버 바로 앞 | 대리(Reverse/Surrogate) 프록시 | 보안, 캐시, 로드 밸런싱 |
| 인터넷 교환 지점 | 네트워크 교환 프록시 | 혼잡 완화, 트래픽 흐름 감시 |

### 3-2. 프록시 계층 (Proxy Hierarchy)

```
클라이언트 → [자식 프록시] → [부모 프록시] → 원 서버
```

- 서버에 가까울수록 **부모 프록시**, 클라이언트에 가까울수록 **자식 프록시**
- **동적 부모 선택**도 가능:
    - **부하 균형**: 부모들의 작업량에 따라 선택
    - **지리적 인접성** 기반 라우팅
    - **프로토콜/타입** 기반 라우팅
    - 유료 서비스 가입자 라우팅

### 3-3. 트래픽이 프록시로 향하는 4가지 방법

1. **클라이언트 수정**: 브라우저에서 수동/자동으로 프록시 설정
2. **네트워크 수정 (인터셉트 프록시)**: 스위치/라우터가 HTTP 트래픽을 가로채 프록시로 전달 (클라이언트는 모름)
3. **DNS 이름공간 수정**: 웹 서버 도메인의 IP를 프록시 IP로 지정
4. **웹 서버 수정**: HTTP 305 응답으로 클라이언트를 프록시로 리다이렉트

## 4. 클라이언트 프록시 설정

### 4-1. 수동 설정
브라우저 설정에서 직접 프록시 서버 주소와 포트 지정

### 4-2. PAC 파일 (Proxy Auto-Config)
- 상황에 맞게 프록시를 동적으로 계산하는 **자바스크립트 파일**
- 확장자: `.pac` / MIME TYPE: `application/x-ns-proxy-autoconfig`
- 핵심 함수: `FindProxyForURL(url, host)`

```javascript
function FindProxyForURL(url, host) {
    if (url.substring(0, 5) == 'http:')
        return "PROXY http-proxy.mydomain.com:8080";
    else if (url.substring(0, 4) == 'ftp:')
        return "PROXY ftp-proxy.mydomain.com:8080";
    else
        return "DIRECT";
}
```

| 반환값 | 의미 |
|--------|------|
| `DIRECT` | 프록시 없이 직접 연결 |
| `PROXY host:port` | 지정 프록시 사용 |
| `SOCKS host:port` | 지정 SOCKS 서버 사용 |

### 4-3. WPAD (Web Proxy Auto-Discovery)
- 여러 발견 메커니즘을 조합해 브라우저에게 **적합한 PAC 파일을 자동으로 찾아주는** 알고리즘
- 순서: PAC URI 탐색 → PAC 파일 가져오기 → PAC 실행 → 프록시 서버 결정 → 요청 처리

## 5. 프록시 요청의 미묘한 특징들

### 5-1. 프록시 URI vs 서버 URI

| 요청 대상 | URI 형식 | 예시 |
|-----------|----------|------|
| 직접 서버 | 부분 URI (스킴·호스트 생략) | `GET /index.html HTTP/1.1` |
| 프록시 | 완전한(절대) URI | `GET http://www.example.com/index.html HTTP/1.1` |

> HTTP/1.1은 모든 서버 요청에 완전한 URI를 요구하지만, 여전히 많은 서버가 부분 URI만 수용한다.

### 5-2. 인터셉트 프록시의 부분 URI 처리
클라이언트는 프록시 존재를 모르므로 부분 URI를 전송 → 인터셉트 프록시가 이를 처리해야 함

### 5-3. URI 처리 우선순위
1. **완전한 URI** 제공 → 그대로 사용
2. **부분 URI + Host 헤더** → Host 헤더로 원 서버 특정
3. **부분 URI, Host 헤더 없음** → 설정된 원 서버 주소 사용, 없으면 에러 반환

### 5-4. 전송 중 URI 변경 주의
- 사소한 URI 변경도 다운스트림 서버와 상호운용성 문제 유발 가능
- HTTP 명세: **일반 인터셉트 프록시의 절대 경로 수정 금지**
- 유일한 예외: 빈 경로를 `/`로 교체

## 6. 메시지 추적 (Via & TRACE)

### 6-1. Via 헤더

메시지가 지나는 **각 중간 노드(프록시/게이트웨이)의 정보**를 순서대로 나열한다.

```
Via: 1.1 proxy-62.irenes-isp.net, 1.0 cache.joes-hardware.com
```
→ 메시지가 두 개의 프록시를 순서대로 통과했음을 의미

#### Via 항목의 구성요소 (최대 4개)
| 구성요소 | 필수 여부 | 설명 |
|----------|-----------|------|
| 프로토콜 이름 | 선택 (기본: HTTP) | 수신 프로토콜 |
| 프로토콜 버전 | **필수** | 수신 메시지의 버전 (예: 1.1) |
| 노드 이름 | **필수** | 중개자의 호스트[:포트], 가명 사용 가능 |
| 노드 코멘트 | 선택 | 벤더/버전 정보, 진단 정보 등 |

#### Via 헤더의 용도
- 메시지 전달 경로 추적
- **라우팅 루프 탐지**: 요청 전송 전 자신의 고유 문자열 삽입 → 이미 존재하면 루프 감지
- 요청/응답 과정의 프로토콜 능력 진단
- 프록시 통과 시 `Server` 헤더 수정 불가 → **Via 항목 추가**로 대체

#### Via 항목 합치기 (개인정보 보호)
```
# Before (같은 조직 내 여러 프록시)
Via: 1.0 foo, 1.1 devirus.company.com, 1.1 access-logger.company.com

# After (동일 프로토콜 버전인 경우 합칠 수 있음)
Via: 1.0 foo, 1.1 concealed-stuff
```

### 6-2. TRACE 메서드

프록시를 지나는 동안 **메시지가 어떻게 변하는지 진단**하기 위한 HTTP/1.1 메서드

- 서버는 받은 전체 요청 메시지를 클라이언트에게 **그대로 되돌려** 준다.
- `Content-Type: message/http`로 응답

#### Max-Forwards 헤더
```
Max-Forwards: 2
```
- TRACE/OPTIONS 메시지가 전달될 **최대 홉(hop) 수 제한**
- `Max-Forwards: 0`이면 수신자는 즉시 클라이언트에게 반환 (더 이상 전달 안 함)
- 무한 루프나 오동작 프록시 체인 진단에 유용

## 7. 프록시 인증

HTTP는 **프록시 인증(Proxy Authentication)** 메커니즘을 통해 접근 제어를 구현한다.

### 흐름

```
클라이언트                        프록시
    |                               |
    |--- 요청 (자격증명 없음) ------→|
    |←-- 407 Proxy Auth Required ---|  (Proxy-Authenticate 헤더 포함)
    |                               |
    |--- 요청 (자격증명 포함) ------→|  (Proxy-Authorization 헤더 포함)
    |←-- 200 OK (또는 407 재요청) --|
```

| 상태 코드/헤더 | 방향 | 역할 |
|----------------|------|------|
| `407 Proxy Authorization Required` | 프록시 → 클라이언트 | 인증 요구 |
| `Proxy-Authenticate` | 프록시 → 클라이언트 | 자격 제출 방법 안내 |
| `Proxy-Authorization` | 클라이언트 → 프록시 | 자격증명 전달 |

> 프록시 인증은 체인 상에 **인증 참여 프록시가 하나일 때 잘 동작**한다.

## 8. 프록시 상호운용성

### 8-1. 이해하지 못하는 헤더/메서드 처리
- 알 수 없는 헤더와 메서드는 **그대로 전달**해야 한다.
- 같은 이름의 헤더가 여러 개인 경우 **상대적 순서 유지 필수**

### 8-2. OPTIONS 메서드
클라이언트가 서버나 특정 리소스가 지원하는 기능을 미리 확인할 수 있게 해준다.

```http
OPTIONS * HTTP/1.1
Host: www.example.com
```

### 8-3. Allow 헤더
```
Allow: GET, HEAD, PUT
```
- 특정 리소스에서 지원되는 HTTP 메서드 목록 열거
- 새 리소스가 지원했으면 하는 메서드 **추천 용도**로도 요청에 사용 가능
