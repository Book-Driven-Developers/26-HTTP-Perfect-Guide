# HTTP 완벽 가이드 7장 — 캐시(Cache)

## 1. 캐시란

캐시는 자주 쓰이는 문서의 사본을 자동으로 보관하는 HTTP 장치다. 핵심 이점은 다음과 같다.

- **중복 트래픽 제거**: 캐시된 사본이 뒤이은 요청들에 대한 응답으로 사용되어, 원 서버가 중복해서 트래픽을 주고받는 낭비가 줄어든다.
- **네트워크 병목 완화**: 많은 네트워크가 원격 서버보다 로컬 네트워크 클라이언트에 더 넓은 대역폭을 제공하므로 캐시는 네트워크 병목을 줄여준다.
- **서버 부하 감소**: 네트워크 대역폭을 덜 쓰고 웹 서버 부하를 줄여 비용을 절감한다.

## 2. 캐시 적중 / 부적중과 재검사

| 상황 | 설명 |
|------|------|
| 캐시 적중 (Cache Hit) | 요청에 대응하는 사본이 캐시에 있어 바로 응답 |
| 캐시 부적중 (Cache Miss) | 대응되는 사본이 없어 원 서버로 요청이 전달됨 |
| 재검사 적중 (느린 적중) | 사본이 오래되어 재검사한 결과 변경 없음(304) → 사본 제공 |

원 서버 콘텐츠는 변경될 수 있으므로 캐시는 갖고 있는 사본이 여전히 최신인지 서버를 통해 점검해야 하며, 이 신선도 검사를 **HTTP 재검사**라고 한다. 재검사는 모든 요청마다 하는 것이 아니라, 사본이 충분히 오래된 경우에만 진행한다.

재검사 시 세 가지 결과가 나온다.

1. **변경되지 않음** → 서버가 `304 Not Modified` 응답을 보내고, 캐시는 이를 신선하다고 표시한 뒤 클라이언트에 제공한다. 이 작은 응답 덕분에 캐시 적중보다는 느리나 캐시 부적중보다는 빠르다.
2. **변경됨** → 서버는 콘텐츠 전체와 함께 `200 OK` 응답을 보낸다.
3. **삭제됨** → 서버는 `404 Not Found` 응답을 보내고, 캐시는 사본을 삭제한다.

## 3. 캐시 적중률

캐시가 요청을 처리하는 비율을 **캐시 적중률**이라 하며, 0에서 1 사이의 값(또는 퍼센트)으로 표현한다. 현실적으로 적중률이 100%에 가까울수록 좋지만, 오늘날 적중률 40%면 웹 캐시로 괜찮은 편이다.

## 4. 캐시 토폴로지

- **개인 전용 캐시**: 브라우저 내장 캐시처럼 한 사용자 전용
- **공용 프락시 캐시**: 여러 클라이언트들에게 공유된 사본을 제공하여 불필요한 트래픽을 줄임
- **프락시 캐시 계층**: 작은 캐시에서 우선적으로 트래픽을 처리하고, 부적중 시 상위 계층으로 전달

## 5. 캐시 처리 단계 (7단계)

요청을 받으면 캐시는 다음 순서로 처리한다.

1. 요청 받기
2. 파싱
3. 검색
4. 신선도 검사
5. 응답 생성
6. 전송
7. 로깅

응답을 만들 때 주의할 점으로, `Date` 헤더는 객체가 원 서버에서 최초로 생겨난 일시를 표현하기 때문에 조정해선 안 된다.

## 6. 사본을 신선하게 유지하기 (문서 만료)

서버는 응답 헤더로 만료 시점을 알려준다.

- **`Expires`**: 절대적인 만료 날짜를 제공
- **`Cache-Control: max-age=<초>`**: 초 단위의 상대적 만료 시간을 제공

두 방식 중 모든 컴퓨터의 시계가 올바르게 맞춰졌다는 보장이 없으므로 `max-age` 방식이 더 낫다. 시계 차이로 인한 문제를 **클록 스큐(clock skew)**라고 한다.

## 7. 서버 재검사 — 조건부 메서드

### `If-Modified-Since` (IMS, 날짜 기반)

서버에 보내는 GET 요청에 이 헤더를 추가하면, 캐시된 시간 이후에 변경된 경우에만 사본을 보내달라는 의미다.

- 수정되었다면 → `200 OK`로 최신 문서 반환
- 수정되지 않았다면 → 본문 없이 `304 Not Modified` 반환

### `If-None-Match` (엔터티 태그 / ETag 기반)

날짜 대신 콘텐츠의 버전 식별자(ETag)를 비교한다.

### 약한 검사기 / 강한 검사기

ETag에는 사소한 변경을 무시하는 약한 검사기(`W/`)와 바이트 단위 일치를 요구하는 강한 검사기가 있다. `Last-Modified`는 1초 미만 변경을 잡지 못하거나 내용 변화 없이 갱신되는 경우가 있어, 정밀한 비교가 필요할 때 ETag를 사용한다.

## 8. 캐시 제어 응답 헤더

| 헤더 | 의미 |
|------|------|
| `Cache-Control: no-cache` | 캐시 가능하나, 사용 전 반드시 재검사 |
| `Cache-Control: no-store` | 캐시 저장 자체를 금지 |
| `Cache-Control: max-age=<초>` | 신선하다고 간주할 최대 나이 |
| `Cache-Control: must-revalidate` | 만료되면 반드시 재검사 (낡은 사본 제공 금지) |
| `Expires` | 절대 만료 일시 (권장하지 않음) |

서버가 만료 정보를 주지 않으면 **휴리스틱 만료**로 캐시가 추정하며, 클라이언트도 요청 헤더로 신선도 제약(`max-stale`, `min-fresh` 등)을 걸 수 있다.

## 9. 캐시 제어 설정 (7.10)

서버 측에서는 아파치 설정이나 HTML `<meta http-equiv>` 태그로 캐시 동작을 제어할 수 있다. 다만 meta 태그 방식은 프락시 캐시에는 잘 적용되지 않아 권장되지 않는다.

## 10. 자세한 알고리즘 — 나이와 신선도 계산 (7.11)

캐시는 사본을 제공할지 재검사할지를 두 값을 비교해 판단한다.

- **신선도 수명(freshness lifetime)**: 문서가 신선하다고 간주되는 기간 (서버가 `max-age`나 `Expires`로 지정)
- **나이(age)**: 문서가 만들어진 후 지금까지 흐른 시간

판단 규칙은 간단하다.

```
if (문서의 나이 < 신선도 수명)
    아직 신선함 → 캐시 사본을 그대로 제공
else
    오래됨 → 원 서버에 재검사
```

**나이 계산**은 단순해 보이지만 까다롭다. 응답이 네트워크를 지나오는 시간, 여러 프락시 캐시에 머문 시간, 클록 스큐(시계 차이) 등을 모두 고려해야 정확하다. 그래서 책은 보수적으로 나이를 약간 더 크게 추정하는 **완전한 나이 계산 알고리즘**(겉보기 나이와 응답 지연을 합산하는 방식)을 제시한다. 모든 계산의 기준점은 응답이 원 서버에서 생성된 시점을 나타내는 `Date` 헤더다.

## 11. 캐시와 광고 (7.12)

캐시는 트래픽과 부하를 줄여주지만, 비즈니스 측면에서 충돌을 일으킨다.

- **광고 회사의 딜레마**: 광고 수익은 노출(접근) 횟수로 정산되는데, 캐시가 요청을 가로채면 원 서버는 실제 노출 수를 알 수 없게 된다.
- **퍼블리셔의 응답**: 이를 막으려 콘텐츠를 캐시 불가로 설정하거나 매 접근마다 URL을 바꾸는 **'캐시 무력화'** 기법을 쓰는데, 이런 시도는 캐싱의 긍정적인 효과를 감소시킨다.
- **로그 마이그레이션 / 적중 측정**: 보다 협력적인 대안으로, 캐시가 보유한 접근 로그를 원 서버로 보내거나(로그 마이그레이션), 캐시가 일정 주기로 원 서버에 적중 횟수를 보고하는 방식이 제안된다. 캐시 효율을 유지하면서도 노출 측정을 가능하게 하려는 시도다.

---

## 개발자 관점 핵심 요약

실무에서 가장 자주 쓰는 패턴은 다음과 같다.

- `Cache-Control: max-age`로 신선도를 관리
- 만료된 리소스는 `ETag`(`If-None-Match`) 또는 `Last-Modified`(`If-Modified-Since`)로 조건부 요청을 보내 `304`로 대역폭을 절약

제목은 캐시지만 `304`로 응답하는 조건부 요청을 함께 다루며, 웹 프로그래머라면 반드시 써먹게 되는 내용이다.
