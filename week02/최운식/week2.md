우리는 모두 리소스를 구분하기 위한 표준화된 체계를 모든 곳에서 이미 사용중이다.

- 버스 노선, 주소, 전화번호, 주민번호, 책 고유번호 등
- 각기 다른 이름들에 대해 작명 표준에 동의하기 때문에 도시에 있는 리소스를 모두 쉽게 공유할 수 있다.

URL은 인터넷의 리소스를 가리키는 표준 이름이다.

## 2.1 인터넷의 리소스 탐색하기

http://www.naver.com/profile/daemoon.html

- `http`: URL 스키마. 어떻게 리소스에 접근하는지 방법을 알려줌. HTTP 프로토콜을 사용한다는 의미.
- `www.naver.com`: 도메인 명, 서버의 위치이다. 리소스가 어디에 호스팅 되어있는지 알려준다.
- `/profile/daemoon.html`: 리소스 경로이다.
- 구성정보: `[어떻게.어디에.무엇을]`

## 2.2 URL 문법

URL 스킴 문법

- `<스킴>://<사용자이름>:<비밀번호>@<호스트>:<포트>/<경로>;<파라미터>?<질의>#<프래그먼트>`
- 대부분 URL 이 모든 컴포넌트를 갖진 않는다.
- http://anonymous:anonymous@www.naver.com/
  - 이렇게 입력해도 정상 접속이 된다.

- 프래그먼트는 큰 리소스(텍스트 문서 등)에서 특정 절을 가리키는 용도이다.
  - https://github.com/choiwoonsik/weverse-backend-project#2-아키텍처-개요
  - `#2-아키텍처-개요` 여기에 해당.
- 따라서 클라이언트는 서버로 프래그먼트를 전달하지 않는다.

## 2.3 단축 URL

- 상대 URL / 절대 URL
- 상대 URL: Base 를 기반으로 리소스 경로만 입력하면 스키마 & 호스트를 재사용한다.

## 2.4 안전하지 않은 문자

URL에 아무 문자들이나 다 담을 수 있도록 설계하지 않았다. 영어와 숫자만 담는다.

안전하지 않은 문자를 표현하기 위한 인코딩 방식이 고안됐다.

- % + ASCII 코드숫자 로 바꾼다.
  ex)
- 공백 => %20
- % => %25
- ~ => %7E

## 2.6 미래

- URL은 리소스의 주소를 가리키는것이지 리소스의 독립적인 이름을 가리키는건 아니다.
- 주소에 존재하는 리소스가 이동하면 URL이 가리키는 리소스가 없어진다. 찾을수가 없어진다.
- 이사를 갔을 때 기존 URL 주소에 (기존 자원은 이사함. 어디로 가시오) 같은 팻말을 잠시 둘수도 있겠지만 영구적으로 유효한 방법은 아니다. 새로운 리소스가 해당 주소를 차지할 수 있기 때문이다.
- PURL 기법이 논의됐다.
  - 중간에 리소스 위치 중개 서버를 두는 방식이다.
  - PURL은 리소스의 실제 URL 목록을 관리하고 추적한다. 위치(주소)가 바뀌면 그걸로 유지보수한다.
  - PURL은 클라이언트에게서 리소스를 영구적으로 가져올 수 있는 URL로 요청 받는다.
