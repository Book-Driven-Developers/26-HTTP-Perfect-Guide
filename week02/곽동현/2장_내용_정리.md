# 2.1 인터넷의 리소스 탐색하기

## URL이란?

**URL(Uniform Resource Locator)** 은 인터넷의 리소스를 가리키는 표준 이름이다.  
URL은 전자정보 일부를 가리키고, 그것이 어디에 있고 어떻게 접근할 수 있는지 알려준다.

---

## URI, URL, URN의 관계

```
URI (Uniform Resource Identifier)
├── URL (Uniform Resource Locator) → 위치로 리소스를 식별
└── URN (Uniform Resource Name)  → 이름으로 리소스를 식별 (위치 무관)
```

- **URI** 는 URL과 URN을 포함하는 더 일반화된 개념이다.
- **URL** 은 리소스가 **어디에 있는지** 설명해서 리소스를 식별한다.
- **URN** 은 리소스가 어디에 있든 상관없이 **이름만으로** 리소스를 식별한다.
- HTTP 명세에서는 URI를 더 일반화된 개념으로 사용하지만, 실제 HTTP 애플리케이션은 URL을 URI의 한 부분으로 취급한다.

---

## URL의 구조 (예시)

```
http://www.joes-hardware.com/seasonal/index-fall.html
```

| 부분 | 값 | 의미 |
|------|-----|------|
| 스킴 | `http` | 어떻게 접근할 것인가 (프로토콜) |
| 호스트 | `www.joes-hardware.com` | 어디에 있는가 (서버 위치) |
| 경로 | `/seasonal/index-fall.html` | 무엇을 원하는가 (리소스 경로) |

---

## 2.1.1 URL이 있기 전 암흑의 시대

URL 등장 이전에는 리소스에 접근하기 위해 각 애플리케이션마다 다른 방식을 사용했다.

**예시: FTP로 파일 공유 시 필요한 절차**
```
1. "ftp.joes-hardware.com에 FTP로 접속해"
2. "익명 사용자로 로그인"
3. "비밀번호로 네 이름을 입력해"
4. "pub 디렉터리로 이동"
5. "바이너리 형식으로 전환"
6. "complete-catalog.xls 파일을 로컬로 받아"
```

URL 등장 이후:
```
ftp://ftp.lots-o-books.com/pub/complete-catalog.xls를 열어봐
```

URL은 애플리케이션이 리소스에 접근할 수 있는 방법을 **단일 인터페이스** 로 제공한다.  
브라우저는 URL 하나로 HTTP, FTP, SMTP 등 다양한 프로토콜을 추상화한다.

---

## URL이 지원하는 다양한 프로토콜

| 예시 URL | 프로토콜 | 설명 |
|----------|----------|------|
| `http://www.joes-hardware.com/` | HTTP | 웹 페이지 |
| `mailto:president@whitehouse.gov` | SMTP | 이메일 |
| `ftp://ftp.lots-o-books.com/pub/catalog.xls` | FTP | 파일 전송 |
| `rtsp://www.joes-hardware.com:554/interview/cto_video` | RTSP | 스트리밍 |

---

# 2.2 URL 문법

## URL의 9가지 컴포넌트

대부분의 URL 스킴 문법은 일반적으로 9개 부분으로 나뉜다.

```
<스킴>://<사용자 이름>:<비밀번호>@<호스트>:<포트>/<경로>;<파라미터>?<질의>#<프래그먼트>
```

| 컴포넌트 | 설명 | 기본값 |
|----------|------|--------|
| **스킴** | 리소스를 가져오려면 어떤 프로토콜을 사용해야 하는지 | 없음 |
| **사용자 이름** | 몇몇 스킴은 리소스에 접근하기 위해 사용자 이름 필요 | `anonymous` |
| **비밀번호** | 사용자 이름 뒤에 콜론(`:`)으로 이어서 기술 | `<이메일 주소>` |
| **호스트** | 리소스를 호스팅하는 서버의 호스트 명이나 IP 주소 | 없음 |
| **포트** | 서버가 열어놓은 네트워크 포트번호. HTTP 기본 포트: 80 | 스킴에 따라 다름 |
| **경로** | 서버 내 리소스가 어디에 있는지. `/`로 구분된 경로조각 | 없음 |
| **파라미터** | 이름/값 쌍. 경로 및 다른 파라미터와 세미콜론(`;`)으로 구분 | 없음 |
| **질의** | 애플리케이션(DB, 게이판, 검색엔진 등)에 파라미터를 전달. `?`로 구분 | 없음 |
| **프래그먼트** | 리소스의 조각이나 일부분. `#`으로 구분. **서버에 전달되지 않음** | 없음 |

> **핵심:** 모든 컴포넌트를 가지는 URL은 거의 없다. 가장 중요한 세 가지는 **스킴, 호스트, 경로** 이다.

---

## 2.2.1 스킴: 사용할 프로토콜

스킴은 주어진 리소스에 어떻게 접근하는지 알려주는 중요한 정보이다.

- 알파벳으로 시작해야 하며, 나머지 부분과 **첫 번째 `:` 문자** 로 구분한다.
- 대소문자를 가리지 않는다: `http`와 `HTTP`는 같다.

```
http://www.joes-hardware.com/index.html
```

---

## 2.2.2 호스트와 포트

```
http://www.joes-hardware.com:80/index.html
http://161.58.228.45:80/index.html   ← 동일한 리소스 (IP 주소로)
```

- **호스트:** 접근하려는 리소스를 가지고 있는 인터넷상의 호스트 장비 (호스트명 또는 IP 주소)
- **포트:** 서버가 열어놓은 네트워크 포트 번호
- HTTP는 내부적으로 TCP 프로토콜을 사용하며, **기본 포트는 80** 이다.

---

## 2.2.3 사용자 이름과 비밀번호

```
ftp://ftp.prep.ai.mit.edu/pub/gnu                      ← 사용자 이름/비밀번호 없음 → 기본값 사용
ftp://anonymous@ftp.prep.ai.mit.edu/pub/gnu            ← 사용자 이름만 (anonymous)
ftp://anonymous:my_passwd@ftp.prep.ai.mit.edu/pub/gnu  ← 사용자 이름 + 비밀번호
```

- `@` 문자는 URL로부터 사용자 이름과 비밀번호 컴포넌트를 분리한다.
- FTP처럼 사용자 이름/비밀번호를 요구하는 스킴에서 생략하면, 기본값(`anonymous` / 브라우저 기본값)을 사용한다.

---

## 2.2.4 경로

```
http://www.joes-hardware.com:80/seasonal/index-fall.html
```

- 경로: `/seasonal/index-fall.html` (유닉스 파일 시스템 경로와 유사한 구조)
- HTTP URL에서 경로 컴포넌트는 `/` 문자를 기준으로 **경로조각(path segment)** 으로 나뉜다.
- 각 경로조각은 자체만의 파라미터 컴포넌트를 가질 수 있다.

---

## 2.2.5 파라미터

많은 스킴이 호스트 및 경로 정보만으로는 리소스를 찾지 못한다.  
URL의 파라미터 컴포넌트는 **이름/값 쌍의 리스트** 로, `;` 문자로 구분한다.

```
ftp://prep.ai.mit.edu/pub/gnu;type=d
```
→ `type=d` 파라미터: 디렉터리 형식으로 전송

```
http://www.joes-hardware.com/hammers;sale=false/index.html;graphics=true
```
→ 두 개의 경로조각 각각이 별도의 파라미터를 가짐:
- `/hammers` → `sale=false`
- `/index.html` → `graphics=true`

---

## 2.2.6 질의 문자열

데이터베이스 같은 서비스들은 요청받을 리소스 형식의 범위를 좁히기 위해 질문이나 질의를 받을 수 있다.

```
http://www.joes-hardware.com/inventory-check.cgi?item=12731
```

- `?` 오른쪽에 있는 값들이 질의 컴포넌트이다.
- 질의 컴포넌트는 URL의 경로 컴포넌트와 함께 게이트웨이로 전달된다.
- 보통 `&`로 나뉜 **이름=값** 쌍 형식으로 사용한다.

```
http://www.joes-hardware.com/inventory-check.cgi?item=12731&color=blue
```
→ `item=12731`과 `color=blue` 두 개의 질의 컴포넌트

> Spring의 `@RequestParam`으로 받는 쿼리 파라미터가 바로 이 질의 컴포넌트이다.

---

## 2.2.7 프래그먼트

HTML 같은 리소스 형식들은 본래의 수준보다 더 작게 나뉠 수 있다.  
URL은 리소스 내의 조각을 가리킬 수 있는 **프래그먼트 컴포넌트** 를 제공한다.

```
http://www.joes-hardware.com/tools.html#drills
```

- `#` 문자 이후가 프래그먼트이다.
- **프래그먼트는 서버에 전달되지 않는다.**
- 클라이언트(브라우저)가 전체 리소스를 내려받은 후, 프래그먼트를 사용하여 해당 부분을 보여준다.

```
[흐름]
(a) 사용자가 http://www.joes-hardware.com/tools.html#drills 링크 선택
(b) 브라우저는 http://www.joes-hardware.com/tools.html 을 요청 (프래그먼트 제외)
(c) 서버는 전체 HTML 페이지를 반환
(d) 브라우저는 'drills' 프래그먼트 위치부터 스크롤하여 보여줌
```

---

# 2.3 단축 URL

웹 클라이언트는 몇몇 단축 URL을 인식하고 사용한다.  
URL은 **상대 URL** 과 **절대 URL** 두 가지로 나뉜다.

---

## 2.3.1 상대 URL

### 절대 URL vs 상대 URL

| 구분 | 설명 | 예시 |
|------|------|------|
| **절대 URL** | 리소스에 접근하는 데 필요한 모든 정보를 가지고 있음 | `http://www.joes-hardware.com/tools.html` |
| **상대 URL** | 모든 정보를 담고 있지 않음. 기저(base) URL을 사용해야 함 | `./hammers.html` |

상대 URL은 URL을 짧게 표기하는 방식이다. HTML을 직접 작성할 때 매우 유용하다.

**예시: HTML 코드 내 상대 URL**
```html
<HTML>
<HEAD><TITLE>Joe's Tools</TITLE></HEAD>
<BODY>
<H1> Tools Page </H1>
<H2> Hammers <H2>
<P> Joe's Hardware Online has the largest selection of
<A HREF="./hammers.html">hammers</A> on earth.
</BODY>
</HTML>
```

위 HTML은 `http://www.joes-hardware.com/tools.html`에 있는 리소스이다.  
`./hammers.html`이라는 상대 URL은 기저 URL을 통해 절대 URL로 변환된다.

---

### 기저 URL

변환 과정의 첫 단계는 기저 URL을 찾는 것이다.

**기저 URL을 가져오는 방법:**

1. **리소스에서 명시적으로 제공**  
   HTML 문서 내 `<BASE>` 태그로 기저 URL을 명시할 수 있다.

2. **리소스를 포함하고 있는 기저 URL**  
   기저 URL이 명시되지 않은 경우, 해당 리소스의 URL을 기저 URL로 사용한다.

3. **기저 URL이 없는 경우**  
   보통 절대 URL만으로 이루어진 경우이다. 하지만 불완전하거나 깨진 URL일 수도 있다.

---

### 상대 참조 해석하기 (알고리즘)

상대 URL → 절대 URL 변환은 **RFC 1808, RFC 2396** 에 정의된 알고리즘을 사용한다.

**예시:**
```
기저 URL: http://www.joes-hardware.com/tools.html
상대 URL: ./hammers.html

변환 과정:
1. 경로: ./hammers.html, 기저 URL: http://www.joes-hardware.com/tools.html
2. 스킴이 비어있다 → 기저 URL의 스킴 상속 (http)
3. 컴포넌트가 비어있지 않다 → 호스트와 포트 상속 (www.joes-hardware.com, 80)
4. 합치면: http://www.joes-hardware.com/hammers.html
```

---

## 2.3.2 URL 확장

어떤 브라우저들은 URL을 입력한 다음이나 입력하고 있는 동안에 자동으로 URL을 확장한다.

### 호스트 명 확장

- 주소 입력란에 `yahoo`를 입력하면 → 브라우저가 자동으로 `www.yahoo.com`으로 확장
- 단순한 **휴리스틱** 기반 기능

> **주의:** 호스트 명에 대한 확장 기능은 프락시와 같은 다른 HTTP 애플리케이션에 문제를 발생시킬 수 있다. (6장에서 자세히 다룬다)

### 히스토리 확장

- 브라우저가 과거에 사용자가 방문했던 URL의 기록을 저장한다.
- URL을 입력하면, 입력한 글자들을 포함하는 완결된 형태의 URL들을 선택하게 해준다.
- 예: `http://www.joes-`를 입력 → `http://www.joes-hardware.com` 추천

# 2.4 안전하지 않은 문자

URL은 잘 호환되도록 설계되었다. 모든 프로토콜이 데이터를 전송하기 위해 서로 다른 장치를 가지고 있기 때문에, 어떤 인터넷 프로토콜을 통해서든 안전하게 전송될 수 있도록 URL을 설계하는 것은 중요했다.

---

## 2.4.1 URL 문자 집합

- 컴퓨터 시스템의 기본 문자 집합은 역사적으로 **US-ASCII** 를 사용해왔다.
- US-ASCII는 7비트를 사용하여 영문 자판의 키 대부분과 출력되지 않는 제어 문자를 표현한다.
- US-ASCII는 적은 수의 문자만 포함하고 있어, 유럽 언어나 비 라틴계 언어의 변형된 문자들을 지원하지 않는다.

→ 이를 지원하기 위해 URL 설계자들은 URL에 **이스케이프 문자열** 을 쓸 수 있게 설계하였다.  
이스케이프 문자열은 US-ASCII에서 사용이 금지된 문자들을, 특정 문자나 데이터를 인코딩할 수 있게 함으로써 이동성과 완성도를 높였다.

---

## 2.4.2 인코딩 체계

안전하지 않은 문자들을 URL에 표현할 수 있는 인코딩 방식이 고안되었다.

> **인코딩 방식:** 안전하지 않은 문자를 `%`로 시작해, ASCII 코드로 표현되는 두 개의 16진수 숫자로 이루어진 **이스케이프 문자** 로 바꾼다.

**예시:**

| 문자 | ASCII 코드 | URL 예 |
|------|-----------|--------|
| `~` | 126 (0x7E) | `http://www.joes-hardware.com/%7Ejoe` |
| 빈 문자(공백) | 32 (0x20) | `http://www.joes-hardware.com/more%20tools.html` |
| `%` | 37 (0x25) | `http://www.joes-hardware.com/100%25satisfaction.html` |

---

## 2.4.3 문자 제한

몇몇 문자는 URL 내에서 특별한 의미로 예약되어 있다.  
예약된 문자를 **본래의 목적이 아닌 다른 용도** 로 사용하려면 반드시 인코딩해야 한다.

| 문자 | 선점 및 제한 |
|------|-------------|
| `%` | 인코딩된 문자에 사용할 이스케이프 토큰으로 선점 |
| `/` | 경로 컴포넌트에서 경로 세그먼트를 나누는 용도로 선점 |
| `.` | 경로 컴포넌트에서 선점 |
| `..` | 경로 컴포넌트에서 선점 |
| `#` | 프래그먼트의 구획 문자로 선점 |
| `?` | 질의 문자열의 구획 문자로 선점 |
| `;` | 파라미터의 구획 문자로 선점 |
| `:` | 스킴, 사용자 이름/비밀번호, 호스트/포트의 구획 문자로 선점 |
| `$ , +` | 선점 |
| `@ & =` | 특정 스킴에서 특별한 의미가 있기 때문에 선점 |
| `{ } \ \| \ ^ [ ] \`` | 게이트웨이와 같은 여러 전송 에이전트에서 불안전하게 다루기 때문에 제한됨 |
| `< > "` | 안전하지 않음. URL 범위 밖에서 역할이 있는 문자이기 때문에 반드시 인코딩해야 한다 |
| `0x00~0x1F, 0x7F` | 제한됨. 인쇄되지 않는 US-ASCII 문자 |
| `> 0x7F` | 제한됨. 7비트 US-ASCII 문자가 아님 |

---

## 2.4.4 좀 더 알아보기

- 예전에 안전하지 않은 문자를 URL에 사용해도 문제가 없었던 것처럼 보이는 경우도 있다.
    - 예: `http://www.joes-hardware.com/~joe`의 `~` 문자는 인코딩하지 않아도 동작하기도 한다.
    - 하지만 **애플리케이션 개발자가 안전하지 않은 문자를 인코딩하지 않는 것은 실수** 이다.

**인코딩 결정 원칙:**
- 입력받은 URL에서 어떤 문자를 인코딩할지 결정하는 것은 **최초로 URL을 입력받는 애플리케이션(브라우저 등)에서 하는 것이 가장 적절** 하다.
- 각 URL 컴포넌트마다 사용할 수 있거나 없는 문자들이 있고, 스킴에 따라 가용성이 달라지기 때문이다.

> **주의:** 악의적인 사람들은 URL의 패턴 매칭을 하는 애플리케이션(필터링 애플리케이션)을 회피하여 다른 문자들을 인코딩하기도 한다.  
> 안전한 URL 컴포넌트를 인코딩하면, 패턴 매칭 애플리케이션이 원하는 패턴을 찾지 못하는 원인이 될 수 있다.

---

# 2.5 스킴의 바다

URL로 인터넷상의 모든 리소스를 찾을 수 있지만, 그 리소스들은 다른 스킴(HTTP, FTP, SMTP 등)을 통해 접근할 수 있다.  
대부분의 URL은 동일하게 **`스킴://서버위치/경로`** 구조로 이루어져 있다.

---

## 주요 스킴 목록

| 스킴 | 설명 | 기본형식 | 예시 |
|------|------|----------|------|
| **http** | 하이퍼텍스트 전송 프로토콜. 사용자 이름/비밀번호 없음. 기본 포트 80 | `http://<호스트>:<포트>/<경로>?<질의>#<프래그먼트>` | `http://www.joes-hardware.com/index.html` |
| **https** | http와 거의 같으나 HTTP 커넥션 양 끝단에서 **SSL** 을 사용해 암호화. 기본 포트 443 | `https://<호스트>:<포트>/<경로>?<질의>#<프래그먼트>` | `https://www.joes-hardware.com/secure.html` |
| **mailto** | 이메일 주소를 가리킴. 다른 스킴과 다르게 동작하기 때문에 일반 URL 포맷과 다른 포맷. RFC 822에 기술 | `mailto:<RFC-822-addr-spec>` | `mailto:joe@joes-hardware.com` |
| **ftp** | 파일 전송 프로토콜. FTP 서버에 있는 파일을 내려받거나 올리고, FTP 서버의 디렉터리 콘텐츠 목록을 가져오는 데 사용 | `ftp://<사용자 이름>:<비밀번호>@<호스트>:<포트>/<경로>;<파라미터>` | `ftp://anonymous:joe%40joes-hardware.com@prep.ai.mit.edu:21/pub/gnu/` |
| **rtsp, rtspu** | 실시간 스트리밍 프로토콜(RTSP)을 통해 읽을 수 있는 오디오 및 비디오 미디어 리소스 식별자. `rtspu`의 `u`는 UDP 프로토콜 사용 의미 | `rtsp://<사용자 이름>:<비밀번호>@<호스트>:<포트>/<경로>` | `rtsp://www.joes-hardware.com:554/interview/cto_video` |
| **file** | 주어진 호스트 기기에서 바로 접근할 수 있는 파일들을 나타냄. 호스트가 생략되면 URL을 사용하는 기기의 로컬 호스트가 기본값 | `file://<호스트>/<경로>` | `file://OFFICE-FS/policies/casual-fridays.doc` |
| **news** | RFC 1036에 정의. 특정 뉴스 그룹 또는 뉴스 서버의 특정 기사에 접근하는 데 사용. 리소스의 위치 정보를 충분히 포함하지 않는 특이한 속성 | `news:<newsgroup>` 또는 `news:<news-article-id>` | `news:rec.arts.startrek` |
| **telnet** | 대화형 서비스에 접근하는 데 사용. telnet URL 자체가 객체를 가리키지는 않지만, 리소스라고 할 수 있는 대화형 애플리케이션은 telnet 프로토콜로 접근 가능 | `telnet://<사용자 이름>:<비밀번호>@<호스트>:<포트>/` | `telnet://slurp:webhound@joes-hardware.com:23/` |

---

> **부록 A** 에는 스킴 목록이 상당히 자세히 나열되어 있으며, 스킴 각각에 대한 문서 정보가 기술되어 있다.

# 2.6 미래 — URL의 한계와 URN

## URL의 한계

URL은 강력한 도구이지만 완벽하지 않다.

- URL은 **주소** 이지 **실제 이름** 이 아니다.
- URL은 특정 시점에 어떤 것이 위치한 곳을 알려준다.
- **리소스가 옮겨지면 URL을 더는 사용할 수 없다.**
    - 기존 URL이 가리키고 있던 객체를 찾을 방법이 없어진다.

---

## URN (Uniform Resource Name)

**IETF(Internet Engineering Task Force)** 는 이 문제를 해결하기 위해 **URN(Uniform Resource Names)** 이라는 새로운 표준 작업에 착수하였다.

- URN은 객체가 옮겨지더라도 (웹 서버 내에서나 웹 서버 간 모두) 항상 객체를 가리킬 수 있는 이름을 제공한다.
- **위치가 아닌 이름** 으로 리소스를 식별한다.

---

## PURL (Persistent Uniform Resource Locators)

**PURL(지속 통합 자원 지시자)** 을 사용하면 URL로 URN의 기능을 제공할 수 있다.

**동작 방식:**
1. PURL은 리소스의 실제 URL 목록을 관리하고 추적하는 **리소스 위치 중개 서버** 를 둔다.
2. 클라이언트는 위치 할당자에게 리소스를 가져올 수 있는 **영구적인 URL** 을 요청한다.
3. 영구적인 URL은 클라이언트를 리소스의 실제 URL로 연결해준다.

```
[흐름]
클라이언트 → PURL 서버 (영구 URL 요청)
            → PURL 서버가 실제 URL로 리다이렉트
            → 클라이언트가 실제 리소스 접근
```

---

## 2.6.1 지금이 아니면, 언제?

한동안 URN 방식이 활용되었다. 하지만 왜 아직 채택되지 않았는가?

- URL에서 URN으로 주소 체계를 바꾸는 것은 **매우 큰 작업** 이다.
- URN을 지원하려면 많은 변화가 필요하다:
    - 표준 제정
    - 여러 HTTP 애플리케이션을 수정하기 위한 **벤더들과의 합의**
- 변화를 만들어내기 위해서는 엄청난 작업량이 필요하다.
- 유감스럽게도 URN으로 전환하기 위한 모든 것이 아직 갖춰지지 않은 상태이다.

> **핵심:** URL은 현재로서는 여전히 표준이며, URN은 미래의 가능성이다.
