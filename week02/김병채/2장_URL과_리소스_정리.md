# 2장 URL과 리소스

## 1. URL 컴포넌트 종류, 설명, 기본값 정리
URL은 일반적으로 다음과 같은 구조를 가집니다:
`<스킴>://<사용자이름>:<비밀번호>@<호스트>:<포트>/<경로>;<파라미터>?<질의>#<프래그먼트>`

| 컴포넌트 | 설명 | 기본값 |
| --- | --- | --- |
| **스킴 (Scheme)** | 리소스를 가져오기 위해 사용할 프로토콜 (예: `http`, `https`, `ftp`) | 없음 (필수) |
| **사용자 이름 (User)** | 일부 스킴(예: ftp)에서 리소스 접근을 위해 요구하는 사용자 이름 | `anonymous` (ftp의 경우) |
| **비밀번호 (Password)** | 사용자의 비밀번호 | `<이메일 주소>` (ftp의 경우) |
| **호스트 (Host)** | 리소스를 호스팅하는 서버의 호스트 명이나 IP 주소 | 없음 (필수) |
| **포트 (Port)** | 서버가 수신하고 있는 포트 번호 | `http`: 80, `https`: 443 |
| **경로 (Path)** | 서버 내 리소스의 로컬 경로 | `/` |
| **파라미터 (Parameters)** | 애플리케이션이 정확한 요청을 처리하기 위해 필요한 이름/값 쌍 (`;`로 구분) | 없음 |
| **질의 (Query)** | 리소스 형식을 좁히거나 파라미터를 전달하기 위한 질의 문자열 (`?`로 시작, `&`로 구분) | 없음 |
| **프래그먼트 (Fragment)** | 리소스 내부의 특정 조각(부분)을 가리키는 앵커 (`#`로 시작). 클라이언트에서만 사용되며 서버로는 전송되지 않음 | 없음 |

## 2. RFC 2396: 상대 참조(Relative Reference) 해석 알고리즘
상대 참조는 Base URL을 기준으로 절대 URL로 변환(해석)되어야 합니다. 변환 과정은 대략적으로 다음과 같습니다:

1. **Base URL 추출**: 문서의 명시적 Base URL이 있는지 확인하고, 없다면 해당 문서의 URL을 Base URL로 사용합니다.
2. **스킴 확인**: 상대 URL에 스킴이 명시되어 있다면 그 자체로 절대 URL로 간주하고 종료합니다.
3. **컴포넌트 병합**:
   - 상대 URL이 `//`로 시작하면 (네트워크 경로 참조), Base URL의 스킴을 상속하고 나머지는 상대 URL을 따릅니다.
   - 상대 URL이 `/`로 시작하면 (절대 경로 참조), Base URL의 스킴, 인증, 호스트, 포트를 상속하고 경로부터 상대 URL을 따릅니다.
   - 상대 URL이 그 외의 문자로 시작하면 (상대 경로 참조), Base URL의 경로의 마지막 슬래시(`/`) 이후를 제거하고 상대 URL의 경로를 덧붙입니다.
4. **경로 정규화**: 병합된 경로에서 `.` (현재 디렉터리)와 `..` (상위 디렉터리)를 처리하여 경로를 정규화합니다.
   - `/./`는 `/`로 변환합니다.
   - `/디렉터리/../`는 `/`로 변환하여 상위 디렉터리로 이동합니다.

## 3. URL 인코딩 체계, 예약어, 이스케이프 방법
URL은 안전하게 전송될 수 있도록 US-ASCII 문자셋만 사용해야 합니다. 

* **예약어 (Reserved Characters)**: URL에서 특별한 의미를 가지는 문자들. (`;`, `/`, `?`, `:`, `@`, `&`, `=`, `+`, `$`, `,` 등)
* **비예약어 (Unreserved Characters)**: 알파벳(A-Z, a-z), 숫자(0-9), 하이픈(`-`), 언더스코어(`_`), 마침표(`.`), 틸드(`~`)
* **이스케이프(퍼센트 인코딩, Percent-Encoding)**: 
  * 안전하지 않은 문자나 예약어를 URL 내에서 일반 데이터로 사용할 경우 이스케이프해야 합니다.
  * 인코딩 방법: 해당 문자의 아스키 코드(또는 UTF-8) 값을 16진수로 표현하고 앞에 `%`를 붙입니다.
  * 예: 공백(Space)은 `%20`, `~`는 `%7E` 등으로 변환됩니다.

## 4. URL에서 값을 전달할 때 Base64 인코딩을 사용하는 이유
URL 내부에 바이너리 데이터나 URL에서 사용할 수 없는 특수 문자가 포함된 데이터를 안전하게 전달하기 위해 Base64 인코딩을 사용합니다.

* **Base64 인코딩이란?**: 8비트 이진 데이터를 문자 코드에 영향을 받지 않는 공통 ASCII 영역의 문자들로만 이루어진 일련의 문자열로 바꾸는 인코딩 방식입니다. 64개의 출력 가능한 문자(A-Z, a-z, 0-9, `+`, `/`)와 패딩 문자(`=`)를 사용합니다.
* **사용 이유**:
  1. **안전성 확보**: HTTP URL에 포함할 수 없는 제어 문자나 비-ASCII 문자(한글, 이미지 바이너리 등)를 ASCII 문자열로 안전하게 변환하여 데이터 손실이나 오동작을 방지합니다.
  2. **URL Safe Base64 변환**: 기존 Base64의 `+`와 `/` 문자는 URL에서 특별한 의미(예약어)를 가지므로, URL 파라미터로 전달할 때는 주로 `+`를 `-`로, `/`를 `_`로 대체하는 'URL-safe Base64' 변환을 수행하여 URL 파싱 오류를 막습니다.
