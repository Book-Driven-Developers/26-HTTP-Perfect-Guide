# 2장 디깅(Deep Dive) 정리: 프로토콜과 네트워크 환경의 진화

## 1. 주요 프로토콜 요약
* **FTP**: 파일 전송용 표준. 보안 취약(평문 전송), 2개의 포트(제어, 데이터) 사용.
* **RTSP / RTSPU**: 멀티미디어 스트리밍 제어 프로토콜. 재생/일시정지 등의 명령어 담당. RTSPU는 UDP 기반.
* **Telnet**: 원격 접속(CLI) 제어. 암호화가 없어 현재는 대부분 SSH로 대체됨.
* **SOAP**: 웹 서비스 통신을 위한 XML 기반 프로토콜. 무겁고 복잡하여 최근엔 REST API로 많이 대체됨.
* **SMTP**: 이메일 전송(송신)을 위한 표준 프로토콜.

## 2. 브라우저에서 FTP가 사라지고 HTTP가 대세가 된 이유
과거에는 브라우저도 FTP를 지원했지만 현재는 모두 제거되었고, HTTP가 다운로드의 표준이 되었습니다.
* **NAT/방화벽 문제**: FTP는 구조상(특히 2개의 포트 사용) 공유기/방화벽 환경에서 오류가 잦음. HTTP는 80/443 단일 포트로 문제없이 통과함.
* **HTTP의 발전**: HTTP/1.1부터 `Range` 헤더가 도입되어 이어받기 등 기존 FTP의 장점을 완벽히 흡수.
* **전용 프로그램(FileZilla 등)**: 서버 관리를 위해서는 단순 다운로드가 아닌 세밀한 디렉터리 제어와 강력한 보안이 필요하므로 FTP가 아닌 **SFTP(SSH 기반)** 전용 클라이언트를 사용함.

## 3. HTTP `Range` 헤더
"파일 전체가 아닌 특정 범위의 데이터만 줘"라고 요청하는 기능입니다.
* **용도**: 다운로드 이어받기, 동영상/오디오 부분 탐색(스트리밍), 분할 다운로드
* **작동 방식**: 클라이언트가 `Range: bytes=4000-` 와 같이 요청하면, 서버는 `206 Partial Content` 상태 코드와 함께 `Content-Range: bytes 4000-9999/10000` 헤더를 붙여서 조각 데이터만 응답함.

## 4. 스트리밍 프로토콜의 진화 (RTSP/UDP에서 HTTP/TCP로)
과거에는 속도와 끊김 방지를 위해 UDP/RTSP를 썼으나, 현재 유튜브/넷플릭스는 HTTP(TCP)를 사용합니다.
* **HTTP/TCP의 승리 요인**: 
  1. 방화벽 문제 없음 (80/443 포트)
  2. 전 세계에 깔린 거대한 웹 캐시 인프라(CDN) 활용의 용이성
  3. 인터넷 속도 향상으로 TCP 오버헤드 감당 가능
* **ABR(적응형 비트레이트 스트리밍)**: 영상을 수초 단위의 아주 작은 파일들로 쪼갠 뒤, 네트워크 상태에 맞춰 알맞은 화질의 조각을 HTTP 통신으로 연속해서 다운로드(Range 활용)하는 방식. 

## 5. CDN과 WebRTC
* **CDN (Content Delivery Network)**: 데이터(영상 조각 등)를 사용자와 가장 가까운 전 세계 통신사 내부(Edge Server)에 미리 복사해두는 시스템. 원본 서버의 트래픽을 분산시키고 지연시간을 획기적으로 줄여줌.
* **WebRTC**: 브라우저끼리 중간 서버를 거치지 않고 P2P로 직접 연결하는 실시간 통신 기술. 스트리밍의 버퍼링 딜레이(3~10초)를 허용할 수 없는 화상회의(Zoom, Meet) 등에서 초저지연을 달성하기 위해 사용되며, 빠른 통신을 위해 **UDP**를 기반으로 함.

## 6. 공유기(NAT)와 HTTP의 천하통일
* **IPv4 고갈과 NAT**: 진짜 IP 주소(공인 IP)의 부족을 해결하기 위해, 공유기는 외부 통신용 공인 IP 1개를 쓰고 내부 기기들엔 사설 IP를 부여함.
* **P2P 통신의 어려움**: 이 구조 때문에 외부에선 내 기기의 진짜 주소를 알 수 없어, 먼저 통신을 걸어올 수 없음. 즉, 양방향 통신을 요구하는 프로토콜(P2P, Active FTP 등)은 동작하기 매우 어려워짐.
* **HTTP의 적합성**: 무조건 클라이언트가 먼저 요청하고 서버가 응답하는 단방향 구조인 HTTP는 공유기(NAT) 환경을 통과하는 데 최적이었음. 결과적으로 채팅, 영상 스트리밍 등 사실상 모든 인터넷 서비스가 HTTP 통신 위로 올라가게 됨.
