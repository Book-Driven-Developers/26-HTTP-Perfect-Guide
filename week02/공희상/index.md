# 2장 URL과 리소스

택시 기사에게 246번지 맥엘리스터 거리라는 특정 주소 가달라고 하면, 택시 기사는 어느 곳을 말하는지 이해한다.
이것이 가능한 이유가 해당 주소의 표준 이름이 있기 때문이다.
이처럼 URL은 인터넷의 [리소스](https://datatracker.ietf.org/doc/html/rfc2396#section-1.1:~:text=%EB%A6%AC%EC%86%8C%EC%8A%A4%EB%8A%94%20%EC%A0%95%EC%B2%B4%EC%84%B1%EC%9D%84%20%EA%B0%80%EC%A7%84%20%EB%AA%A8%EB%93%A0%20%EA%B2%83%EC%9D%84%20%EC%9D%98%EB%AF%B8%ED%95%A0%20%EC%88%98%20%EC%9E%88%EC%8A%B5%EB%8B%88%EB%8B%A4.%20%EC%B9%9C%EC%88%99%ED%95%9C%0A%20%20%20%20%20%20%20%20%20%EC%98%88%EC%8B%9C%EB%A1%9C%EB%8A%94%20%EC%A0%84%EC%9E%90%20%EB%AC%B8%EC%84%9C%2C%20%EC%9D%B4%EB%AF%B8%EC%A7%80%2C%20%EC%84%9C%EB%B9%84%EC%8A%A4%20%EB%93%B1%EC%9D%B4%20%EC%9E%88%EC%8A%B5%EB%8B%88%EB%8B%A4.%0A%20%20%20%20%20%20%20%20%20(%EC%98%88%3A%20%22%EC%98%A4%EB%8A%98%20%EB%A1%9C%EC%8A%A4%EC%95%A4%EC%A0%A4%EB%A0%88%EC%8A%A4%20%EB%82%A0%EC%94%A8%20%EC%98%88%EB%B3%B4%22)%2C%20%EA%B7%B8%EB%A6%AC%EA%B3%A0%0A%20%20%20%20%20%20%20%20%20%EB%8B%A4%EB%A5%B8%20%EB%A6%AC%EC%86%8C%EC%8A%A4%EB%93%A4%EC%9D%98%20%EB%AA%A8%EC%9D%8C%EC%9E%85%EB%8B%88%EB%8B%A4.%20%EB%AA%A8%EB%93%A0%20%EB%A6%AC%EC%86%8C%EC%8A%A4%EA%B0%80%20%EB%84%A4%ED%8A%B8%EC%9B%8C%ED%81%AC%20%EB%A6%AC%EC%86%8C%EC%8A%A4%EB%8A%94%20%EC%95%84%EB%8B%99%EB%8B%88%EB%8B%A4.%0A%20%20%20%20%20%20%20%20%20%22%ED%9A%8C%EC%88%98%20%EA%B0%80%EB%8A%A5%ED%95%9C%22%3B%20%EC%98%88%3A%20%EC%9D%B8%EA%B0%84%2C%20%EA%B8%B0%EC%97%85%2C%20%EA%B7%B8%EB%A6%AC%EA%B3%A0%20%EA%B5%AC%EC%86%8D%EB%90%9C%0A%20%20%20%20%20%20%20%20%20%EB%8F%84%EC%84%9C%EA%B4%80%EC%97%90%20%EC%9E%88%EB%8A%94%20%EC%B1%85%EB%93%A4%EB%8F%84%20%EC%9E%90%EC%9B%90%EC%9C%BC%EB%A1%9C%20%EA%B0%84%EC%A3%BC%EB%90%A0%20%EC%88%98%20%EC%9E%88%EC%8A%B5%EB%8B%88%EB%8B%A4.)를 가리키는 표준 이름이다.

## 2.1 인터넷의 리소스 탐색하기

URL은 브라우저가 정보를 찾는데 필요한 리소스의 위치를 가리키며, URL을 통해 HTTP 및 다른 프로토콜를 통해 접근할 수 있다.

URL이 없었을 때는 어땠을까?
웹과 URL이 있기 전에는 친구와 complete-catalog.xls이라는 파일을 공유하려고 했다면 이렇게 말해야 했을 것이다.

> "ftp.joes-hardware.com에 FTP로 접속해. 익명 사용자로 로그인한 다음 비밀번호로 네 이름을 입력해. pub 디렉토리로 이동한 다음, 바이너리 형식으로 전환해. 이제 complete-catalog.xls란 이름의 파일을 너의 로컬 파일 시스템에 내려 받은 다음 보면 될 거야."

오늘날 브라우즈들은 URL을 사용해서 하나의 인터페이스를 통해 일관된 방식으로 리소스에 접근할 수 있게 했다.

> "ftp://ftp.lots-o-books.com/pub/complete-catalog.xls를 열어봐"

## 2.2 URL 문법

앞서 언급했듯이 URL은 리소스의 위치와 해당 리소스에 접근하는 방식을 함께 나타내는 표준 형식이다.
HTTP, FTP 등 서로 다른 스킴을 사용하더라도 공통된 URL 문법을 따르기 때문에, 일관된 방식으로 리소스를 식별하고 접근할 수 있다.

URL 스킴의 문법은 일반적으로 9개 부분으로 나뉜다.  
`<스킴>://<사용자 이름>:<비밀번호>@<호스트>:<포트>/<경로>;<파라미터>?<쿼리>#<프래그먼트>`

| 컴포넌트   | 설명                                             | 기본값         |
|--------|------------------------------------------------|-------------|
| 스킴     | 사용할 프로토콜                                       | 없음          |
| 사용자 이름 | 특정 스킴(e.g. FTP)에서 인증할 때 요구하는 사용자의 이름           | `anonymous` |
| 비밀번호   | 사용자의 비밀번호                                      | `<이메일 주소>`  |
| 호스트    | 호스팅하는 장비의 호스트 명이나 IP 주소                        | 없음          |
| 포트     | 호스팅하는 장비 내에서 리소스를 접근할 수 있는 서버를 특정              | 스킴에 따라 다름   |
| 경로     | 서버 내 리소스의 경로                                   |             |
| 파라미터   | 애플리케이션이 요청을 정확하게 처리하기 위한 용도                    | 없음          |
| 질의     | 리소스를 특정하기 위해 사용하는 컴포넌트                         | 없음          |
| 프래그먼트  | 리소스 내부의 특정 부분을 가리키는 컴포넌트<sup>[1](#frg_1)</sup> | 없음          |

---
<a name="frg_1">1</a>: 프래그먼트는 브라우저 단에서 처리하기에 서버에 전달하지 않는다.

## 2.3 단축 URL

URL은 절대 URL과 상대 URL로 나타내는 나뉜다.
절대 URL은 리소스에 접근하는데 필요한 모든 정보를 가지고 있는 반면,
상대 URL은 리소스에 접근하는데 필요한 정보를 기저(Base)라고 하는 다른 URL을 사용해야 한다.

과정은 다음과 같다.

### RFC 2396

1. Base URI를 정한다.
2. 참조를 scheme, authority, path, query, fragment로 나눈다.
3. scheme이 있으면 절대 URI로 처리한다.
4. authority가 있으면 base의 scheme만 상속한다.
5. path가 / 로 시작하면 절대 경로로 처리한다.
6. 그 외에는 base path와 상대 path를 합친다.
7. . 과 .. 를 제거해 경로를 정리한다.
8. query, fragment를 반영해 최종 URI를 만든다.

RFC 2396은 상대 URI를 어떻게 절대 형태로 바꾸는가를 설명한 초기 일반 규칙에 가깝고, RFC 3986은 그 규칙을 더 엄밀하고 일반적으로 다시 정리한 최신 기준이다.

### RFC 3986

1. Base URI를 확정한다.
2. Base URI와 참조를 각각 구성 요소로 파싱한다.
3. scheme이 있으면 참조 자체를 사용한다.
4. authority가 있으면 network-path reference로 처리한다.
5. path가 비어 있으면 base path를 유지한다.
6. path가 / 로 시작하면 absolute-path reference로 처리한다.
7. 그 외에는 merge 규칙으로 path를 병합한다.
8. remove_dot_segments로 . 과 .. 를 제거한다.
9. query, fragment를 반영해 target URI를 만든다.

## 2.4 안전하지 않은 문자

안전한 전송이란 정보가 유실없이 전송되는 것을 의미한다.

출력이 되지 않거나 보이지 않는 문자를 사용하지 못하게 이러한 문자들을 URL에서는 금지가 되어있고, 안전하게 전송될 수 있도록 US-ASCII 문자셋만 사용해야 한다.

- 예약어: URL에서 특별한 의미를 가지는 문자들. (e.g. ;, /, ?, :, @, &, =, +, $, , 등)
- 이스케이프: 안전하지 않은 문자나 예약어를 URL에 포함시킬 경우 이스케이프 문자로 바꿔야 한다.
    - 인코딩 방식은 문자의 ASCII 코드 값을 16진수로 표현하고 앞에 %를 붙인다.
