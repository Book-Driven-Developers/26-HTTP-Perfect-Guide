## 0. 웹 서버란 무엇인가

- 웹 서버는 **HTTP 요청을 처리하고 응답을 제공**한다. 모든 웹 서버는 리소스에 대한 HTTP 요청을 받아 콘텐츠를 클라이언트에게 돌려준다.
- HTTP 통신을 하는 모든 애플리케이션이 웹 서버는 아니지만(예: iOS 앱), **모든 웹 서버는 HTTP 요청을 처리할 수 있어야 한다.**
- 웹 서버는 HTTP 및 그와 관련된 TCP 처리를 구현한 것이다. TCP 커넥션을 쓰기 때문에 **운영체제와 커넥션 관리 책임을 나눠 갖는다.**
    - 운영체제의 역할: 하드웨어 관리, TCP/IP 네트워크 지원, 리소스를 위한 파일 시스템, 연산을 제어하는 프로세스 관리.

### 웹 서버의 형태
| 형태 | 설명 | 예 |
|---|---|---|
| 다목적 소프트웨어 웹 서버 | 표준 컴퓨터에 설치·실행 | Apache, Nginx, Microsoft 웹서버 |
| 임베디드 웹 서버 | 제품에 내장되도록 작게 만든 서버 | 프린터, 가전제품 등 |

### 💬 최근 웹 서버 트렌드 (보강)
책 출간 당시는 Apache가 우위였으나, Nginx 등장 이후 점유율 지형이 바뀌었다. 2026년 기준은 다음과 같다.

| 순위 | 웹 서버 | 점유율(대략) |
|---|---|---|
| 1 | Nginx | 약 40.7% |
| 2 | Apache | 약 35.8% |
| 3 | LiteSpeed | 약 15.6% |

전통적 범용 웹 서버 외에 Node.js, Netty 기반 애플리케이션 서버도 마이크로서비스 환경에서 빠르게 성장 중이다.

## 웹 서버가 하는 일 — 7단계

모든 웹 서버가 공통으로 수행하는 작업은 다음 7단계다.

1. **커넥션을 맺는다** — 클라이언트 접속을 받아들이거나, 원치 않으면 닫는다.
2. **요청을 받는다** — HTTP 요청 메시지를 네트워크로부터 읽는다.
3. **요청을 처리한다** — 메시지를 해석하고 행동을 취한다.
4. **리소스에 접근한다** — 메시지가 지정한 리소스에 접근한다.
5. **응답을 만든다** — 올바른 헤더를 포함한 HTTP 응답을 생성한다.
6. **응답을 보낸다** — 응답을 클라이언트에게 돌려준다.
7. **로그를 남긴다** — 트랜잭션 완료 기록을 로그 파일에 남긴다.

## 1단계. 커넥션 맺기 — 클라이언트의 정체 밝히기

- 웹 서버는 클라이언트의 정체를 확인해 커넥션을 맺을지 결정할 수 있다. IP나 호스트명이 인가되지 않았거나 악의적이면 커넥션을 닫는다.
- 클라이언트의 호스트명을 알아낼 때는 **역방향 DNS(reverse DNS)** 를 사용한다.
    - TCP 통신은 IP 기반인데도 서버 로그에 호스트명이 남는 이유: 서버가 직접 hostname lookup을 수행해 알아낸 것.

### 💬 커널이란 무엇인가 (보강)
1단계의 TCP 커넥션 수락은 운영체제, 정확히는 **커널**의 도움 없이는 불가능하다.

- **커널(kernel)**: 운영체제의 핵심으로, 하드웨어(CPU·메모리·디스크·네트워크 카드)를 직접 다루는 유일한 소프트웨어. "알맹이, 핵"이라는 뜻.
- 프로그램은 하드웨어를 직접 만지지 못하고, 반드시 커널을 거쳐 요청한다.

**유저 공간 vs 커널 공간**

| | 유저 공간 (User space) | 커널 공간 (Kernel space) |
|---|---|---|
| 권한 | 낮음 | 높음 |
| 도는 것 | 웹 서버, 브라우저, 자바 앱 등 | 커널 |
| 하드웨어 | 직접 제어 불가 | 직접 제어 가능 |

- 이렇게 나눈 이유는 **안정성·보안**. 프로그램이 하드웨어를 직접 만지면 버그 하나로 시스템 전체가 죽을 수 있어서, 위험한 작업은 전부 커널을 통하게 막아둔 것. 프로그램이 죽어도 OS가 멀쩡한 이유다.
- **시스템 콜(system call)**: 유저 프로그램이 커널에 일을 부탁하는 통로.

```java
in.read(buffer);   // "이 소켓에서 데이터 읽어줘" → 커널에 부탁(시스템 콜)
```

- 소켓·네트워크 연결은 전부 커널이 관리하는 자원이라 유저 공간에서는 볼 권한이 없다. 그래서 뒤에 나올 이벤트 루프도 "어느 소켓이 준비됐는지"를 커널에게 물어볼 수밖에 없다.

**OS별 커널 이름**: Linux는 커널 이름이 곧 Linux(epoll 제공), macOS/BSD는 XNU(kqueue 제공), Windows는 NT 커널.

> 비유: 커널은 건물 관리사무소. 입주민(프로그램)은 전기실·보일러실(하드웨어)에 직접 못 들어가고 인터폰(시스템 콜)으로 요청한다.

## 2단계. 요청 받아들이기 — 동시성 처리 아키텍처 4가지

안전한 클라이언트로 판별되면 요청을 받아들인다. 책에서는 동시에 들어오는 수많은 요청을 처리하기 위한 아키텍처를 4가지로 소개한다.

### ① 단일 스레드 I/O 아키텍처
- 하나의 스레드/프로세스가 한 커넥션을 차례대로 처리한다. 처리 중 다른 커넥션은 무시.
- latency가 매우 높아, 부하가 적은 서버나 진단 도구 정도에만 적합.

### ② 멀티 스레드(프로세스) I/O 아키텍처
- 여러 스레드/프로세스가 각각 요청을 처리한다. 미리 만들어두거나 커넥션마다 새로 만든다.
- 보통 과도한 생성을 막기 위해 최대 개수를 제한한다.
- 단점: 연결 수만큼 스레드/프로세스가 늘어 **메모리·컨텍스트 스위칭 비용**이 급증.
- 대표 예: Apache prefork MPM(연결당 1 프로세스, mod_php 호환은 좋지만 메모리 부담 큼).

### ③ 다중 I/O 아키텍처 (멀티플렉스, 이벤트 루프)
- TCP 지속 커넥션으로 커넥션을 다중화하고, 준비된 요청만 골라 처리한다. 요즘 말하는 **이벤트 루프 방식**.
- `epoll`(Linux), `kqueue`(macOS/BSD), `select` 같은 커널 메커니즘으로 단일 스레드가 수천 연결을 감시한다.

### ④ 다중 + 멀티 스레드 I/O 아키텍처
- ③의 이벤트 루프를 각 CPU마다 수행한다. 이벤트 루프 담당 스레드는 보통 **CPU 개수만큼** 생성.
- 멀티플렉싱 효율 + 멀티코어 활용을 모두 얻는, 현재 고성능 서버의 사실상 표준.

### 💬 주요 웹 서버의 아키텍처 매핑 (보강)

| 웹 서버 | 해당 아키텍처 | 특징 |
|---|---|---|
| Apache (prefork) | ② 멀티 프로세스 | 요청당 1 프로세스, 메모리 부담 높음 |
| Apache (Event MPM, 현재 기본) | ④ 다중 + 멀티스레드 | keep-alive 연결을 비동기 처리 |
| Nginx | ③+④ 이벤트 루프 + 멀티워커 | CPU 코어 수만큼 워커, C10K 문제 해결 |
| Node.js | ③ 다중 I/O (단일 스레드) | libuv 이벤트 루프, 보조 스레드 풀 병행 |
| Tomcat (NIO) | ④ 다중 + 멀티스레드 | NIO Connector + 스레드 풀, Java 웹앱 표준 |
| Netty / Spring WebFlux | ④ 다중 + 멀티스레드 | 이벤트 루프 그룹(Reactor), 비동기 Java 서버 |

**왜 이 아키텍처를 선택했나**
- Nginx: 요청마다 프로세스를 만드는 대신 이벤트 기반으로 처리해 메모리를 절약하고 동시성 확보(C10K — 1만 동시 연결 — 해결이 설계 목표).
- Apache: prefork의 메모리 문제를 보완하려 Event MPM에서 keep-alive를 비동기로 처리하며 Nginx 모델에 근접.
- Java 진영: Netty/WebFlux가 Reactor 패턴 기반 이벤트 루프 그룹으로 ④를 구현, Tomcat(블로킹) 대비 동시 연결에 유리.

### 💬 ③④는 어떻게 비블로킹으로 처리하나 (보강)

핵심: **블로킹은 스레드의 본질적 성질이 아니라, 어떤 시스템 콜을 쓰느냐의 문제다.**

- 블로킹 `read()`: 데이터가 없으면 커널이 그 스레드를 재워버린다(blocked 상태). 이게 블로킹의 정체.
- 그래서 ②는 "어차피 한 스레드가 read에서 잠드니, 연결마다 스레드를 따로 두자"는 발상.

비블로킹은 두 가지가 맞물려 동작한다.

1. **소켓을 non-blocking 모드로 설정**
    - 데이터 있음 → 즉시 읽어 반환 / 데이터 없음 → 재우지 않고 "지금 없음(EWOULDBLOCK)" 즉시 반환.
    - 스레드가 개별 소켓에 묶여 잠들지 않는다.
2. **epoll/kqueue로 다중화(multiplexing)**
    - 단순 비블로킹만 쓰면 "없음" 반환을 무한 반복하는 busy-waiting(CPU 100%)이 된다.
    - epoll은 "1만 개 소켓 중 **준비된 것만** 골라줘"라고 커널에 부탁하는 메커니즘.

> Java에서는 `java.nio`의 `Selector`가 epoll/kqueue를 추상화한 것.
> ```java
> Selector selector = Selector.open();
> channel.configureBlocking(false);              // ① non-blocking
> channel.register(selector, SelectionKey.OP_READ);
> while (true) {
>     selector.select();                          // ② epoll_wait 역할: 준비될 때까지 대기
>     Set<SelectionKey> ready = selector.selectedKeys();
>     for (SelectionKey key : ready) { /* 준비된 채널만 처리 */ }
> }
> ```
> Tomcat NIO Connector, Netty가 내부적으로 이 구조를 사용한다.

**중요 포인트**: `epoll_wait()` 자체도 준비된 소켓이 없으면 블로킹된다. 차이는 *어디서* 자느냐다.
- ② 모델: 스레드 하나가 **특정 소켓 하나**의 read에서 잠 → 1만 연결 = 1만 스레드.
- ③④ 모델: 스레드 하나가 **1만 개 소켓 전체**를 보는 epoll_wait 한 곳에서 잠 → 1개 스레드로 충분.

> 비유: ②는 상담원 1만 명이 각자 전화기 하나씩 붙잡고 대기. ③④는 한 명이 1만 개 전화기 알림판을 보며 불 들어온 것만 받음.

### 💬 이벤트 루프란 (보강)

흔한 오해: "루프를 돌면서 소켓을 하나씩 탐색한다" → **틀림.** 이벤트 루프는 소켓을 직접 훑지 않는다.

```
while (true) {                  // ← 이 while이 "루프"
    준비된_목록 = epoll_wait();   // ① 소켓 탐색은 커널이 대신 해줌
    for (이벤트 : 준비된_목록) {   // ② 준비된 것만 순회 (전체가 아님)
        해당_콜백_실행();
    }
}
```

- "루프"는 소켓 배열을 도는 게 아니라 위 사이클을 반복한다는 뜻.
- 소켓 1만 개를 훑는 일은 커널(epoll)이 하고, 루프는 커널이 골라준 짧은 목록만 처리한다.
- 옛 `select()`는 매 바퀴 전체 소켓을 검사해 C10K 문제의 원인이 됐다. epoll은 준비된 것만 돌려준다.

**"워커에게 전달한다"의 의미 — 워커는 두 가지 뜻으로 쓰인다**

| 구분 | ⓐ 워커 프로세스 | ⓑ 스레드 풀 워커 |
|---|---|---|
| 의미 | 이벤트 루프를 가진 일꾼 | 블로킹 작업 대신 처리하는 보조 |
| 목적 | 멀티코어 활용 | 루프가 막히지 않게 함 |
| 예 | Nginx worker, Netty EventLoopGroup, Node.js cluster | Node.js libuv 스레드 풀 |

- 기본형(③): 이벤트 루프가 준비된 이벤트를 **자기가 직접** 콜백 실행. 전달할 워커가 따로 없다.
- 확장형(④): 네트워크 read/write는 루프가 직접 처리하고, **파일 I/O·DB 같은 진짜 블로킹 작업만** 스레드 풀 워커(ⓑ)에게 넘긴다.

> 정리: 이벤트 루프는 커널(epoll)이 골라준 준비된 이벤트 목록을 받아, 루프 한 바퀴마다 그 콜백을 대부분 자기가 직접 실행한다. 다만 불가피한 블로킹 작업만 별도 스레드 풀에 넘기고, 자신은 막히지 않고 다음 이벤트를 받는다.

## 4단계. 리소스의 매핑과 접근

- 웹 서버의 역할은 요청을 받아 리소스를 돌려주는 것. **정적·동적 콘텐츠를 모두** 돌려줄 수 있어야 한다.
- 정적 콘텐츠 식별의 가장 쉬운 방법은 리소스 경로를 URI로 받는 것.
- 동적 리소스: 콘텐츠를 생성하는 프로그램에 URI를 매핑한다.
    - 예: Nginx가 요청을 받아 우리가 만든 Spring Boot 컨트롤러에 URI를 매핑하는 식.
- **접근 제어**: IP에 따라 접근을 제한하거나 패스워드를 물어볼 수 있다. (12장에서 상세히 다룸)

### 💬 서버 사이드 인클루드(SSI) (보강)
동적 콘텐츠를 만드는 한 방법으로, 책 5장 맥락에서 등장한다.

- **정의**: 웹 서버가 HTML을 클라이언트에 보내기 전에 특수 주석(`<!--#directive -->`)을 해석해 동적 콘텐츠를 생성하는 간단한 서버 사이드 스크립팅 언어.
- **역사**: 1990년대 초 NCSA HTTPd에서 시작, 현재는 Apache의 `mod_include` 모듈이 가장 일반적인 구현체.
- **용도**: 공통 헤더/푸터 삽입, CGI 스크립트 실행, 날짜·환경 변수 출력. CGI/JSP/Servlet 없이도 HTML에 동적 내용을 추가할 수 있다.

```html
<!-- 파일 인클루드 -->
<!--#include virtual="/common/header.html" -->

<!-- 환경 변수 출력 -->
<!--#echo var="DATE_LOCAL" -->

<!-- 조건문 -->
<!--#if expr="${QUERY_STRING} = foo" -->
  <p>foo입니다</p>
<!--#endif -->
```

**지금도 쓰이나?**
- 정적 사이트 생성기 등 일부 틈새 영역에서 명맥은 유지. 그러나 현대 웹에서는 더 강력한 대안이 표준이다.

| 용도 | SSI 대신 쓰는 것 |
|---|---|
| 공통 헤더/푸터 | React/Vue 컴포넌트, Thymeleaf 레이아웃 |
| 동적 콘텐츠 삽입 | JSP, PHP, Jinja2, Thymeleaf |
| 정적 사이트 | Jekyll, Hugo, Next.js (SSG) |

- **보안 주의**: SSI 인젝션은 1990년대 공격 기법이지만, 잘못 설정된 서버에서는 지금도 동작한다.

---

## 5단계. 응답 만들기

- 응답 메시지에 **상태 코드, 헤더, 본문**을 넣는 과정.
- 본문이 있으면 MIME 타입을 알려주는 `Content-Type`, 길이를 알려주는 `Content-Length` 헤더가 필요. MIME 타입은 보통 리소스 확장자로 결정한다.
- **리다이렉션**: 성공 대신 리다이렉션 응답을 줘야 할 때는 `Location` 헤더에 목적지 주소를 넣는다.
    - 리소스가 옮겨졌거나 이름이 바뀐 경우, URL 확장, 과부하 서버의 부하 분산 등 다양한 이유로 수행.

---

## 6단계. 응답 보내기

- 지속 커넥션(keep-alive)이면 고려할 점이 많다. `Content-Length`를 정확히 계산해야 하고, 클라이언트가 응답 끝을 알 수 있어야 커넥션 유지가 가능.
- 비지속 커넥션이면 서버는 모든 메시지를 전송한 뒤 커넥션을 닫는다.

---

## 7단계. 로깅

- 트랜잭션이 완료되면 수행 기록을 로그 파일에 남긴다. (21장에서 상세히 다룸)

---

## 한눈에 보는 흐름 요약

```
[1] 커넥션 맺기 (커널이 TCP 수락, reverse DNS로 정체 확인)
        ↓
[2] 요청 받기 (동시성 아키텍처 4가지 中 선택 — 이벤트 루프/멀티스레드 등)
        ↓
[3] 요청 처리 (메서드·헤더 해석)
        ↓
[4] 리소스 매핑·접근 (정적/동적, SSI, 접근 제어)
        ↓
[5] 응답 만들기 (상태코드·헤더·본문, MIME, 리다이렉션)
        ↓
[6] 응답 보내기 (keep-alive 여부 고려)
        ↓
[7] 로깅
```

### 비블로킹 내부 동작 (2단계 심화)
```
커널이 소켓 관리 (유저 공간은 직접 못 봄)
   ↓
이벤트 루프가 epoll_wait() 시스템 콜로 "준비된 것만" 요청
   ↓
커널이 준비된 소켓 목록 반환
   ↓
이벤트 루프가 콜백 직접 실행 (네트워크 I/O)
   ↓
블로킹 작업(파일·DB)만 스레드 풀 워커에게 위임
   ↓
다시 epoll_wait() 반복
```
