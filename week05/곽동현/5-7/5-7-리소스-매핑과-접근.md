# 5.7 단계 4: 리소스의 매핑과 접근

웹 서버는 **리소스 서버**다.  
HTML 페이지나 JPEG 이미지 같은 미리 만들어진 콘텐츠를 제공하며, 마찬가지로 서버 위에서 동작하는 리소스 생성 애플리케이션을 통해 만들어진 **동적 콘텐츠**도 제공한다.

웹 서버가 콘텐츠를 전달하려면, 그전에 요청 메시지의 URI에 대응하는 알맞은 콘텐츠나 콘텐츠 생성기를 웹 서버에서 찾아서 그 콘텐츠의 원천을 식별해야 한다.

---

## 5.7.1 Docroot

리소스 매핑의 가장 단순한 형태는 **요청 URI를 웹 서버의 파일 시스템 안에 있는 파일 이름으로 사용**하는 것이다.  
일반적으로 웹 서버 파일 시스템의 특별한 폴더를 웹 콘텐츠를 위해 예약해 둔다. 이 폴더를 **문서 루트(document root)** 혹은 **docroot** 라고 부른다.  
웹 서버는 요청 메시지에서 URI를 가져와서 문서 루트 뒤에 붙인다.

```
요청 URI:     /specials/saw-blade.gif
문서 루트:    /usr/local/httpd/files
반환 파일:    /usr/local/httpd/files/specials/saw-blade.gif
```

아파치에서는 `httpd.conf` 에 `DocumentRoot` 줄을 추가하여 문서 루트를 설정한다.

```apache
DocumentRoot /usr/local/httpd/files
```

⚠️ 서버는 상대적인 URL이 docroot를 벗어나서 파일 시스템의 docroot 이외 부분이 노출되는 일이 생기지 않도록 주의해야 한다.  
대부분의 성숙한 웹 서버는 다음과 같이 문서 루트 위의 파일을 보려는 URI(**경로 탐색, path traversal**)를 허용하지 않는다.

```
http://www.joes-hardware.com/../
```

### 가상 호스팅된 docroot

**가상 호스팅(virtual hosting)** 웹 서버는, 각 사이트에 그들만의 분리된 문서 루트를 주는 방법으로 한 웹 서버에서 여러 개의 웹 사이트를 호스팅 한다.  
가상 호스팅 웹 서버는 URI나 `Host` 헤더에서 얻은 IP 주소나 호스트 명을 이용해 올바른 문서 루트를 식별한다.

```
요청 A: GET /index.html  Host: www.joes-hardware.com   → /docs/joe/index.html
요청 B: GET /index.html  Host: www.marys-antiques.com  → /docs/mary/index.html
```

아파치에서는 각 가상 웹 사이트의 `VirtualHost` 블록이 `DocumentRoot` 지시자를 포함하도록 설정한다.

```apache
<VirtualHost www.joes-hardware.com>
    ServerName www.joes-hardware.com
    DocumentRoot /docs/joe
    TransferLog /logs/joe.access_log
    ErrorLog /logs/joe.error_log
</VirtualHost>

<VirtualHost www.marys-antiques.com>
    ServerName www.marys-antiques.com
    DocumentRoot /docs/mary
    TransferLog /logs/mary.access_log
    ErrorLog /logs/mary.error_log
</VirtualHost>
```

> 가상 호스팅에 대한 자세한 내용은 18장 "가상 호스팅" 참고.

### 사용자 홈 디렉터리 docroots

docroot의 또 다른 대표적 활용은, 사용자들이 한 대의 웹 서버에서 각자의 개인 웹 사이트를 만들 수 있도록 하는 것이다.  
보통 빗금(`/`)과 물결표(`~`) 다음에 사용자 이름이 오는 것으로 시작하는 URI는 그 사용자의 개인 문서 루트를 가리킨다. 개인 docroot는 주로 사용자 홈 디렉터리 안에 있는 `public_html` 디렉터리지만, 설정에 따라 다르다.

```
요청 A: GET /~bob/index.html   → /home/bob/public_html/index.html
요청 B: GET /~betty/index.html → /home/betty/public_html/index.html
```

## 5.7.2 디렉터리 목록

웹 서버는 경로가 파일이 아닌 **디렉터리를 가리키는** URL에 대한 요청을 받을 수 있다.  
대부분의 웹 서버는 클라이언트가 디렉터리 URL을 요청했을 때 다음 중 하나를 취하도록 설정할 수 있다.

- 에러를 반환한다.
- 디렉터리 대신 특별한 **'색인 파일'** 을 반환한다.
- 디렉터리를 탐색해서 그 내용을 담은 HTML 페이지를 반환한다.

대부분의 웹 서버는 요청한 URL에 대응되는 디렉터리 안에서 `index.html` 혹은 `index.htm` 으로 이름 붙은 파일을 찾는다.  
아파치에서는 `DirectoryIndex` 지시자로 기본 디렉터리 파일로 사용될 파일 이름의 집합을 **우선순위 순**으로 설정한다.

```apache
DirectoryIndex index.html index.htm home.html home.htm index.cgi
```

기본 색인 파일이 없고 디렉터리 색인 기능도 켜져 있다면, 많은 웹 서버는 자동으로 그 디렉터리의 파일들을 크기·변경일·링크와 함께 열거한 HTML 파일을 반환한다.  
⚠️ 이 파일 열거는 편리하지만 **일반적으로는 발견할 수 없는 파일도 드러나게 된다는 단점**이 있다. 다음과 같이 디렉터리 색인 자동 생성을 끌 수 있다.

```apache
Options -Indexes
```

## 5.7.3 동적 콘텐츠 리소스 매핑

웹 서버는 URI를 **동적 리소스에 매핑**할 수도 있다. 즉, 요청에 맞게 콘텐츠를 생성하는 프로그램에 URI를 매핑하는 것이다.  
사실, 웹 서버들 중에서 **애플리케이션 서버**라고 불리는 것들은 웹 서버를 복잡한 백엔드 애플리케이션과 연결하는 일을 한다.

어떤 리소스가 동적 리소스라면, 애플리케이션 서버는 그 동적 콘텐츠 생성 프로그램이 **어디에 있는지**, 그리고 **어떻게 실행하는지** 알 수 있어야 한다.

아파치는 URI의 경로명을 실행 가능한 프로그램이 위치한 디렉터리로 매핑되도록 설정하는 기능을 제공한다. 다음 지시자는 URI 경로가 `/cgi-bin/` 으로 시작하면 `/usr/local/etc/httpd/cgi-programs/` 에서 프로그램을 찾아 실행하라는 의미다.

```apache
ScriptAlias /cgi-bin/ /usr/local/etc/httpd/cgi-programs/
```

또한 특정 확장자의 파일만 실행하도록 설정할 수도 있다. 다음은 `.cgi` 로 끝나는 모든 웹 리소스를 실행되어야 함을 명시한다.

```apache
AddHandler cgi-script .cgi
```

> 웹 초창기에 널리 쓰였던 **CGI(Common Gateway Interface)** 는 서버사이드 애플리케이션을 실행하기 위한 간단한 인터페이스다. 오늘날의 애플리케이션 서버는 마이크로소프트의 액티브 서버 페이지(ASP)와 **자바 서블릿(Java Servlet)** 같은 한층 더 강력하고 효과적인 서버사이드 동적 콘텐츠 지원 기능을 갖고 있다.

> **Java 개발자 관점**: 책이 직접 언급하듯 서블릿이 CGI의 후계자다. CGI는 요청마다 프로세스를 새로 fork했지만, 서블릿은 컨테이너 안에서 스레드로 재사용되어 훨씬 효율적이다. `ScriptAlias`·`AddHandler` 가 하던 "이 URI는 프로그램으로 매핑하라"는 역할을, 지금은 `@RequestMapping` 이나 `web.xml`/`@WebServlet` 의 URL 패턴이 한다.

## 5.7.4 서버사이드 인클루드(Server-Side Includes, SSI)

많은 웹 서버가 **서버사이드 인클루드(SSI)** 를 지원한다.  
어떤 리소스가 SSI를 포함하는 것으로 설정되어 있다면, 서버는 그 리소스의 콘텐츠를 클라이언트에게 보내기 전에 처리한다.

- 서버는 콘텐츠에 **변수 이름이나 내장된 스크립트가 될 수 있는 특별한 패턴**(주로 특별한 HTML 주석 안에 포함)이 있는지 검사한다.
- 특별한 패턴은 변수 값이나 실행 가능한 스크립트의 출력 값으로 치환된다.
- 동적 콘텐츠를 만드는 쉬운 방법이다.

> **Java 개발자 관점**: JSP의 `<%= ... %>` 나 Thymeleaf 템플릿이 발전된 형태의 SSI라고 볼 수 있다. "응답을 보내기 전 서버에서 특정 패턴을 치환한다"는 발상이 동일하다.

## 5.7.5 접근 제어

웹 서버는 각각의 리소스에 **접근 제어**를 할당할 수 있다.  
접근 제어되는 리소스에 대한 요청이 도착했을 때 웹 서버는 다음을 할 수 있다.

- 클라이언트의 **IP 주소**에 근거하여 접근을 제어한다.
- 리소스에 접근하기 위한 **비밀번호**를 물어본다.

> HTTP 인증에 대한 자세한 정보는 12장 참고.
