# 5.8 단계 5: 응답 만들기

서버가 리소스를 식별하면, 서버는 요청 메서드로 서술되는 동작을 수행한 뒤 응답 메시지를 반환한다.  
응답 메시지는 **응답 상태 코드, 응답 헤더, 응답 본문**(생성되었다면)을 포함한다. (HTTP 응답 코드는 3장 "상태 코드" 참고)

---

## 5.8.1 응답 엔터티

트랜잭션이 응답 본문을 생성한다면, 그 내용을 응답 메시지와 함께 돌려보낸다.  
본문이 있다면, 응답 메시지는 주로 다음을 포함한다.

- 응답 본문의 MIME 타입을 서술하는 `Content-Type` 헤더
- 응답 본문의 길이를 서술하는 `Content-Length` 헤더
- 실제 응답 본문의 내용

```http
HTTP/1.0 200 OK
Content-type: image/gif
Content-length: 8572

[...이미지 바이너리 데이터...]
```

---

## 5.8.2 MIME 타입 결정하기

웹 서버에게는 응답 본문의 **MIME 타입을 결정해야 하는 책임**이 있다.  
다음은 MIME 타입과 리소스를 연결하는 여러 가지 방법이다.

### ① mime.types (확장자 기반)

웹 서버는 MIME 타입을 나타내기 위해 **파일 이름의 확장자**를 사용할 수 있다.  
각 리소스의 MIME 타입을 계산하기 위해 확장자별 MIME 타입이 담겨 있는 파일(`mime.types`)을 탐색한다. **가장 흔한 방법**이다.

```
서버 MIME 타입 테이블 (mime.types):
application/msword       doc
application/postscript   ai eps ps
audio/mpeg               mpga mp2
image/gif                gif
image/jpeg               jpeg jpg jpe
text/html                html htm
text/plain               txt
video/mpeg               mpeg mpg mpe
video/quicktime          qt mov
```

### ② 매직 타이핑 (Magic typing)

아파치는 각 파일의 MIME 타입을 알아내기 위해 **파일의 내용을 검사**해서, 알려진 패턴에 대한 테이블(**매직 파일**이라 불린다)에 해당하는 패턴이 있는지 찾아본다.  
- **장점**: 파일이 표준 확장자 없이 이름 지어진 경우 특히 편리하다.
- **단점**: 내용을 직접 검사하므로 느리다.

### ③ 유형 명시 (Explicit typing)

특정 파일이나 디렉터리 안의 파일들이, 파일 확장자나 내용에 상관없이 어떤 MIME 타입을 갖도록 웹 서버를 설정할 수 있다.

### ④ 유형 협상 (Type negotiation)

어떤 웹 서버는 한 리소스가 **여러 종류의 문서 형식**에 속하도록 설정할 수 있다.  
이때 웹 서버가 사용자와의 협상 과정을 통해, 사용하기 가장 좋은 형식(그리고 대응하는 MIME 타입)을 판별할 것인지의 여부도 설정할 수 있다.

> **Java 개발자 관점**: 서블릿/스프링에서 `response.setContentType("application/json")` 으로 직접 지정하는 것이 ③ 유형 명시에 해당한다. ④ 유형 협상은 클라이언트의 `Accept` 헤더를 보고 JSON/XML 중 골라주는 Spring의 Content Negotiation(`produces` 속성, `ContentNegotiatingViewResolver`)과 정확히 같은 개념이다.

## 5.8.3 리다이렉션

웹 서버는 종종 성공 메시지 대신 **리다이렉션 응답**을 반환한다.  
요청을 수행하기 위해 브라우저가 다른 곳으로 가도록 리다이렉트 할 수 있다. 리다이렉션 응답은 **3XX 상태 코드**로 지칭된다.  
`Location` 응답 헤더는 콘텐츠의 새로운 혹은 선호하는 위치에 대한 URI를 포함한다.

리다이렉트가 유용한 경우:

| 상황 | 설명 | 사용 상태 코드 |
|------|------|---------------|
| **영구히 리소스가 옮겨진 경우** | 리소스에 새 URL이 부여됨. 클라이언트가 북마크를 갱신할 수 있게 알려줌 | **301 Moved Permanently** |
| **임시로 리소스가 옮겨진 경우** | 이름이 임시로 변경됨. 나중엔 원래 URL로 찾아오길 원함 | **303 See Other / 307 Temporary Redirect** |
| **URL 증강** | 상태 정보를 포함하기 위해 재작성된 URL로 리다이렉트 ('뚱뚱한(fat) URL') | **303 / 307** |
| **부하 균형** | 과부하된 서버가 덜 부하 걸린 서버로 리다이렉트 | **303 / 307** |
| **친밀한 다른 서버가 있을 때** | 사용자 정보를 가진 다른 서버로 리다이렉트 | **303 / 307** |
| **디렉터리 이름 정규화** | 디렉터리 URI 끝에 빗금(`/`)이 빠졌을 때 슬래시를 추가한 URI로 리다이렉트 | (슬래시 추가 리다이렉트) |

> **301 vs 307의 핵심 차이**: 301은 "영구 이동"이므로 클라이언트(브라우저)가 북마크/캐시를 갱신해 다음부턴 새 URL로 직접 간다. 307은 "임시 이동"이므로 클라이언트는 갱신하지 않고 다음에도 원래 URL로 다시 찾아온다.

> **Java 개발자 관점**: `response.sendRedirect(url)` 은 기본적으로 302를 내보내며, Spring에서 `redirect:` 접두사가 이 동작을 추상화한다. 디렉터리 이름 정규화 리다이렉트는 `/path` 요청에 `/path/` 로 응답하는 동작으로, 상대 경로 링크가 깨지지 않게 하기 위한 것이다.
